Self-screen path is dead code — schema_fail cascade fires before parsed.skip is checked
breaking-news/index.ts:425 · _shared/llm/registry.ts:493-501 · _shared/llm/schema.ts:74-75 · _shared/llm/factory.ts:114-119

The self-screen prompt tells the model to return {"skip": true, "skip_reason": "...", "vela_read": "", "analysis": ""}. The BREAKING_TELEGRAM_SCHEMA requires vela_read with minLength: 3 and analysis with minLength: 10. The schema validator in schema.ts:74-75 flags both empty strings as violations.

Because breaking_telegram has cascadeOnSchemaFail: true (registry.ts:496), the factory converts that schema_fail to a retryable synthetic error and throws (factory.ts:114-119). The error bubbles out of the try { const result = await llmComplete(...) } block at index.ts:396-408, triggering the catch path: console.error + admin Telegram warning + return null. The skip check at line 425 is never reached.

Effect: every model self-screen fires a false-alarm admin WARNING ("Telegram analysis failed") instead of the intended silent console.log. The story is still suppressed (step 9 placeholder was already written at line 1040 before generateTelegramAnalysis was called, so dedup is safe). But admin noise accumulates and the "distinct from a generation failure" guarantee is broken — operators cannot distinguish real failures from intentional skips in Telegram alerts.

The fix requires either (a) relaxing the schema so skip responses pass validation, or (b) intercepting the skip response before schema validation.

Fix: In BREAKING_TELEGRAM_SCHEMA (registry.ts:437-444), add skip and skip_reason to properties and change vela_read and analysis to minLength: 0 (i.e. remove minLength). Add a server-side guard in the caller at line 431 that enforces non-empty analysis after confirming skip !== true — this is already there (the if (!analysis) block at line 433). The schema minLength constraints are redundant given that guard, so dropping them is safe.
Alternative: Change the prompt to instruct the model to return vela_read: "skip" and analysis: "skip" (three-char strings that clear minLength) so existing schema passes. Then check parsed.skip === true before the non-empty analysis guard. Simpler schema change but slightly awkward prompt instruction.

Analysis section silently lost six banned terms from the generation prompt
breaking-news/index.ts:361-364 · _shared/jargon-constants.ts (missing terms)

The old analysis BANNED JARGON block contained six terms not present in jargon-constants.ts: easing cycle, rate hikes, rate cuts, accommodative, restrictive policy, tightening policy. The new prompt acknowledges them with "plus: easing cycle, rate hikes → ..., ...)" in the label text above ${BANNED_JARGON_LINE}, but BANNED_JARGON_LINE does not contain them. The model sees a parenthetical note followed by a list that omits those terms.

These were never eval-enforced (Layer 1 only checks the imported arrays), so this does not break the eval gate. But the generation-prompt instruction is weaker: the model may reason that the canonical list does not include them and use them. The difference will appear as an increase in macro/rate language in analysis copy — catchable by Layer 2 judge as "jargon" concern, but not guaranteed.

Fix: Add the six terms to EQUITIES_MACRO_JARGON in jargon-constants.ts so both the generation prompt and Layer 1 eval enforce them. Note the existing comment in news-broadcast-eval.ts explaining why "rate cuts" was previously excluded from the eval (false-positive on "rate cuts now look further away" examples). If that exclusion still applies, add only the four unambiguous ones (easing cycle, rate hikes, accommodative, restrictive policy) and leave rate cuts and tightening policy as prompt-only.

SELF-SCREEN asset allow-list is static and will drift from the live asset table
breaking-news/index.ts:307-309

The third disqualifier condition hardcodes BTC ETH SOL HYPE ZEC NEAR NVDA AAPL AMZN MSFT GOOGL META MU SNDK INTC DELL BB gold oil silver copper. This list is an advisory hint to the model; the real enforcement gate is Layer 1's hasAssetRelevance(), which uses the live tradeableAssets set from getTradeableAssetIds(). So correctness at the gate level is not at risk.

The risk is model behavior: when a new asset is added to Vela (e.g. a future equity), the model may self-screen stories about it because the asset isn't in the prompt list, even though Layer 1 would have passed them. This produces silent suppression with no admin alert (the self-screen path intentionally skips admin pings) that is hard to debug.

Consider: Add a comment adjacent to the list ("advisory only — actual enforcement is Layer 1 asset relevance check") so the next person editing the asset list knows to update this prompt too. Low effort, prevents a future "why aren't we broadcasting NEAR stories" puzzle.
