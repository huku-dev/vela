Worth a Look (3)
1. /\bbb\b/i will false-positive on Bollinger Band references in finance news
news-fetcher.ts:443 · twitter-fetcher/index.ts:71

Finance news and technical analysis articles routinely abbreviate "Bollinger Band" as "BB" (e.g., "BTC broke above its BB resistance", "upper BB band tested", "BB squeeze signal"). The pattern /\bbb\b/i has no guard against this context and will attribute these articles to BlackBerry. Contrast with HYPE: the comment at news-fetcher.ts:423 explicitly records that /\bhype\b/i was removed because it matched "AI hype / market hype / tech hype". The BB collision risk is analogous and arguably higher because Bollinger Bands are a direct finance-domain term, not just a generic English word.

The ASSET_KEYWORDS_EXACT list for bb (line 398) is strong and specific: "blackberry", "blackberry qnx", "blackberry cylance", etc. In practice, the exact keywords will correctly classify most genuine BlackBerry articles. The boundary pattern is the exposure: it fires as a secondary score signal, boosting articles that contain "BB" anywhere.

Option A (preferred): Add a safety comment documenting the Bollinger Band collision risk, analogous to the HYPE removal comment. Monitor false-positive rate in news_cache.matched_assets for the first week post-deploy. If the rate is high, remove /\bbb\b/i and rely solely on the exact keyword list plus /\$bb\b/i and /\bblackberry\b/i.

Option B (conservative): Remove /\bbb\b/i now and keep only /\$bb\b/i and /\bblackberry\b/i. The cashtag pattern catches market-context "BB" with high precision; the name catches the rest.
2. Migration missing ON CONFLICT clause; inconsistent with newest precedent
supabase/migrations/20260601000001_add_dell_bb.sql:2–5

The ZEC migration (20260520000002_add_zec_asset.sql) uses ON CONFLICT (id) DO UPDATE SET hl_symbol = COALESCE(...), icon_url = COALESCE(...). The dell/bb migration uses a plain INSERT with no conflict handling. If a developer runs this migration manually against staging after it has already been applied (e.g. after a staging reset + re-run), Postgres will throw a primary-key duplicate error and abort the migration.

The older batch migration (20260429000002_add_assets_batch.sql) also omits ON CONFLICT, so this is not new behavior. But ZEC established a stricter pattern that this migration does not follow.

Add ON CONFLICT (id) DO NOTHING (or the safer DO UPDATE with COALESCE, matching the ZEC pattern) to both INSERT rows.
3. TF-ASSET test verifies "blackberry" string but not the /\bbb\b/i boundary pattern
supabase/functions/_shared/twitter-fetcher.test.ts:234–248

The ZEC TF-ASSET test (twitter-fetcher.test.ts:250–258) explicitly checks both the exact keyword "zcash" AND the boundary pattern "/\bzec\b/i". The BB portion of the equity TF-ASSET test only checks fetcherSource.includes('"blackberry"'). If someone removes the boundary regex entry for bb from twitter-fetcher/index.ts while keeping the exact keywords, the test passes but the ticker-cashtag matching breaks silently.

Add fetcherSource.includes('/\\\\bbb\\\\b/i') to the equity TF-ASSET assertion (or split bb into its own test matching the ZEC pattern), so boundary regex coverage is verified independently of exact keywords.

ZEC (11), DELL (12), and BB (13) are also effectively locked from Standard users (Standard gets max_assets=8, positions 1–8). Update the comment to say "lock OIL + HYPE + all assets beyond position 10 behind the upgrade paywall."
N2. Migration uses assets table reference; ZEC used public.assets
supabase/migrations/20260601000001_add_dell_bb.sql:2

Minor style inconsistency. Both resolve correctly in Supabase (search_path is set), but the ZEC migration and the RLS migrations use the fully-qualified public.assets form. Not a correctness bug.

Optionally prefix with public. for consistency with ZEC precedent.
N3. "momentum" appears 4x in blackberry.md body copy (in the banned jargon list)
vela-marketing/src/content/assets/blackberry.md:24, 46, 54, 79

"Momentum" is in the CLAUDE.md banned jargon list for user-facing surfaces. However, the reference page intel.md also uses "momentum" twice (lines 24, 68), suggesting this rule has been applied loosely for marketing pages. Context matters: BB's meme-stock nature makes "momentum" somewhat load-bearing for describing the product. The intel.md precedent makes this a nit rather than a blocking voice violation.

If the jargon ban applies to marketing pages, replace with plain-English alternatives: "price swings driven by retail interest" instead of "momentum-driven moves". Confirm scope with Henry before changing — intel.md sets a permissive precedent here.
