Blocking
B-1 deposit-monitor/index.ts:756-761

dex_transfer idempotency check still queries created_at — the exact bug this PR fixes for deposits

Symptom

The deposit idempotency bug being fixed is: the query window is derived from transfer.timestamp (on-chain time), but created_at (wall-clock insertion time) is compared against it. When a late-detection path catches the event, created_at falls outside the window and the check returns empty. This PR correctly fixes this for the deposit idempotency check at lines 789-790. But the dex_transfer check immediately above it (lines 760-761) has exactly the same bug and was left unchanged.

Root cause

The dex_transfer lookup at line 760-761 still uses .gte("created_at", windowStart).lte("created_at", windowEnd). The window is computed the same way: transfer.timestamp ± 5 min. If trade-executor logged the dex_transfer row, its created_at is the wall-clock insertion time — which may be minutes away from the on-chain transfer time if there was any processing lag. Result: the dex_transfer check falsely returns nothing, the function classifies the internal margin shuffle as a real deposit, and the user gets a spurious deposit notification and an incorrect funding_events row.

Evidence
// Lines 755-761 in deposit-monitor/index.ts (unchanged by this PR):
const { data: dexTransfers } = await supabase
  .from("funding_events")
  .select("id, amount_usdc")
  .eq("user_id", userId)
  .eq("event_type", "dex_transfer")
  .gte("created_at", windowStart)   // <-- should be "completed_at"
  .lte("created_at", windowEnd);    // <-- should be "completed_at"

Note: unlike the deposit rows, dex_transfer rows ARE inserted with completed_at set — check trade-executor and the 20260407000002_dex_transfer_event_type.sql migration context. If they are, the fix is identical to the deposit fix.

Action

Change lines 760-761 to .gte("completed_at", windowStart).lte("completed_at", windowEnd) — same fix applied to the deposit check. Verify that trade-executor sets completed_at on dex_transfer inserts before deploying. If trade-executor does NOT set completed_at, document that here and use a different discriminator field.

W-1 deposit-monitor/index.ts:727-741 + migrations

Cron schedule is 1-56/5 (every 5 min), not every 10 min — lookback comment and value are inconsistent

Symptom

The code comment at line 728-732 says: "deposit-monitor runs every 10 min (pg_cron: 5-59/10)". The actual cron schedule — confirmed by reading every migration that touches deposit-monitor — is 1-56/5, which fires at :01, :06, :11, :16, :21, :26, :31, :36, :41, :46, :51, :56 — i.e. every 5 minutes, 12 times per hour. The last rescheduling migrations (20260403000001, 20260411000001, 20260507000001) all use 1-56/5. There is no migration setting the schedule to 5-59/10.

Root cause

The code comment cites a schedule string (5-59/10) that does not exist in any migration. The author appears to have misread or reconstructed the schedule from memory. The original bug diagnosis — "deposits in the second 5 minutes of any 10-min interval were permanently missed" — is therefore incorrect. The actual cadence is 5 minutes, not 10.

Consequence analysis

With a 5-min cron and a 12-min lookback, consecutive runs now overlap by 7 minutes (instead of the 2-minute overlap described). A deposit landing anywhere in that 7-minute overlap zone is visible to two consecutive cron runs. The idempotency fix (querying completed_at) is still correct and prevents double-insertion. However, the increase in overlap raises the rate of "second-pass idempotency check" DB reads from ~20% of runs to ~58% of runs — a significant increase in read load per deposit event, though still bounded by deposit frequency.

More importantly: if the original bug was really a 5-min lookback on a 5-min cron, there should have been no gap at all (lookback == cadence exactly covers the interval). The described bug may not have existed for the stated reason. The root cause of any real missed deposits was more likely the created_at idempotency bug.

Action

Verify the live cron schedule via SELECT jobname, schedule FROM cron.job WHERE jobname = 'deposit-monitor' on production before deploy. Update the code comment to reflect the actual schedule. Consider whether 12 min is still the right lookback given the 5-min cadence (7-min overlap is fine for correctness but is unnecessarily wide).

W-2 20260612000001_funding_events_deposit_dedup_index.sql

Unique index on (user_id, completed_at) will reject two legitimate same-second deposits

Symptom

The partial unique index is defined as UNIQUE (user_id, completed_at) WHERE event_type = 'deposit' AND completed_at IS NOT NULL. Two deposits arriving at exactly the same millisecond (or within the same timestamp resolution) for the same user would produce two rows with identical user_id and completed_at, which the index would reject as a duplicate. The DB would raise a unique-constraint violation, the insert would fail, and the second deposit would never be logged or notified.

How realistic is this?

HL's userNonFundingLedgerUpdates returns timestamps in millisecond precision (item.time is an integer ms epoch). Two external deposits landing at the exact same millisecond is extremely rare in practice. However, a HL-to-HL usdSend (type=send) from two different senders to the same recipient could share the same ms timestamp if they were batched in the same HL block. This index would then permanently suppress logging of the second deposit for that user, with no error surfaced to the user.

Action

Add amount_usdc to the index definition: UNIQUE (user_id, completed_at, amount_usdc) WHERE event_type = 'deposit' AND completed_at IS NOT NULL. This reduces the collision surface to same-user, same-second, same-amount, which is near-zero. It does slightly weaken the defense-in-depth guarantee against the exact-duplicate case, but the exact-duplicate case is already prevented by the application-level idempotency check. Alternatively, keep the current index and accept the theoretical risk, but document it explicitly.

W-3 deposit-monitor/index.ts:789-790 / refresh-balance/index.ts:316-317

completed_at IS NULL on existing rows silently bypasses the idempotency check

Symptom

The idempotency check queries .gte("completed_at", windowStart).lte("completed_at", windowEnd). In PostgreSQL (and PostgREST), comparisons against a NULL value return NULL (neither true nor false). PostgREST translates .gte("completed_at", x) to completed_at >= x, which evaluates to NULL when completed_at IS NULL. That row is excluded from the result. So if an existing deposit row somehow has completed_at = NULL, the idempotency check will not find it — and will insert a duplicate.

How can this happen?

The current code always sets completed_at: transferTime on insert, so new rows should never be null. But historical rows inserted before this fix (when the code ran successfully before any crash mid-insert) could theoretically be null if an insert was partially applied. More concretely: the migration adds a NOT NULL constraint only indirectly via the partial index (which excludes NULLs rather than preventing them). No ALTER COLUMN ... SET NOT NULL is added. So the column remains nullable at the DB level.

The partial unique index intentionally excludes NULL completed_at rows (a correct design choice for pending events). But this means the safety net does not apply to any row that slips through with a null completed_at.

Action

Verify that no existing funding_events rows of type 'deposit' with status 'completed' have completed_at IS NULL in production: SELECT COUNT(*) FROM funding_events WHERE event_type = 'deposit' AND status = 'completed' AND completed_at IS NULL. If any exist, backfill them before deploying the migration. Consider adding an explicit NOT NULL constraint on completed_at for completed deposit rows via a CHECK constraint or column constraint.

S-1 deposit-monitor/index.ts:789-790 / refresh-balance/index.ts:316-317

Core idempotency fix is logically correct

Both callers set completed_at = new Date(transfer.timestamp).toISOString() at insert time. The idempotency window is transfer.timestamp ± 5 min. Querying completed_at against this window correctly finds existing rows regardless of when they were inserted (whether by deposit-monitor, refresh-balance, or a late-detection path hours later). The fix closes the described bug precisely.

S-2 Both callers

Overlap-zone race between consecutive cron runs is handled correctly

With a 12-min lookback and a 5-min cron, any deposit in the 7-minute overlap zone is seen by two consecutive runs. The second run's idempotency check will find the row inserted by the first run (using completed_at), set alreadyLogged = true, and skip. No double-insert or double-notification. The fix is race-safe for the nominal case (one row written before the next run starts).

S-3 20260612000001_funding_events_deposit_dedup_index.sql

Migration is non-destructive and safe to apply to an existing table

CREATE UNIQUE INDEX on a live table takes a share lock and builds concurrently if issued as CREATE INDEX CONCURRENTLY (this is a non-concurrent build, so it takes ShareLock — acceptable for a low-traffic table like funding_events at current user count). The partial predicate (WHERE event_type = 'deposit' AND completed_at IS NOT NULL) limits the index scope. No existing rows should violate the uniqueness constraint assuming the application-level idempotency was working before this fix (each user had at most one deposit row per on-chain transfer). No schema reversals or data loss risk.
