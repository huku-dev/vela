Worth a look
renderNewsModule SUMMARY_CAP fallback still appends "…" — re-introduces ellipsis for any legacy clean summary > 150 chars
notify.ts:1869 — renderNewsModule() SUMMARY_CAP path
The stated goal is zero ellipsis in rendered output. capSummary() path B was updated to not append "…" so the render-side strip doesn't double-fire. But renderNewsModule() has its own SUMMARY_CAP (150) truncation path at line 1869 that still appends "…":

summaryText = (lastSpace > 120 ? slice.slice(0, lastSpace) : slice).replace(/[.,;:]\s*$/, "") + "…";

This path is the backstop for legacy news_cache rows whose stored summary exceeds 150 chars and does NOT end with "…" (so the ellipsis strip at 1857 doesn't fire). Those rows get truncated at the word boundary and rendered with a trailing "…". The fix eliminated "…" from write-time storage and from the ellipsis strip path, but not from this independent render-time truncation path. Any legacy row where the stored summary is a clean long sentence will ship with trailing ellipsis in the rendered email despite the fix. The practical impact depends on how many such rows exist — worth checking with SELECT COUNT(*) FROM news_cache WHERE length(summary) > 150 AND summary NOT LIKE '%…' AND summary NOT LIKE '%...' before deciding whether to remove the append.
Remove + "…" from notify.ts line 1869 to match the capSummary contract. The word-boundary truncation already produces clean output; the ellipsis append is the only remaining source of "…" in rendered digests.
classifyOne() in news-enrich has no diagnostic log when capSummary strips to null
news-enrich/index.ts:313 — classifyOne() after capSummary call
When capSummary(rawSummary) returns null (ellipsis input, no sentence stop found), classifyOne() writes summary: null to news_cache with no log entry. The parallel path in classifyNewsItem() (news-classifier.ts:332-337) was explicitly updated in this diff to emit "stripped to null (ellipsis, no sentence stop)" vs "trimmed N->M chars". classifyOne() was not given the same split. The news-enrich path processes the bulk of newly-ingested rows (every item that lands in news_cache and hasn't been classified yet) — it is the higher-volume path. Silent null-summary writes make it impossible in logs to distinguish "LLM returned no summary field" from "summary stripped to null due to ellipsis". Low impact on correctness, high impact on future debugging.
Add the same two-branch warn block to classifyOne() after line 313: log "stripped to null" when summary === null && rawSummary !== null, and "trimmed N->M chars" when summary !== null && summary !== rawSummary.
Nit
Consider unit tests for the capSummary strip behavior
_shared/news-classifier.ts — capSummary() export
capSummary is exported and has non-trivial branching: ellipsis-strip path (returns null or shorter string), pass-through path, length-cap path. No behavioral tests appear to exist for it. Six distinct cases are exercised by this review: "creating strong…" returns null; "Fed holds rates. Inflation sticky…" truncates to first stop; "..." behaves like "…"; no-ellipsis passthrough; bare "…" returns null; long clean string caps at word boundary. A news-classifier.test.ts with assertions for these cases would catch regressions on future edits to the strip logic — the logic is subtle enough (regex, Math.max, slice(0, lastStop+1)) that a mis-edit is plausible.
