Blocking

1. Root cause diagnosis is wrong: the cron runs every 5 minutes, not every 10 minutes.
deposit-monitor/index.ts line 14 comment; migrations/20260403000001_reduce_disk_io.sql; migrations/20260411000001_fix_pgnet_timeouts.sql; migrations/20260507000001_cron_shared_secret_header.sql

The diagnosis states the cron schedule is 5-59/10 * * * * (every 10 minutes). The actual schedule in every migration that sets it is 1-56/5 * * * * (fires at :01, :06, :11 ... every 5 minutes). This has been the schedule since at least 20260403000001 and has been re-applied unchanged in 20260411000001 and 20260507000001. The function header comment confirms it: "Schedule: pg_cron every 5 minutes (see migration 20260411000001)".

The correct diagnosis is therefore: the cron runs every 5 minutes, the lookback window is 5 minutes, and the two values should match — but they do not in practice because the cron fires at the scheduled minute, the ledger data is fetched at some wall-clock time after the HTTP round-trip, and Date.now() inside filterRealInboundTransfers is evaluated at fetch time, not at cron schedule time. The effective window is closer to 4.5–4.8 minutes depending on function cold-start and multi-wallet parallelism. Any deposit that arrives in the 0–30 second gap between the last window's close and the next run's execution will be permanently missed.

The 5-to-10-minute window mismatch cited in the brief does not exist. The real mismatch is smaller (sub-minute) but still structural. Before writing any code, the actual gap must be measured from function logs to confirm how wide it really is.

Action: verify the real cron expression in production via SELECT jobname, schedule FROM cron.job WHERE jobname = 'deposit-monitor'. Do not write code until the actual gap is established from evidence, not assumption.
2. The proposed fix's 12-minute window is sized against the wrong cadence and overshoots by 2.4x.
deposit-monitor/index.ts line 733–737 (proposed change site)

The fix proposes 12 minutes on the assumption the cron runs every 10 minutes. The actual cadence is 5 minutes. A 12-minute window on a 5-minute cron means every consecutive pair of runs shares a 7-minute overlap region. Every deposit in that region will be seen by two runs. The idempotency guard (checked below) is the sole protection against double-notification.

More concretely: if the correct fix for a 5-minute cron with sub-minute execution jitter is 6 minutes (5 min + 1 min margin), the proposed 12-minute window is twice the necessary size. This is not just waste: it increases the surface area for idempotency to fail (two concurrent runs that both start within the same overlap region could both pass the alreadyLogged check before either inserts its row, as Supabase has no advisory lock here).

The comment in 20260403000001 that documented the old 2-minute cadence explicitly called out "The 5-min ledger lookback window still overlaps consecutive runs" when they moved from 2 min to 5 min — so the team has already accepted one overlap. Going to 12 minutes doubles the overlap region for no corresponding benefit.

Action: if the real gap is sub-minute jitter, the correct fix is 6 minutes (not 12). If the real gap is something else entirely, fix the root cause, not the symptom. Do not use 12 minutes without fixing the cadence assumption first.

Worth a look

3. Idempotency is NOT race-condition-safe when two runs overlap on the same deposit.
deposit-monitor/index.ts lines 775–788 (idempotency check); refresh-balance/index.ts lines 308–321

The idempotency check is a read-then-write with no locking:

const { data: existing } = await supabase.from("funding_events").select(...)...
const alreadyLogged = existing.some(e => ...)
if (alreadyLogged) continue;
await supabase.from("funding_events").insert({ ... })

If deposit-monitor and refresh-balance race (both read existing as empty, both proceed to insert), or if two deposit-monitor runs start within the overlap window and both fetch ledger data before either writes, both will insert and both will notify. Any proposed fix that increases the overlap window (including the proposed 12-minute value) increases the probability of this window being open simultaneously across two runs.

The dex-transfer check at lines 752–769 has the same race: two concurrent runs could both pass the isDexTransfer check on the same transfer.

This is not a new bug introduced by the fix, but the fix's larger overlap window elevates it from theoretical to practically likely during the first run after a longer window is deployed.

Mitigate by adding a unique constraint on (user_id, event_type, completed_at, amount_usdc) on funding_events and using INSERT ... ON CONFLICT DO NOTHING. The application-layer check can stay as a fast-path skip.
4. refresh-balance idempotency window is also wrong if the deposit detection delay is fixed.
refresh-balance/index.ts lines 304–314

refresh-balance correctly uses a 30-minute lookback window but its idempotency check uses a hardcoded ±5 minutes around the transfer timestamp:

const windowStart = new Date(transfer.timestamp - 5 * 60_000).toISOString();
const windowEnd   = new Date(transfer.timestamp + 5 * 60_000).toISOString();

deposit-monitor uses the same ±5 minute window for its idempotency check (lines 744–746). This is not a lookback into the past — it is a tolerance window to match the created_at of existing funding_events rows against the on-chain transfer.timestamp. The logic is: "was a deposit of this amount logged within 5 minutes of when the transfer happened on-chain?"

This is conceptually separate from the lookback duration and is correct as long as both callers insert with completed_at = transfer.timestamp (they do). However, if one caller somehow stores a deposit row with a created_at that is more than 5 minutes offset from transfer.timestamp — for example, due to clock skew or a retry path that re-inserts with a later created_at — the dedup check will miss it and double-notify.

Low urgency but worth noting. The fix does not change this logic, so it does not make it worse. Flag for a future migration to add the unique constraint.

Nit

5. Tests hardcode 5 * 60_000 and will silently pass regardless of what the fix ships.
_shared/deposit-monitor.test.ts lines 324, 365

The test suite calls filterRealInboundTransfers(ledger, wallet, 5 * 60_000) directly and asserts on filter results. These tests exercise the filter logic correctly but do not test the value that deposit-monitor/index.ts passes as the lookback argument. If the constant in index.ts is changed to 12 minutes (or any other value), these tests will still pass — they do not catch a regression where the constant is wrong.

There is also a structural test at line 249 that checks the source code contains "filterRealInboundTransfers" (import verification), but no test that asserts the actual numeric argument passed at the call site.

After the fix, add a source-scan test: assert(src.includes("12 * 60_000"), "lookback must be 12 minutes") (or whatever the final value is). Similar to the pattern already used for the dex-discriminator check at line 280.

refresh-balance is user-triggered and already uses a 30-minute lookback — it does not need updating. deposit-monitor is the only cron caller. No other edge functions import the filter.

S3. The first-sync guard and partial-read guard correctly fence deposit detection.

The !wallet.balance_last_synced_at guard (first-ever run) and sync.partialRead guard (incomplete HL API response) both return early before detectAndNotifyDeposits is called. Widening the lookback window does not affect these guards — they remain correct regardless of the lookback value.

migrations/20260403000001 (first set to 5min), 20260411000001, 20260507000001 (both preserve 1-56/5)

The schedule expression 1-56/5 * * * * has been set consistently across three separate migrations and never changed back. The 20260528000002 migration only disables it on staging (active=false), does not reschedule it. Production cadence is definitively 5 minutes, not 10.
