Blocking — B1
ALTER TABLE on extension-owned tables will fail hard in Supabase managed Postgres
Proposed fix — Step 2 (no existing migration reference)

Symptom: The migration will throw ERROR: must be owner of table _http_response (or similar) when it reaches the ALTER TABLE net._http_response statement. The pg_net extension creates and owns net._http_response under a dedicated extension owner role, not the postgres role. Supabase's managed Postgres does not grant postgres ownership of extension-internal tables. The same applies to cron.job_run_details, which is owned by the pg_cron extension.

Root cause: The diagnosis assumes postgres has ALTER TABLE privilege on extension-owned tables. That is true on self-hosted Postgres (where postgres is superuser), but Supabase's managed instances restrict this. The postgres role is a privileged role but not a true superuser; it lacks ownership of extension-catalog tables. The May 26 migration (lines 12-13) actually asserted "execute as the postgres role (which can VACUUM any table, including pg_net-owned ones)" for VACUUM — that claim was directionally correct for VACUUM (which is a special command, not DML) but does not carry over to DDL like ALTER TABLE. The ownership check for DDL is stricter.

Failure mode: Not silent. Supabase migrations run in a transaction; the ALTER TABLE failure rolls back the entire migration, meaning the cron reschedules in steps 1 also roll back. The crons stay at the broken May 26 state (with VACUUM) and continue failing every 6 hours.

Evidence gap: This can be confirmed cheaply before deploying: run SELECT tableowner FROM pg_tables WHERE tablename = '_http_response' AND schemaname = 'net' on production. If the owner is not postgres, the ALTER TABLE will fail.

Fix: Wrap the ALTER TABLE statements in a BEGIN ... EXCEPTION WHEN insufficient_privilege OR not_owner THEN RAISE NOTICE '...'; END guard to make them advisory (degrade gracefully) rather than migration-breaking. Or drop them from this migration entirely and address autovacuum tuning via Supabase dashboard storage settings, which does have the required privilege pathway. The cron reschedule (step 1) is the real fix; don't let step 2 nuke it.
Blocking • B2
No idempotency guard on cron.unschedule: will hard-fail on fresh branches / staging re-runs
Proposed fix — Step 1, all four unschedule calls

Symptom: On a branch where the May 26 migration has not run (e.g. a freshly seeded staging DB, a new Supabase preview branch, or a CI test environment), cron.unschedule('cleanup-net-responses') will throw because the job name does not exist. pg_cron's unschedule raises an error on unknown job names.

Contrast with past practice: The original Apr 3 migration (lines 33-34) correctly wrapped unschedule calls with exception handlers: 

BEGIN PERFORM cron.unschedule('deposit-monitor'); EXCEPTION WHEN OTHERS THEN NULL; END;

The May 2 migration (lines 25, 33) and the proposed fix drop this safety pattern entirely, using bare SELECT cron.unschedule(...). The May 2 migration got away with it because the jobs were always present after Apr 3 ran. The proposed fix is operating in a context where May 26 may or may not have run.

Root cause: No pg_cron equivalent of DROP CRON IF EXISTS. The workaround is the exception wrapper shown above, which already exists in this codebase.

Fix: Wrap all four cron.unschedule calls using the same exception handler pattern from the Apr 3 migration (lines 33-34). This is a one-liner per call and already the established project pattern.

Worth a look • W1
The "autovacuum never triggered" claim in the May 26 migration may be partially right — and autovacuum tuning alone may not be enough
20260526000001_cleanup_crons_append_vacuum.sql, lines 5-8

The May 26 migration explains bloat as: "new inserts consume dead tuple space before autovacuum samples n_dead_tup, so the threshold is never reached." This is a real mechanism. If rows are inserted fast enough that they fill newly freed slots before autovacuum's n_dead_tup sampling window, the table's pg_stat_user_tables.n_dead_tup counter stays low, and autovacuum never fires.

The problem with threshold=50: Even if autovacuum fires after every 50 dead tuples, it only reclaims space within pages for reuse — it does not shrink the file. If net._http_response grew to 9.8 MB before the manual VACUUM FULL, it will grow back to a similar level if the write pattern that caused the bloat hasn't changed. A regular VACUUM updates the Free Space Map (FSM) so new inserts reuse freed pages, but this only works if inserts happen after the VACUUM. Under high write rates where inserts precede any VACUUM cycle, new pages are allocated anyway.

Practical question not answered: What is the actual insert rate to net._http_response? The Apr 3 migration notes the table was 26 MB before the daily cleanup was added. If inserts outpace the DELETE window (1 day retention, 6h cleanup), the table will grow regardless of autovacuum tuning. The diagnosis doesn't address whether the retention window is appropriately sized for the write rate.

Verify: Query SELECT count(*), min(created), max(created) FROM net._http_response on production to confirm the 1-day retention window keeps the table bounded. If the row count at cleanup time is consistently above a few thousand, the bloat mechanism is write rate, not autovacuum tuning, and the fix is wrong.
Worth a look • W2
cron.unschedule + cron.schedule orphans job_run_details history rows
Proposed fix — Step 1 (both cron reschedules) / project MEMORY.md

Per project notes, cron.unschedule + cron.schedule creates a new job with a fresh jobid. This is the pattern used in May 2 (lines 25-38) and the proposed fix. The cron.job_run_details table stores historical run records keyed by jobid.

When the old jobid is removed and a new one created, the existing job_run_details rows for the old jobid remain in the table permanently (or until the 7-day DELETE catches them), but they now have a jobid that no longer exists in cron.job. This is cosmetic for the cleanup jobs specifically, but it means the history for cleanup-cron-history itself shows an unexplained gap after the migration.

More practically: if any monitoring or alerting relies on querying job_run_details by joining to cron.job on jobid, those rows become invisible. The vela-debug skill checks signal freshness via cron history. If it joins by jobid, the cleanup-cron-history entries will appear to have never run successfully before this migration.

Verify that any cron health checks query by job name (via JOIN to cron.job on jobname), not by raw jobid. If they join on jobid, the history discontinuity will produce false alarms.
Worth a look • W3
autovacuum_vacuum_threshold = 50 on cron.job_run_details may cause autovacuum contention
Proposed fix — Step 2, second ALTER TABLE block

cron.job_run_details accumulates ~500 rows/day (conservative estimate from the Apr 3 note: 48 MB, 61k rows after a period without cleanup). At threshold=50, autovacuum will fire roughly 10 times per day based purely on DELETE churn from the cleanup cron. But it also fires from INSERT churn: every cron execution inserts a row. With 40+ crons running at varying frequencies, the insertion rate could be several hundred rows per hour.

Autovacuum on cron.job_run_details with threshold=50 could mean autovacuum runs dozens of times per day on this table. Each run acquires a ShareUpdateExclusiveLock and scans the table. While this won't block reads or writes, it consumes autovacuum worker slots (default 3 on Supabase) and could delay autovacuum on other high-churn tables (like net._http_response itself). The diagnosis notes this as "IO-positive" but does not account for autovacuum worker slot contention.

Caveat: This is unlikely to be a severe problem in practice, but on a Supabase instance already flagged for high Disk IO, adding frequent autovacuum passes on two tables is worth verifying post-deploy.

Monitor autovacuum frequency post-deploy via SELECT relname, last_autovacuum, n_dead_tup FROM pg_stat_user_tables WHERE relname IN ('_http_response', 'job_run_details') ORDER BY last_autovacuum DESC. If autovacuum is thrashing, raise the threshold to 200.

Safe • S1
Root cause diagnosis is correct: VACUUM cannot run in a transaction block

The VACUUM failure is correctly diagnosed. pg_cron does wrap multi-statement commands in a transaction block, and VACUUM cannot execute inside a transaction. This is a hard PostgreSQL constraint. The May 26 migration comment (lines 12-13) was wrong to assert otherwise. DELETE-only is the right approach for the cron command itself.

Safe • S3
VACUUM FULL is correctly treated as a one-time emergency op, not a cron concern

The diagnosis correctly distinguishes regular VACUUM (marks dead tuples for reuse, no file shrink) from VACUUM FULL (rewrites the table, shrinks the file, requires exclusive lock). Not attempting to schedule VACUUM FULL via cron is correct. The baseline was already reset by the manual May 26 VACUUM FULL — the cron just needs to keep the table from re-bloating.

Direct answer: Q5 — Did DELETE-only demonstrably fail before, and does autovacuum tuning actually change the outcome?

Did DELETE-only fail? The evidence is ambiguous. Here is what the migrations actually record:

DateStateOutcome
Apr 3DELETE-only, daily cleanupnet._http_response grew to 32 MB by May 2 (29 days). This is 26 MB -> 32 MB, so bloat worsened but incrementally.
May 2DELETE-only, 6-hourly cleanupMay 26 migration says table re-bloated to 9.8 MB in "8 days after the May 18 VACUUM FULL." That is May 18-26, 8 days, while DELETE-only was running every 6h.
May 26DELETE + VACUUM (broken)VACUUM errors, cron partially failing. State unknown.

So yes: DELETE-only with 6-hourly runs demonstrably did not prevent re-bloat during May 18-26. The table grew 9.8 MB in 8 days despite cleanup every 6 hours.

Does autovacuum tuning change the outcome? Possibly, but only if the May 18-26 bloat was caused by autovacuum failing to fire (dead tuple recycling). If the bloat was caused by new page allocation outpacing VACUUM's FSM updates, threshold=50 does not help at all — autovacuum can still only reclaim pages, not shrink the file. Given the table grew 9.8 MB in 8 days with 6-hourly DELETEs, the write rate into net._http_response is high enough that each DELETE cycle creates substantial dead tuple churn. Whether autovacuum at threshold=50 can keep pace with that churn is unverified.

The honest answer: The proposed fix is the right fix for the VACUUM error (step 1: DELETE-only crons). The autovacuum tuning (step 2) is a reasonable hypothesis for preventing re-bloat, but it failed silently once before (May 18-26) without the tuning, and there is no evidence that threshold=50 will be sufficient if the write rate is the bottleneck rather than autovacuum sampling. Additionally, step 2 may hard-fail on Supabase's managed Postgres due to extension ownership (B1 above).

Recommended approach: Deploy step 1 (cron fix) unconditionally. Test step 2 (ALTER TABLE) in a sandbox first by confirming table ownership. If ALTER TABLE is blocked, the alternative path is to accept that re-bloat will require periodic manual VACUUM FULL operations, and schedule a reminder rather than attempting to automate it via autovacuum tuning on extension-owned tables.
