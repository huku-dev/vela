1. Migration resets active=true on staging — cleanup cron re-enables itself

20260528000001_expire_executing_proposals.sql:10-27

The project MEMORY.md records a hard-won rule: "cron.unschedule + cron.schedule resets active=true: Any migration that reschedules an existing cron creates a fresh job with active=true, wiping prior disabled state." expire-pending-proposals is explicitly listed in 20260527000001_staging_disable_llm_crons.sql as a cron that is safe to leave active on staging — but the comment on line 12 of that migration says the same pattern has already caused problems with other crons.

The real question: does this migration need a staging re-disable? No — the staging cron hygiene file correctly marks expire-pending-proposals as safe to run on staging (it hits no LLM APIs). So there is no LLM quota bleed. However, the new schedule changes from '5 * * * *' (1/hr) to '5,35 * * * *' (2/hr). The migration resets the job to active=true without checking the prior disabled state via cron.alter_job, and — crucially — the job was enabled on staging to begin with (per the exemption list), so no disabled state is lost here.

The actual blocking issue is different: the cron.unschedule on line 10 is called as a bare SELECT, not wrapped in a BEGIN ... EXCEPTION WHEN OTHERS THEN NULL; END; block. The previous cron was defined by 20260403000002_consolidate_crons.sql. If for any reason the job was manually deleted on an environment (or migration order differs), cron.unschedule('expire-pending-proposals') on a non-existent job raises an exception and aborts the migration, leaving the executing cleanup UPDATE never scheduled. All subsequent migrations in the same transaction also roll back. Compare to 20260403000002 line 96 and 20260307000001 lines 38-43 which all use the safe BEGIN PERFORM cron.unschedule(...); EXCEPTION WHEN OTHERS THEN NULL; END; pattern. This migration skips the defensive wrapper.

Fix: wrap the unschedule in a BEGIN...EXCEPTION block identical to the pattern in 20260307000001 lines 38-43:
BEGIN PERFORM cron.unschedule('expire-pending-proposals'); EXCEPTION WHEN OTHERS THEN NULL; END;
Then call SELECT cron.schedule(...) (or PERFORM inside a DO block). Without this, any environment where the cron name differs (e.g. a fresh branch DB) will abort the migration entirely.

2. Failed proposals missing from tradedAssets — asset filter hides them unexpectedly

src/pages/TrackRecord.tsx:70-79 (tradedAssets derivation)

The tradedAssets array (line 70-79) is built from positions (open) and closedPositions. It drives the asset filter dropdown. Failed proposals are now rendered in the "Failed trades" section, but their asset_ids are never added to tradedAssets.

Concrete failure scenario: a user has zero positions but two failed proposals (e.g. a GOLD trade and an ETH trade that both timed out). The filter dropdown does not appear (because tradedAssets.length > 1 is false — there are no positions). All failed proposals display correctly. So far, fine. But if the user does have positions for only one asset (e.g. BTC), and failed proposals for a different asset (GOLD), the dropdown only shows "BTC". The user selects BTC. recentFailedProposals at line 84-89 correctly filters to p.asset_id === selectedAsset — so GOLD failures disappear silently. The user has no way to see why their GOLD trade failed because GOLD never appears as a filter option.

Additionally: the hasUserTrades guard at line 137 does not include recentFailedProposals.length > 0. If a user has only failed proposals and no positions or pending proposals, the entire hasUserTrades block returns false and the empty state renders ("When you approve your first trade..."), even though failed trades exist. The "Failed trades" section is inside the truthy branch of hasUserTrades.

Fix 1: include recentFailedProposals asset IDs in the tradedAssets derivation loop.
Fix 2: add || recentFailedProposals.length > 0 to the hasUserTrades expression at line 137 — but only after computing recentFailedProposals without the selectedAsset filter (compute a separate allRecentFailedProposals without the asset filter for the guard, and use the filtered version for rendering).

WORTH A LOOK

3. execUpdate Supabase error silently falls through to position creation

trade-executor.ts:1275-1282

The destructured form const { data: execUpdate } = await supabase... discards the error field. If the Supabase client throws a network error or RLS violation, execUpdate is undefined. The guard !execUpdate?.length evaluates to true (undefined is falsy), so the function bails correctly and does not create a position — which is the safe behavior.

However, the bail path fires a Sentry alert with reason cron_claimed, which would be misleading: Sentry would show "proposal claimed by cleanup cron" when in reality the DB query failed. This creates a false signal in incident triage. The distinction matters: a cron_claimed event means "HL order may exist without a Vela position" (requires manual reconciliation); a DB error means "DB write failed, recheck if proposal is truly orphaned."

Destructure the error: const { data: execUpdate, error: execUpdateErr } = await supabase.... If execUpdateErr is set, fire a Sentry alert with a distinct tag (reason: "db_error", not cron_claimed) and bail. The bail path and position-skip behavior stay the same; only the Sentry context changes.

4. Orphaned HL fill scenario has no mitigation path documented

trade-executor.ts:1283-1298

When the bail path fires, the code logs "HL order may exist on exchange" and returns without creating a position. This is correct behavior to prevent a phantom Vela position. But there is no follow-up path: no admin notification beyond Sentry, no proposal status column that distinguishes "failed by cleanup cron after confirmed HL fill" from "failed by cleanup cron on a legitimately dead proposal." Both write the same error_message = 'Execution timed out. No position created.' (from the cron) with no record that the fill actually happened.

The user sees "Failed: Execution timed out. No position created." in the UI. If they actually have an open position on HL (because the fill completed), they now have a real position with no Vela record — no stop-loss, no monitoring, no fee tracking. Sentry fires, but the Sentry alert alone is not an action path. The executor's bail return sets success: false which would normally not create a position — but the user's HL account still has one.

Document a runbook step in the Sentry alert body or docs/runbooks/: when a cron_claimed Sentry alert fires, an admin must check the HL position API for that user+asset and manually reconcile (insert a position row or close on exchange). Without this, the "HL order may exist" comment is the only trace.

5. Two-instance double-fire: cleanup UPDATE is not idempotent in effect

20260528000001_expire_executing_proposals.sql:20-26

The cleanup UPDATE is safe against double-execution in isolation: if two cron instances run simultaneously, both attempt to UPDATE WHERE status='executing'. The first one wins and sets status='failed'. The second finds zero rows matching (status is now 'failed') and silently updates zero rows. No data corruption. This is idempotent.

The concern is a tighter race: the cron fires and sets status='failed' on proposal X at time T. The executor is concurrently at step 11 doing the conditional UPDATE WHERE status='executing'. The executor's UPDATE returns 0 rows (cron already claimed it). Executor bails correctly. But the executor had already successfully placed the HL order at step 10 (fill confirmed). The cron's error_message "Execution timed out. No position created." is now factually wrong: position creation was blocked not because of a timeout but because of a race with an order that actually filled. This is the same orphaned-fill scenario as finding #4, but the trigger condition here (cron fires at exactly T+5min while executor is between step 10 and step 11) is more likely than it sounds: at 150s Deno wall-clock limit, a 5-minute cron window is generous, but a slow HL RPC (step 10 takes 140s) plus step 11 DB write could land in the window.

Consider tightening: before the executor's step 10 order placement, do a preflight check that the proposal is still in 'executing' status (the proposal was set to 'executing' earlier in the flow). If the cron has already claimed it by then, abort before placing the order — not just after. This is a belt-and-suspenders add, not a blocker.

6. TradeProposalCard: error_message for "failed" status uses generic fallback when cron sets the message

src/components/TradeProposalCard.tsx:44, 248-251

getStatusConfig returns description: 'Execution failed. Check your balance and try again.' for status === 'failed'. The card renders proposal.error_message when it exists (line 248-251), so timeout-failures will show "Execution timed out. No position created." — good.

However, the generic description "Check your balance and try again" is shown as a fallback when error_message is null. For proposals failed by the cleanup cron, the cron always sets error_message (line 22-23 of migration). For other failure paths in the executor, error_message is also always set (line 1196-1206). So in practice the fallback should never fire. But proposals that were manually force-failed via SQL (e.g. admin tooling) with no error_message would show "Check your balance and try again" — potentially misleading for a timeout or infra failure. Low risk, but worth flagging since the description is now reachable by the new "Failed trades" section.

Change the fallback description from "Check your balance and try again." to something direction-neutral: "This trade did not execute. Contact support if your balance was affected." This avoids blaming the user's balance for an infra failure.

NIT

7. Migration does not add semicolon after second UPDATE inside the cron body

20260528000001_expire_executing_proposals.sql:20-26

The $$ body contains two UPDATEs. The first ends with a semicolon (line 18). The second UPDATE (lines 20-26) has no trailing semicolon before $$. PostgreSQL allows this (the final statement in a function body doesn't require a terminal semicolon), and pg_cron runs each statement individually. But it's inconsistent with the first statement and could confuse future editors.

Add a semicolon after the closing WHERE clause on line 26: AND updated_at < now() - interval '5 minutes';

8. No test covers the new execUpdate?.length guard path

trade-executor.ts:1282-1299 (bail path)

The existing 110 trade-executor tests were passing before this change. The new bail path (cron claimed the proposal) is a new code branch with no coverage. A test that mocks the Supabase .update().eq().eq().select() chain to return { data: [], error: null } should assert: (a) no position row is inserted, (b) Sentry is called with reason: cron_claimed, (c) the function returns success: false with no positionId.

Add one behavioral test: "returns success:false and skips position creation when proposal claimed by cleanup cron". This is the regression case the fix introduces.
