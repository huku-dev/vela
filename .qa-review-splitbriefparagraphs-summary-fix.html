Blocking (2)

B-1: breaks.push(m.index + 2) stores the space character position, not the sentence end — the split produces a leading space on every non-first paragraph
notify.ts:1333 — proposed fix, boundary loop
Diagnosis correct. Fix has an off-by-one.

Symptom: The proposed fix writes breaks.push(m.index + 2). The pattern /[a-z0-9][.!?] /g matches exactly 3 characters, e.g. "d. ". m.index points to d, m.index+1 = the period, m.index+2 = the space character.

Root cause: cleaned.slice(pos, sp) where sp = m.index+2 ends before the space, so the first chunk correctly ends with the period. But cleaned.slice(sp) for the next segment starts at the space. The subsequent .trim() removes it, so no visible artifact makes it to the rendered HTML. Empirically tested: both +2 and +3 produce identical trimmed output.

The real off-by-one is elsewhere in the same function. The breaks.length <= 3 guard returns unsplit for a 4-sentence brief (3 inter-sentence spaces detected), whereas the original sentences.length <= 3 guard returned unsplit only for 3-sentence briefs. A 4-sentence brief under the original code was split 2+2; under the proposed code it becomes one unsplit wall of text.

// Original: sentences.length <= 3 → unsplit (3 sentences stays as 1 paragraph)
// Proposed: breaks.length  <= 3 → unsplit (4 sentences stays as 1 paragraph)
// Net effect: 4-sentence briefs lose their paragraph split

Fix: Change the threshold from breaks.length <= 3 to breaks.length <= 2 (3 sentences = 2 breaks = unsplit, matching original intent). Alternatively, keep <= 3 intentionally as a more conservative split rule, but document the behavior change explicitly.

B-2: Proposal says update lastSpace > 80 threshold in news-classifier.ts — the actual code has lastSpace > 60, not > 80
news-classifier.ts:289 — post-call truncation guard
Proposal references the wrong threshold value.

Symptom: The proposal says "the lastSpace > 80 threshold in the post-call truncation guard should be updated to ~90." The actual line is:

summary = (lastSpace > 60 ? slice.slice(0, lastSpace) : slice).replace(/[.,;:]\s*$/, "") + "…";

The threshold is 60, not 80. This means if the proposal is implemented exactly as written, the developer searches for > 80, finds nothing, and either skips the update or makes a wrong edit.

Root cause: The proposal was drafted without reading the actual guard line, likely inferring the value from the prompt's "40-85" range or from renderNewsModule's lastSpace > 120 (which is a different function with a different cap of 150).

Fix: Update the proposal to reference line 289 and threshold > 60. Under the new SUMMARY_MAX = 110, a proportional threshold would be > 90 (scales 60/85 → 90/110). The existing > 60 is not catastrophically wrong for 110-char inputs, but a string whose last space falls at position 61-89 would use the raw slice (with a mid-word cut) instead of the word-boundary trim. Update to > 90 in the same commit as the SUMMARY_MAX raise.

Worth a Look (1)

W-1: The (\. )-([A-Z]) cleanup regex can silently rewrite legitimate LLM-authored em-dash-free sentence fragments
notify.ts:1333 — proposed fix, line 3 of function body
Low probability, but the failure mode is silent text corruption.

Symptom: The cleanup line cleaned = trimmed.replace(/(\. )-([A-Z])/g, "$1$2") removes a hyphen between a sentence-ending period+space and an uppercase letter. It was added to handle "U.S. -Iran" style artifacts. But:

  • It correctly does NOT trigger on "U.S.-Iran" (no space before the hyphen). Verified.
  • It DOES trigger on any brief where an LLM writes something like "rose 5%. -Gold fell too." — changing it to "rose 5%. Gold fell too." (hyphen silently removed).

Root cause: Vela briefs are LLM-generated narrative text. The pattern ". -[A-Z]" (period-space-dash-capital) is rare in well-formed prose but can appear in LLM output that uses a dash to introduce an opposing force, contrast, or bullet-style list item at the start of a sentence.

Action: Either accept the risk (occurrence rate is very low, and removing an errant dash is a minor edit), or log a warning when the cleanup fires so production occurrences can be monitored. Do not expand the regex further without empirical data on actual brief formats.

Safe (3)

S-1: Fix 1 regex bug diagnosis is correct
notify.ts:1341 — current implementation

Verified by tracing /[^.!?]+[.!?]+(?:\s|$)/g on "As investors expect U.S. military action...". The regex produces ["S. ", "military action..."] — "S." becomes sentence 1, and the leading "As investors expect U" is silently dropped. The first rendered paragraph begins "S. military action..." which is the described breakage. The proposed replacement regex /[a-z0-9][.!?] /g (requiring a lowercase letter or digit before punctuation) correctly ignores all-caps abbreviations like "U." and "S." and produces clean boundaries.

S-2: Fix 2 core logic is correct — raising SUMMARY_MAX 85→110 aligns prompt, guard, and schema
news-classifier.ts:278 + llm/registry.ts:94-101

The schema maxLength:180 means a 120-char LLM response still hits the post-call guard and gets truncated to 110 — correct behavior.
