W1. news-detail-generate writes to news_cache.summary without calling capSummary
news-detail-generate/index.ts:266, 397-398 | news-enrich/index.ts (W1 completeness claim)
The PR brief describes W1 as "the cron path was NOT calling capSummary." The fix correctly adds it to classifyOne() in news-enrich. But a second write path exists: news-detail-generate runs the news_summary LLM task (targeting "3-4 sentences, 80 words") and writes the result to news_cache.summary at line 266 with no length cap. The generateSummary() function at line 397 returns the raw string directly. An 80-word summary averages roughly 450-500 characters, well above SUMMARY_MAX (110). The render-side SUMMARY_CAP=150 in renderNewsModule will truncate these rows before display, so no email breakage — but the stored value exceeds the new invariant.

This is pre-existing (not introduced by this PR), and the news_summary task is intentionally richer than the news_classify one-liner summary. However, the claim that W1 is fully closed is overstated. Decide: either document that news-detail-generate is intentionally uncapped (different use case, relies on the render-side backstop), or add a note to the PR/commit. No code change required if the distinction is intentional.
W2. Test verifies capSummary and rawSummary appear in source separately — not that capSummary(rawSummary) is the actual call pattern
news-enrich/news-enrich.test.ts:191-197
The new test does:
assertStringIncludes(SRC, "capSummary");
assertStringIncludes(SRC, "rawSummary");

Both strings are present in the source — test passes. But the test would also pass if someone wrote capSummary("other string") and then used rawSummary uncapped. The assertion does not verify that capSummary is called with rawSummary as its argument. A stronger check:
assertStringIncludes(SRC, "capSummary(rawSummary)");
This is a one-token change to the test that closes the gap. Current code is correct (the call at line 313 IS const summary = capSummary(rawSummary)), but the test doesn't guard against a future regression where the call is replaced.

Nit (1)

N1. capSummary can return SUMMARY_MAX+1 chars (111) when no spaces exist in the truncated slice
_shared/news-classifier.ts:255-259
When raw.length > 110 and the 110-char slice has no spaces (or the last space is at index <= 90), the function falls back to slice.replace(/[.,;:]\s*$/, "") + "…". If no trailing punctuation is stripped, the result is 110 chars + "…" (U+2026, 1 char) = 111 chars — one over SUMMARY_MAX.

In practice this never fires with LLM output: single-sentence summaries always contain spaces. The render-side 150-char backstop in renderNewsModule absorbs it. Consider changing slice.slice(0, SUMMARY_MAX - 1) + "…" in the hard-cut branch to make the invariant strict. Low priority.
