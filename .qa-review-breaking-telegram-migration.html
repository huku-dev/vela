Blocking

[B1] brief-generator recordCall uses asset symbol as taskName
_shared/brief-generator.ts : ~423, ~443
Symptom

Every recordCall in callClaudeAPIWithRetry passes label as the taskName. label is the asset symbol (e.g. "BTC", "ETH-text"), not a TaskName enum value. The task_name column in llm_call_log will accumulate one row per symbol per run with a garbage key, making every cost query, budget alert, and per-task audit for brief_generate return zero rows. The records aren't rejected (column is text), so there's no error signal.

Root cause

The callClaudeAPIWithRetry function predates the factory and has its own label parameter used for logging, but recordCall's taskName field is meant to be a TaskName value. The two concepts were conflated when the logging call was inserted.

Fix

Add a taskName: TaskName parameter to callClaudeAPIWithRetry (alongside label), and pass the actual task name from callers: "brief_generate" for the main brief path and "brief_generate" (or a new "brief_events" task if you want separate telemetry) for the Haiku events path. Pass label only to the console lines. Alternatively, derive the task name inside the function by mapping model to a known task name, but the parameter approach is simpler and doesn't require a new mapping table.

Worth-a-look

[W1] schema_fail cascade exhausted: result.parsed is null, no admin alert fires
breaking-news/index.ts : 391–408
Symptom

When both Groq and NVIDIA return schema_fail and cascadeOnSchemaFail: true is set, the factory throws a synthetic retryable error from the primary, cascades to the fallback, and then the fallback itself may also return schema_fail. Reading factory.ts lines 139–141: the fallback result is returned as-is (return { ...res, fallbackUsed: true }) without checking res.status === "schema_fail" again. So when the fallback returns schema_fail, complete() returns (does not throw), and result.parsed is null (or whatever the adapter set it to on a failed parse). The try/catch at line 391 does not catch this. Control falls through to the if (!parsed) check at line 406, which logs a console error and returns null with no admin alert. A broken Telegram analysis silently drops the broadcast.

Root cause

The cascade in factory.ts only re-checks schema_fail on the primary result (line 114). Fallback and lastResort results are returned verbatim. If fallback returns schema_fail (and there's no lastResort), the caller gets a schema_fail response with parsed: null silently. The same issue would occur for any task where both providers fail schema validation.

Fix

After the if (!parsed) guard at line 406, add an admin alert before returning null: await sendAdminTelegram(`[breaking-news] WARNING: Telegram analysis schema_fail (both providers) for: ${title}`).catch(() => {}). This surfaces the double-failure rather than silently dropping the broadcast. Separately, consider extending the factory.ts fallback return to re-check schema_fail + cascadeOnSchemaFail for the lastResort path (that's a factory-wide change, out of scope here, but the no-alert path in the caller is the immediate fix).

[W2] velaRead empty string silently omitted without any alert
breaking-news/index.ts : 439–447
Symptom

If velaRead is empty or exceeds 12 words, it is quietly set to "" and the broadcast continues with a blank image subtitle. This is intentional per the comment ("renders headline-only rather than with a malformed subtitle"), but there's no admin alert or logging when the overshoot occurs in production, making it invisible in telemetry. The warn log on line 442 is emitted but doesn't reach admin Telegram. Over time this accumulates silently degrading broadcasts without any signal.

Root cause

The drop-on-overshoot path was designed to be graceful, but graceful degradation without observability becomes invisible regression.

Fix

Either: (a) add sendAdminTelegram(...).catch(() => {}) on the overshoot branch to surface the model drift in production, or (b) add a velaReadDropped counter to the response JSON so the caller can aggregate it. Low urgency if overshots are rare, but worth catching early after a model change.

[W3] schema maxLength:400 and truncation guard at >80 words are not aligned
registry.ts : 443 vs breaking-news/index.ts : 423–433
Symptom

The schema declares analysis: { maxLength: 400 } (characters). The post-generation truncation guard fires when words.length > 80 (words). At average English word length of ~5 chars plus a space, 80 words is roughly 480 characters, so a 400-character analysis could be up to ~66 words and would never trigger the word-count guard. The problem runs the other way: an analysis near the 80-word limit (legal per the prompt's "max 50 words" instruction, which the schema doesn't enforce at 50) could be 400+ characters and pass the schema check but still hit the word-count truncation. The two signals are measuring different things and neither is a strict superset of the other.

Root cause

The schema was written with a character limit; the truncation guard was ported from an older version that counted words. There's no single consistent enforcement unit.

Fix

Align the two. The simplest fix: change the truncation guard from words.length > 80 to analysis.length > 400 (characters), matching the schema. This makes the guard a true safety net for the declared constraint. Update the log message accordingly. The 280-word inner loop guard for paragraphs can stay as-is since it protects a different (now unreachable) old multi-paragraph path.

[W4] generateBreakingTweet still references outer-scope anthropicApiKey conceptually but actually takes its own param
breaking-news/index.ts : 184–188, 212–230
Symptom

generateBreakingTweet takes anthropicApiKey: string as a parameter and uses it to make a direct fetch to https://api.anthropic.com/v1/messages. This function is NOT migrated to the factory. The function is currently paused (no call sites; the comment at line 1013 says "PAUSED"). However, if tweet generation is re-enabled in the future by restoring the block from git history, it will (a) bypass the factory and llm_call_log, and (b) require the outer scope to have an anthropicApiKey variable, which was removed. The current handler never calls generateBreakingTweet, so this is not a production bug today, but it creates a latent trap for whoever re-enables tweets.

Root cause

Tweet generation was paused before the outer-scope anthropicApiKey removal, so the function body was left in place without updating it to the factory pattern.

Fix

Either: (a) delete generateBreakingTweet entirely and leave a comment in git history pointing to where to find the old implementation, or (b) migrate it to use llmComplete now (before it's re-enabled) and add a breaking_tweet task to the registry. Option (a) is safer since the function body has a direct Anthropic fetch that would cost money without logging.

[N1] Truncation log at line 244 reports the same number twice
breaking-news/index.ts : 244

console.log(`... from ${text.length + sourceTag.length} to ${text.length + sourceTag.length} chars`): both sides of "from X to Y" are computed the same way (after truncation) so the log always reads "from N to N". [N2] Migration uses ON CONFLICT DO NOTHING, masking re-runs
migrations/20260610000001_add_breaking_telegram_routing_flag.sql

ON CONFLICT DO NOTHING is correct for idempotent seeds, but it means if the row already exists with enabled = false (e.g. manually disabled via a flag override), the migration silently leaves it disabled and the task is dead in production without any indication. Consider an ON CONFLICT DO UPDATE SET enabled = true WHERE llm_routing_flags.enabled IS NULL if the intent is always to enable on first seed. As-is, it's fine for a greenfield insert; just know that re-running on a disabled row is a no-op.

Risks to monitor post-deploy

[M1] Groq rate limits: breaking_telegram now competes with news_broadcast_eval for Groq quota

Both breaking_telegram and news_broadcast_eval use Groq llama-3.3-70b as primary. They run sequentially on the same broadcast path: first generateTelegramAnalysis (breaking_telegram), then evaluateBroadcastReadiness (news_broadcast_eval). Each Groq free-tier account has a shared daily-request and daily-token cap across all tasks. Monitor llm_call_log for provider=groq, status=fail rows on either task after deploy. If Groq is rate-limited, breaking_telegram cascades to NVIDIA and news_broadcast_eval cascades to NVIDIA too, potentially exhausting NVIDIA RPM on a busy day with multiple candidates.

[M2] Heartbeat updates on staging, masking health-check reliability
breaking-news/index.ts : 699–715

The staging early-return fires AFTER the heartbeat upsert (line 699–704 fires unconditionally, then if (isStaging) returns at line 709). This means cron_heartbeats["breaking-news"].last_seen_at is updated on every manual staging invocation, keeping the health-check green. This is arguably intentional ("the cron ran, it just skipped work"), but it means a staging environment with the breaking-news cron accidentally enabled would perpetually update the heartbeat while doing zero work, which could mask a misconfigured ENVIRONMENT secret. Low risk if staging cron is confirmed active=false; worth noting for the next time someone manually triggers the function on staging during a debug session.

Safe

factory.ts cascade logic is correct for breaking_telegram

With cascadeOnSchemaFail: true on breaking_telegram, the primary Groq schema_fail correctly throws a synthetic retryable error (factory.ts line 114–119) and cascades to NVIDIA. If NVIDIA also fails with a network/timeout error, the fallback catch rethrows (line 152) and the try/catch in generateTelegramAnalysis catches it and fires the admin alert. The happy path and the retryable-error path are both handled correctly.

additionalProperties: true on BREAKING_TELEGRAM_SCHEMA is correct

Reviewed the factory schema validator path. additionalProperties: true means extra keys from qwen/llama are permitted; only the declared required fields are checked. This is consistent with every other free-tier schema in the registry (all have additionalProperties: true for the same reason: qwen adds reasoning/_meta keys that trip strict validators). No issue here.

TaskName union and registry are in sync

breaking_telegram appears in both types.ts (line 79) and TASK_REGISTRY in registry.ts (line 489). The registry self-check at module load would throw if lastResort.provider equaled primary or fallback, but breaking_telegram has no lastResort, so no issue. The getTaskConfig guard would throw on an unknown task name, so a missing registry entry would surface immediately on first call.

Staging early-return placement is correct for zero-API-call guarantee

The if (isStaging) return fires before any LLM call, any external API call (post-to-x, broadcastNewsBrief), and before price fetching. The only work done on staging is the Supabase heartbeat upsert and news fetch (which is not an LLM call). The zero-API-call rule for LLMs is satisfied.

recordCall in brief-generator uses EdgeRuntime.waitUntil correctly

recordCall in cost.ts uses EdgeRuntime.waitUntil when available and swallows rejections. The isolate-lifetime concern (waitUntil promises lost on fast exit) was raised as a question, but the callClaudeAPIWithRetry function is synchronous in terms of its return: it awaits the retry loop before returning. So the isolate doesn't exit until the loop completes. The recordCall fires synchronously after the await returns, and waitUntil extends the isolate for the write. The pattern is consistent with how recordCall is used throughout the factory adapters. No isolate-exit race here.
