Blocking findings

Blocking
B-1: Missing upgrade_close in-flight guard — double-close race on main positions
index.ts ~line 1959 (hold-duration outer if block, just after the holdH check at line 1965)

Symptom. A main position that has an upgrade_close proposal in status pending, auto_approved, or executing will simultaneously receive a hold-duration market close order and the user-approved upgrade_close trade. The second order to hit HL is a reduce-only on a position that is already flat — HL rejects it. But before that rejection: the SL order was already cancelled (line 1972-1975); if the upgrade_close fills first and the hold-duration close fails with a non-deregistration error, the SL is orphaned-cancelled with no position on HL and an open DB row. Subsequent cycles find a status=open position with no HL position and enter the phantom-reconciliation path, creating a spurious reconciliation close.

Root cause. The BB2 expiry block (lines 1786-1802) queries trade_proposals for in-flight upgrade_closes before acting. The hold-duration block was inserted at lines 1959-2110 without this guard. The main position type is the primary consumer of upgrade_close proposals (BB2 positions never have them), so the risk is higher for this path than for BB2.

Fix.

// Add immediately after line 1965 (the holdH >= 72h check), before the console.log: const { data: inFlightUpgrade } = await supabase .from("trade_proposals") .select("id, status") .eq("user_id", userId) .eq("asset_id", pos.asset_id) .eq("proposal_type", "upgrade_close") .in("status", ["pending", "auto_approved", "executing"]) .limit(1) .maybeSingle(); if (inFlightUpgrade) { console.log( `[position-monitor] HOLD-DURATION skipped for ${symbol} ${pos.side}: ` + `upgrade_close already in-flight (${inFlightUpgrade.status})` ); continue; } // Then proceed with the existing console.log and close logic.

Add a corresponding source-regex test: assert the hold-duration rule block contains upgrade_close and in-flight.

Worth-a-look findings

Worth-a-look
W-1: No test asserting bb2_30m positions are excluded from the hold-duration rule
position-monitor.test.ts — gap between the 21 new tests

The code correctly checks pos.position_type === "main" (line 1961), which excludes bb2 and bb2_30m. But there is no adversarial test confirming this. A future refactor that renames position_type values or adds a new type could silently fire the rule on non-main positions. The existing regex test for "rule only fires for main" is positive-assertion only — it does not assert bb2_30m is NOT caught.

Add test: Deno.test("HOLD-DURATION-ADV: rule does NOT reference bb2 or bb2_30m as eligible types", () => { const ruleBlock = src.slice( src.indexOf("Hold-duration rule: exit main positions"), src.indexOf("Native trailing trigger: place/update when activation met") ); assertNotMatch( ruleBlock, /position_type\s*===\s*"bb2/, "Rule must only check for 'main' — bb2 and bb2_30m have their own expiry path" ); });
Worth-a-look
W-2: No test asserting the rule does NOT fire when HOLD_DURATION_RULE_ENABLED is absent or "false"
position-monitor.test.ts — gap in HOLD-DURATION-ADV suite

All 21 tests assert the rule exists and wires up correctly when enabled. None assert the guard-off path. If the env var check was accidentally inverted (=== "false" instead of === "true"), no test would catch it. This is a staged-rollout feature — the disabled path is the default and the most important safety invariant.

Add test: Deno.test("HOLD-DURATION-ADV: rule is disabled when env var is absent (default-off)", () => { assertMatch( src, /Deno\.env\.get\("HOLD_DURATION_RULE_ENABLED"\)\s*===\s*"true"/, "Rule must use strict equality to 'true' — absent var must default to disabled" ); assertNotMatch( src, /Deno\.env\.get\("HOLD_DURATION_RULE_ENABLED"\)\s*!==\s*"false"/, "Must not use negated false-check — missing env var would then enable the rule" ); });
Worth-a-look
W-3: peakPnl can be negative for a never-profitable position; admin Telegram emits "Peak: +-N.NNN%"
index.ts line 2093

peakPnl = Math.max(pos.trailing_stop_peak_pnl_pct ?? 0, unrealizedPnlPct). The ?? 0 default means a fresh position with no stored peak and current unrealizedPnlPct of, say, -0.3% gives Math.max(0, -0.3) = 0. The rule condition is peakPnl < 0.5, so 0 < 0.5 = true — the rule fires correctly. However, if pos.trailing_stop_peak_pnl_pct was written as a small negative number (rare but possible via cumulative funding on a losing perp), the in-memory peakPnl itself could be negative, and line 2093 unconditionally prepends +: `Peak: +${peakPnl.toFixed(3)}%` would emit Peak: +-0.123%. Not a financial correctness issue, but a garbled ops notification.

Change line 2093: `Held ${holdH.toFixed(1)}h · Peak: +${peakPnl.toFixed(3)}%\n` to: `Held ${holdH.toFixed(1)}h · Peak: ${peakPnl >= 0 ? "+" : ""}${peakPnl.toFixed(3)}%\n`
Worth-a-look
W-4: Successful hold-duration closes on rare profitable positions will generate unexpected social posts
index.ts line 2053 — maybePostTradeClose

maybePostTradeClose posts to X for all winning trades where totalPnl > 0 and position_type === "main". The hold-duration rule is designed for 0%-win-rate positions, but "0% win rate" does not mean "never profitable" — a position could have a fill_price that lands above entry. In that case, the function will queue a social post celebrating a trade that was closed specifically because it was stuck below 0.5% peak. The post is not incorrect, but it may look strange: "ETH long +0.3% profit" on a trade your system exited due to poor momentum. Consider whether to suppress social posts for hold-duration closes in maybePostTradeClose, similar to the BB2 suppression at lines 2761-2764.

Add to maybePostTradeClose (after line 2764): // Skip hold-duration closes (position exited due to stuck momentum, not meaningful for social) if (closeReason === "hold_duration_rule") return;
