Blocking — resolve before writing code

BLOCKING Import structure forces an eager load of the eval module breaking-news/index.ts:1081 · news-broadcast-eval.ts:109

hasAssetRelevance() is currently not exported from news-broadcast-eval.ts. Exporting it is necessary to call it from breaking-news/index.ts. However, the existing eval import at line 1081 is a dynamic await import(...) — deliberately lazy to avoid the cold-start WORKER_RESOURCE_LIMIT that the function previously hit. If the pre-generation filter runs in the candidate loop (~line 950), it is called before that lazy import point. Wiring it to the eval module would require either (a) promoting the import to a top-level static import, undoing the cold-start protection, or (b) a second dynamic import inside the loop, which adds await-latency to every candidate iteration even when the story will be filtered. Resolve before coding by inlining a minimal copy of the four-line hasAssetRelevance logic directly in the loop — no import needed, no cold-start risk, no per-iteration await. Do not export the function from the eval module for this use.

BLOCKING Pre-generation filter must fail-open when resolved is null breaking-news/index.ts:942–945 (catch block)

The existing catch block sets resolved = null on classification failure and proceeds with importance = null, which already fail-opens the quality gate (passesGate stays true). The proposal does not explicitly state how the pre-generation filter handles resolved === null. If the filter reads resolved.assetSentiments or resolved.classification.importance directly without a null guard, it will throw, turning a soft classification failure into a hard skip of all remaining candidates. Write the filter as: if resolved === null, skip the asset-relevance check and let the candidate through — identical to the existing quality-gate behavior. The null path is not an edge case; it is the documented fail-open contract.

Worth a look — design decisions needed before coding

WORTH A LOOK The pre-generation filter is substantially redundant with the existing quality gate for importance-3 stories breaking-news/index.ts:958–968

For importance === 3, passesGate is already nonNeutralChipCount >= 1. A non-neutral chip requires a tradeable-asset entry in assetSentiments with direction bullish or bearish — which is exactly the third clause of hasAssetRelevance(). So for importance-3 stories, any story that passes the quality gate already passes the asset-relevance filter, and any story that fails the filter already fails the gate. The filter adds zero coverage for this entire importance tier. The gap is importance-4+ stories with non-null assetSentiments containing only non-tradeable tickers — currently gated as chips >= 1 (already required), plus importance-4+ stories with null assetSentiments — these pass the existing gate unconditionally (the "macro" carve-out) but could fail the new filter if category != "macro". Confirm this is the intended target: only the "high-importance, null-sentiments, non-macro-category" story class. If so, the filter could be expressed as a single additional condition on line 962 rather than a full function call.

WORTH A LOOK Geopolitical events tagged non-macro will be incorrectly blocked news-broadcast-eval.ts:109–118

hasAssetRelevance() passes unconditionally only when category === "macro". The classifier assigns "macro" to Fed/CPI/NFP/GDP events but does not consistently tag major geopolitical shocks (war, sanctions, election results) as macro — those can land in "equities", "crypto", or "commodities" with relevantAssets=[] and assetSentiments=null. A "US imposes new tariffs on Chinese tech exports" story is unambiguously market-moving but could arrive as category="equities", relevantAssets=[], assetSentiments=null, failing the pre-generation filter and never reaching the LLM. The Layer 2 judge's criterion 1 explicitly passes "geopolitical shock affecting risk assets broadly" regardless of asset tagging — but that check never runs if the pre-generation filter drops the story first. Either (a) confirm the classifier reliably assigns geopolitical shocks to "macro" (grep production news_cache for relevant recent examples before assuming this), or (b) extend the filter's carve-out to also pass when importance >= 4 AND assetSentiments === null — reproducing the existing gate's unconditional pass for high-importance null-sentiment stories.

WORTH A LOOK Pre-generation filter skips should NOT be logged to social_posts breaking-news/index.ts:979–991 (quality-gate dedup log)

The existing quality-gate skip inserts a row with tweet_id=null, error_message="skipped: importance=..." — this is load-bearing dedup: the same story is blocked from re-evaluation on the next hourly tick. If the pre-generation filter fires on a candidate and also logs to social_posts, those stories are deduped out for 48 hours. This is probably wrong: the proposal's goal is to cheaply skip an LLM call, not to permanently retire a candidate. A story that fails the filter today (e.g. scored high but tagged with only non-tradeable assets due to a classification blip) could correctly broadcast tomorrow after a different cross-source context. Do not log pre-generation filter skips to social_posts. Use continue without a DB insert — let the story re-compete on the next tick. Add a console.log for observability. The quality gate's dedup-log pattern exists to prevent re-evaluation of deliberate low-quality rejections; that rationale does not apply here.
