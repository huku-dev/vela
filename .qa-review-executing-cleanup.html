Blocking — must fix before implementing

B1: The executor can overwrite a cron-set "failed" back to "executed" — creating phantom positions

trade-executor.ts:1270–1274

The critical update at step 11 is: .update({ status: "executed" }).eq("id", proposalId) with no guard on current status. If the cleanup cron fires at T+30min and sets the proposal to "failed", and the executor is still mid-flight (e.g., stuck waiting on an HL RPC call), it will complete and unconditionally overwrite status back to "executed". The position row gets created (trade-executor.ts:1276+), notifications go out, and the proposal ends up as "executed" even though the system considered it timed out.

The reverse is also true and worse: the cron fires, sets status="failed". The executor then tries to place the HL order (which may succeed), creates the position, then calls .update({status:"executed"}) — which overwrites "failed" to "executed". The position is real, the proposal says "executed", but the cron already treated it as dead. There is no deduplication guard on position creation that would catch this.

Fix required: Add .eq("status", "executing") to the final status transition at trade-executor.ts:1271–1274. Every intermediate status update should also guard on status = 'executing' so that a cron-failed proposal is immediately abandoned by the executor when it checks its next DB update and gets 0 rows affected. The executor does not currently check the result of most of these updates — add a rows-affected check or re-query status before the HL order call.

B2: Part 1 (manual data fix) SQL is not shown, so it cannot be vetted — provide it explicitly

The proposal says "flip the 4 stuck proposals to 'failed' via a targeted UPDATE by ID." No SQL was provided. The WHERE clause safety cannot be evaluated. The risk: if the IDs are copy-pasted incorrectly, or if the WHERE clause uses IN (id1, id2, ...) without also gating on status = 'executing', the update could hit rows in other statuses if IDs are wrong. More critically, the 4 proposals need to be verified as genuinely stuck (no position row exists for them) before being marked "failed" — otherwise a proposal that IS executing successfully gets force-failed.

Fix required: Write the exact SQL and include a pre-flight SELECT to confirm (a) all 4 IDs have status = 'executing', and (b) none has a corresponding row in positions table. Gate the UPDATE on status = 'executing'. Run SELECT first, review output, then UPDATE.

Worth a look

A1: 30 minutes is plausible but not verified against actual wall-clock limit

run-signals/index.ts:100,146, trade-executor.ts (full file)

Supabase edge functions have a 150-second (2.5 min) wall-clock limit per invocation. The 30-minute timeout in the proposed cleanup is not about "how long can the executor run" — it's about "how long can a proposal be stuck in 'executing' before we assume the executor died." An executor process cannot legitimately run for 30 minutes; the isolate would have been killed by Supabase long before.

However, there is one mid-flight write that moves updated_at without completing execution: the downsize branch at trade-executor.ts:762–768 writes proposed_size_usd + updated_at but leaves status as "executing". This is the only case where updated_at advances mid-flight. The 30-minute window comfortably covers it (the downsize write happens well inside the 150s isolate limit). So 30 minutes is safe as a timeout value.

The concern is the opposite direction: 30 minutes is very conservative. A proposal stuck in "executing" for 30 minutes means the user waited 30 minutes with no feedback. Given the 150s isolate limit, a 5-minute timeout would catch all genuinely stuck proposals while being more responsive. This is a product judgment, not a correctness issue — flag for Henry.

Verify: confirm Supabase isolate wall-clock limit is 150s. If confirmed, consider tightening the timeout to 5 minutes. 30 minutes is safe but leaves users without feedback for too long.

A2: Users get zero notification when their stuck proposal is force-failed by the cron

notify.ts:4493–4499, Layout.tsx:120–128

By design, trade failures go admin-only (notify.ts:4494: "Trade failures are operational errors — admin-only, not user-facing. Users see the failure reflected in proposal status on the app."). This is correct for real-time failures where the user is likely in the app. It is wrong for proposals that have been stuck for days or weeks. The 4 affected users have no idea a trade was attempted and failed.

The Layout.tsx failure toast (Layout.tsx:120–128) fires on a executing → failed transition — but only if the app is open when the cron runs. If the user is not in the app (which is almost certain for a days-old stuck proposal), the toast never fires. There is no email or Telegram fallback for cron-induced failures.

The existing notifyLowBalanceFailure and notifyMarginShortfall functions show the pattern for user-facing failure notifications. The cleanup cron, however, is pure SQL — it cannot call notify.ts. A notification would require either an edge function wrapper or a separate "stale proposal notification" job.

Decide: for proposals force-failed by the cron, should the user receive a Telegram/email nudge? If yes, the cleanup cannot be pure SQL — it needs an edge function that queries affected rows and calls notifyLowBalanceFailure or a new notifyExecutionTimeout. If no, document this as intentional.

A3: TrackRecord.tsx silently drops executing proposals when they expire

src/pages/TrackRecord.tsx:41–43

TrackRecord filters active proposals as: status in ('approved', 'auto_approved', 'executing') AND expires_at > now(). A proposal stuck in "executing" with a past expires_at is already invisible on TrackRecord — it does not appear in the active list, and "failed" is not in the active filter either. When the cleanup cron runs and flips it to "failed", the card appears nowhere on TrackRecord (failed is not in either filter).

The user with a stuck executing proposal currently sees nothing on TrackRecord. After the fix, they still see nothing — but now the proposal has a "failed" error message. Whether this is acceptable depends on whether failed proposals should appear in the history view (useTrading.ts at line 266 does include "failed" in the status list fetched, so the data is available, just not rendered on TrackRecord).

Verify: confirm whether failed proposals intentionally don't render on TrackRecord, or whether there's a history list that includes them. If failed proposals are expected to be visible as a record, add them to TrackRecord's history section.

S1: updated_at is the correct column to gate on

The executor writes updated_at exactly once mid-flight (downsize branch, trade-executor.ts:766) and at claim time (trade-executor.ts:100). Both are inside the 150s isolate window. Using updated_at < now() - interval '30 minutes' correctly identifies proposals whose executor isolate died without finishing, and would not false-positive on an active downsize event.

S2: Position-count gate does not include executing proposals — no slot lockout issue

proposal-generator.ts:955–960, 1094–1106

The max_active_positions enforcement queries positions WHERE status='open' (not trade_proposals WHERE status='executing'). Stuck executing proposals do not consume a position slot from the generator's perspective. Flipping them to "failed" does not free any slot that was being held. No downstream capacity logic breaks.
