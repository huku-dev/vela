Blocking

signal-persistence.ts line 700 / daily-digest/index.ts line 371 / run-signals (parallel call site)
Blocking 1

Change 1: persistDailyDigest cannot write themeLine at the call site without a signature or caller change — and there are TWO call sites

Symptom. The proposal says "no migration needed — detail is already JSONB" and that themeLine can be written "at the call site." But look at the actual call in daily-digest/index.ts line 371–374:

persistDailyDigest(supabase, digestNarrative, digestItems, { mode, detail: { marketPulse } })

The detail object is built right here from marketPulse. Writing themeLine into it at this call site is straightforward: add themeLine to the object literal. BUT there is a second call site inside run-signals/index.ts (the recovery path). If only daily-digest/index.ts is updated, the recovery path will never write themeLine, meaning tomorrow's anti-repetition query returns null for any brief generated via the recovery path. Anti-repetition then silently degrades for ~30% of production briefs with no log warning.

Root cause. There are two callers of persistDailyDigest — one in daily-digest/index.ts and one in run-signals's inline digest path — and the proposal only identified one.

Fix. Search for all calls to persistDailyDigest before writing any code. Update both call sites in the same commit to pass themeLine into the detail object. Add a log line at the query side: "yesterday's theme_line absent" so coverage gaps are visible.

notify.ts line 1688–1695 / line 1738
Blocking 2

Change 2: Placeholder leak in the proposed preheader logic — no isPlaceholder guard

Symptom. The proposed preheader code is:

const preheader = options?.briefHeader?.themeLine && options?.briefHeader?.subhead ? `${themeLine}. ${subhead}` : options?.briefHeader?.themeLine ?? paragraphs[0]?.slice(0,100) ?? ""

The email body already guards against placeholder theme lines at lines 1688–1695 via the placeholderHeadlines Set ("Daily Market Digest", "Evening Market Wrap"). When isPlaceholder is true, themeLineHtml renders as empty string. But the proposed preheader code does NOT check isPlaceholder. If the narrative rewriter returns themeLine="Daily Market Digest" (which it won't ordinarily, but CAN on LLM degradation — the Sonnet prompt at line 800 says "think newspaper cover" but provides "Daily Market Digest" as the fallback heading string that signal-persistence writes), the preheader becomes "Daily Market Digest. [subhead]" — a worse preheader than the current first-100-chars-of-prose fallback.

Root cause. The isPlaceholder guard exists in the body rendering path but was not ported to the proposed preheader path.

Fix. Mirror the existing isPlaceholder check from line 1695 into the preheader logic. When isPlaceholder is true, fall through directly to paragraphs[0]?.slice(0,100) ?? "". Reuse the placeholderHeadlines Set already defined at line 1688 rather than duplicating the check.

brief-generator.ts lines 788–796 / news-fetcher.ts lines 795–823
Blocking 3

Change 3: The news feed has a structural CRYPTO-FIRST ordering bias that the proposed prompt additions cannot overcome

Symptom. formatNewsForPrompt in news-fetcher.ts outputs sections in a hardcoded order: CRYPTO NEWS first, then ECONOMY & MACRO, then STOCK MARKET, then COMMODITIES. The LLM sees crypto items at the top of the context window. The existing prompt at line 606 says "the first sentence should be the most impactful event" — but the proposed fix adds "not just crypto" as a soft correction to an ordering problem that is structural in the formatter.

The proposed instruction "check the full ranked list above — if a tracked asset other than BTC, SPX, or Oil has a top-ranked item" is especially problematic. Read formatNewsForPrompt at line 786–792: the function annotates items with +N sources (cross-source count) but does NOT expose a numeric score. The LLM cannot "check the top-ranked item" because no rank number is visible — only relative position within a category section. The item ranked #1 by score in the COMMODITIES section is rendered AFTER all CRYPTO items.

Root cause. The bug is in the formatter, not the prompt. A free-tier LLM (qwen/llama) instructed to "check the full ranked list" will see items in crypto-first categorical order. Without reordering the formatter output, the prompt instruction is noise.

Fix. Either: (a) change formatNewsForPrompt to output items in a single list sorted by cross-source count and score, without categorical headers; or (b) if category headers must be kept, at minimum put the category with the highest-scored item first. A prompt instruction alone will not overcome formatter ordering bias in a free-tier model. Flag this as a formatter change before writing the prompt change.

Worth a Look

brief-generator.ts (Change 1 query) / daily-digest/index.ts runtime budget
Worth a look 1

Change 1: Yesterday-brief query latency is negligible BUT adds a new failure mode if the brief row is missing

A SELECT ... LIMIT 1 against the briefs table with an index on (brief_type, created_at) will resolve in under 10ms — not a meaningful latency concern against a 150s budget. The concern is the failure path. If the query returns no row (first-run day, or a day when the digest failed), detail->>'themeLine' is null. The proposal notes "there's no hard failure mode." Confirm: the fallback must be a no-op, not a warning that surfaces in Sentry. Verify that the query result being null simply skips the anti-repetition injection rather than logging an error.

brief-generator.ts rewriteDigestAsNarrative line 790
Worth a look 2

Change 1: "First sentence" extraction via .split(".") is fragile on real summary content

Real narratives contain asset tickers (BTC), numbers ($65,000), and source citations like "U.S." that contain periods. A split on "." will fragment mid-sentence at "U.S." or "$65,000." — producing "U" or "$65,000" as the captured "opener" that then pollutes the anti-repetition prompt. Use a sentence boundary regex (/(?<=[.!?])\s+(?=[A-Z])/) or, simpler, split on ". " with a capital-letter lookahead and take the longest leading candidate up to 200 chars. Whatever approach: add a test case with "U.S. rate hike" and "$65,000." in position 0.

brief-generator.ts rewriteDigestAsNarrative fallback path line 816–818
Worth a look 3

Change 1: Anti-repetition context passed into narrative rewriter but not into items generator — where the repetition actually originates

The proposal injects "yesterday's opener" and "yesterday's theme_line" into rewriteDigestAsNarrative (the Sonnet narrative step). But the repetitive opener pattern ("Markets are [X] this morning") originates in the items step: the free-tier LLM selects BTC/SPX/Oil and writes items that all share market-mood framing, which the narrative rewriter then inherits. Injecting anti-repetition context only at the rewrite step means the items text fed into Sonnet already embeds the same fear-opener structure, and Sonnet is constrained by the rule "Do not add any information that isn't in the items below" (line 770). The fix addresses the symptom at the wrong layer. Consider also passing the anti-repetition context into the generateDailyDigest items prompt, or explicitly — to Sonnet — "the opening mood framing may not match the prior day even if the items text implies fear."

notify.ts line 1738 / Change 2
Worth a look 4

Change 2: Combined preheader at 130-160 chars truncates mid-subhead in all major email clients

Gmail desktop clips preheader at ~85-90 chars. Apple Mail shows ~100-110. theme_line (3-6 words) + ". " + subhead (up to 22 words at ~120 chars) = 130-160 chars in the worst case. The preheader will be clipped mid-subhead, often at a natural pause if the subhead is long. This isn't a correctness bug, but it means the subhead — the more informative part — will frequently be cut. Consider reversing the order: subhead. theme_line or using only theme_line if under 85 chars (which it is by construction at 3-6 words/20-35 chars) followed by the subhead truncated to fit. Alternatively, limit the preheader to theme_line alone when it's available — it's purpose-built to be punchy and stands alone.

news-classifier.ts line 128–129
Worth a look 5

Change 4: Raising renderNewsModule SUMMARY_CAP to 150 exposes a mismatch with the existing word-boundary threshold

The current renderNewsModule truncation at line 1804–1808 uses lastSpace > 80 as the word-boundary threshold — meaning: only use the word boundary if it's after position 80 in the 110-char slice. If the cap is raised to 150, the threshold (lastSpace > 80) needs to scale proportionally (to ~120 or so), otherwise the code can still hard-cut at 80 chars on rows where the last space before position 150 is between positions 80-120. The comment says this cap exists to keep summaries to 2 lines at 14px. At 150 chars, that assumption breaks for narrower email viewports (~270px, ~16 chars/line). Confirm the new cap matches the intent (render legacy rows without truncation) before raising it.

Nit 2

Change 1: Anti-repetition instruction placement — put it in the system prompt, not the user turn

The proposal adds "Yesterday's brief opened with X. Use different structural framing today" to the user prompt. But the narrative rewriter's system prompt at line 768 already governs structural framing ("Open with 1 sentence that captures the overall market mood"). Putting the anti-repetition context in the user prompt creates a possible conflict between the system-level instruction (mood-capture opener) and the user-level instruction (don't echo yesterday). news-classifier.ts line 278 / notify.ts line 1802
Nit 3

Change 4: The SUMMARY_MAX constant in news-classifier.ts and SUMMARY_CAP in renderNewsModule are independent; name them clearly in the commit

Both constants do the same job at different layers (write-side vs render-side). Lowering one and raising the other is intentional, but the asymmetry will confuse future readers unless the commit message explains the design: "classifier writes short; renderer accepts legacy-length rows without truncation." Add a comment to SUMMARY_CAP in notify.ts explaining the legacy-row context so this isn't silently reverted.

Risks to Monitor Post-Deploy

  • Anti-repetition coverage (Change 1): Log "anti-rep: yesterday's opener=[X]" and "anti-rep: no previous brief found" at the query site. Check logs the first 5 mornings to confirm the query is returning rows and the context is being injected. A silent null means the feature is no-op.
  • Brief diversity (Change 3): After deploying the prompt change, audit 7 consecutive digest items for asset variety. If BTC/SPX/Oil still appear in all 3 positions, the formatter ordering bias has not been addressed and the prompt change provided no benefit.
  • Preheader render (Change 2): Send a test email to a Gmail address and Apple Mail address and check the inbox preview string before deploying to all subscribers. A 2-minute manual check prevents a bad first impression for all users on day 1.
  • Classifier summary length distribution (Change 4): After deploying the 40-85 char prompt change, query SELECT avg(length(summary)), min(length(summary)), max(length(summary)) FROM news_cache WHERE created_at > now() - interval '24h' the next morning. If avg is still above 100, the LLM is not respecting the new target and the code backstop at 85 is doing all the work (producing truncated summaries).

Safe

  • Change 1, DB query latency: A SELECT ... LIMIT 1 ORDER BY created_at DESC against an indexed column is negligible (~5-10ms) and does not threaten the 150s budget.
  • Change 1, detail JSONB shape: Adding themeLine to the existing JSONB object is additive and non-breaking. No migration needed. The read path using detail->>'themeLine' returns null for rows written before this change, which is the correct fallback.
  • Change 2, fallback path when briefHeader is undefined: The proposed fallback chain is options?.briefHeader?.themeLine ?? paragraphs[0]?.slice(0,100) ?? "" — this correctly falls through to the current preheader behavior when Sonnet fails and the brief has no header. The fallback is safe.
  • Change 4, raising SUMMARY_CAP for existing 110-char rows: Rows written under the old 110-char target are complete sentences; rendering them without the truncation ellipsis is strictly better for users. The concern in Worth-a-look 5 is about the threshold math, not about whether removing truncation from these rows is correct — it is.
