WORTH A LOOK  Plausible bugs / surprising patterns

W1
TPD short-circuit sets retryable=false, silently blocking fallback cascade
openai-compatible.ts:89-97
Symptom: When Groq returns a 429 with error.type === "tokens" (TPD exhausted), isRetryable is set to false and makeError(..., false) is thrown. In factory.ts:123, primaryRetryable = err.retryable ?? true — but this is false — so canFallback evaluates to false at line 124. The factory throws immediately without attempting the NVIDIA fallback.
Root cause: The fix correctly prevents pointless retries within the same provider but treats the error as globally non-retryable. The factory's cascade logic conflates "don't retry this provider" with "don't try any other provider".
Concrete impact: For tasks where Groq is primary (e.g. position_catalyst, signal_crosscheck, telegram_query, engagement_reply), a Groq TPD exhaustion means the NVIDIA fallback is never called, even though NVIDIA has a completely independent quota. The system fails harder than it needs to when Groq's daily budget is gone.
Expected behavior for primary failure: Skip retrying Groq. But still attempt NVIDIA fallback.
The flip side (why the current behavior is not insane): For tasks where Groq is only the fallback (most tasks, since NVIDIA is primary), this code path isn't reached during primary execution. The risk is real for the 4 tasks listed above. Whether the product wants "NVIDIA fallback fires on Groq TPD" is a judgment call — but it is currently not happening and there is no comment documenting the intent.
Consider: introduce a distinct providerExhausted property on the error (or a dedicated error subclass) that signals "skip retries within this provider, but still allow cascade." The factory would check !err.providerExhausted (rather than err.retryable) when deciding whether to fall through to the next provider. If the intent is deliberately to not fall back on TPD (because the fallback would also be burning into a shared Groq budget), document that explicitly with a comment.
W2
signal_crosscheck and admin_task_triage still use additionalProperties: false schemas without cascadeOnSchemaFail
registry.ts:106-117 (CROSSCHECK_SCHEMA) and 399-427 (ADMIN_TASK_TRIAGE_SCHEMA)
Symptom: Both schemas have additionalProperties: false. If qwen (used as signal_crosscheck fallback via meta/llama-3.3-70b-instruct) or NVIDIA (admin_task_triage primary) returns extra keys, the validator fires a schema_fail. Without cascadeOnSchemaFail, the schema_fail passes through silently — the caller receives a degraded result without attempting the next provider.
Root cause: The MEMORY.md entry from 2026-05-19 notes: "earnings_extract, macro_surprise, sec_filing_classify are latent versions of this bug." That list was fixed in this PR (those schemas got additionalProperties: true), but CROSSCHECK_SCHEMA and ADMIN_TASK_TRIAGE_SCHEMA were not touched.
Risk level:
  • signal_crosscheck: primary is Groq llama (not qwen), and CROSSCHECK_SCHEMA is strict (verdict + reason only). Schema_fail here is plausible but lower frequency than qwen-primary tasks. The test that was updated uses this task precisely because it has no cascadeOnSchemaFail — so this is also a semantically correct test fixture. Moderate risk.
  • admin_task_triage: NVIDIA qwen primary, strict schema. High-frequency schema_fail risk given qwen's documented extra-key habit. No cascadeOnSchemaFail means silent degradation on every qwen extra-key response. The downstream effect is a missing admin triage record rather than user-facing data loss, but it is inconsistent with how every other NVIDIA-primary task in the file is configured.
Add additionalProperties: true to ADMIN_TASK_TRIAGE_SCHEMA and add cascadeOnSchemaFail: true to the admin_task_triage task. For signal_crosscheck: the schema is intentionally strict (it is the test anchor for the no-cascade behavior), but add a comment explaining why additionalProperties: false is retained intentionally, to prevent future "fix" attempts.
W3
No test coverage for the new TPD short-circuit branch
openai-compatible.ts:89-95 / llm.test.ts (absent)
Symptom: The TPD detection logic (errJson?.error?.type === "tokens" sets isRetryable = false) has zero test coverage. A future refactor that changes the error path could silently revert to the 120s retry behavior.
Root cause: No test was added alongside the new branch in openai-compatible.ts. The existing 429 test at line 506 tests the retry path but passes a plain-text body ("rate limited"), not a JSON body with error.type === "tokens".
What a good test checks: (a) Groq 429 with {"error":{"type":"tokens","message":"..."}} body results in a single attempt (no retry waits), (b) Groq 429 with plain text / RPM body ({"error":{"type":"rate_limit_exceeded"}}) still retries normally (up to MAX_ATTEMPTS).
Add an LLM-ADV: Groq TPD 429 short-circuits without retry test that stubs the Groq endpoint to return {"error":{"type":"tokens","message":"daily token quota exceeded"}} with status 429, and asserts the factory throws after exactly one attempt (not three).

N1
Comment says "midnight UTC for TPD" but Groq resets on a rolling 24h window
openai-compatible.ts:87
The comment reads: "the quota resets (midnight UTC for TPD)". Groq's actual TPD (tokens per day) resets on a rolling 24-hour window, not at midnight UTC. The health-check at health-check/index.ts:710 already reflects this correctly: "Limit resets on rolling 24h window." Stale comment could mislead a future debugger about when recovery will happen.
Change "midnight UTC for TPD" to "rolling 24h window" to match the health-check comment and Groq's documented behavior.
N2
news_rank timeout comment omits the actual arithmetic for MAX_ATTEMPTS=3
registry.ts:743-746
The comment reads "3×30s retry exhaustion = 92.5s wasted; 3×15s = 47.5s". The 92.5 and 47.5 figures include backoff time (500ms + 2s + 6s = 8.5s across attempts) but this is not spelled out. Fine math, slightly opaque. The decision to halve the timeout is sound.
Consider noting "(includes ~8.5s cumulative backoff)" parenthetically so the 92.5 / 47.5 numbers are self-evident to a reader who doesn't know the backoff table.

The TPD short-circuit assumes Groq returns {"error": {"type": "tokens", ...}}. This was inferred from the production Sentry alert body captured at incident time. If Groq changes their error format (or uses a different field name like code instead of type), the guard silently fails and the 120s retry behavior returns. No regression test can catch this — only monitoring can.

If any have latency exceeding ~15s (suggesting multiple retries fired), the short-circuit is not activating.
