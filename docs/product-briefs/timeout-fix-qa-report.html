Option A — Parallelise Loop

BLOCKING A1: usedHeadlineTitles is shared mutable state — concurrent writes corrupt brief diversity
Line 197: const usedHeadlineTitles = new Set<string>(); is created once and passed into persistSignalResult() (line 252) for every asset. The purpose is to prevent the same headline appearing in briefs for multiple assets. Under Promise.all, all concurrent calls receive the same Set reference and mutate it simultaneously in non-deterministic order. This does not produce a runtime crash — it silently degrades brief quality (some briefs will incorrectly see each other's headlines as "used") or miss dedup entirely depending on execution timing. Not safe to parallelize without first restructuring this to a pre-allocated set or post-processing step.
BLOCKING A2: CoinGecko rate limiting at 4x concurrency will produce 429s
Line 487: the 2s sleep exists specifically to stay under the CoinGecko demo cap of 30 calls/min. With 4 crypto assets running concurrently, each calls CG for candles AND daily closes simultaneously: 4 assets × 2 CG calls = 8 calls in under 1 second. At 30 calls/min (1 call every 2s), this bursts well past the rate limit. CG 429s cause a retry cascade or silently fall back to degraded data depending on fetch4HCandlesHL/fetchDailyClosesHL internals. The author's note that "HL API can handle moderate parallelism" is true for HL — it is NOT true for CoinGecko. A concurrency cap must be applied only to HL calls, with CG calls serialized, OR crypto assets must still be processed sequentially while HL-only assets parallelize.
WORTH A LOOK A3: Digest double-generation risk if async asset processing finishes in a different order
Lines 503-518: digest generation fires after the asset loop if !targetAsset && currentHourUTC >= 8 and no digest exists yet for today. Under parallelism the outer loop still runs once and the digest check is outside the parallelized block, so this specific path appears safe. However: if the implementation naively wraps the loop body in Promise.all, review whether the digest block ends up inside or outside the parallelized scope. If inside, two concurrent runs of run-signals (e.g. 4H cron + volatility-check cascade) could both see no digest and both start generating. The existing DB check is a read-then-write without a lock. This is a pre-existing race condition made more likely by faster completion times.
WORTH A LOOK A4: crosscheckPromises is an array accumulated across the loop — safe under parallelism, but watch ordering
Line 203: crosscheckPromises is a simple push-accumulator. Concurrent pushes to a JS array are safe in a single-threaded isolate. The array is awaited after the loop. This is fine. Note: if concurrency errors cause a promise to reject, the final Promise.allSettled (or equivalent) must be used rather than Promise.all to avoid aborting the remaining crosschecks.
NIT A5: asset-intel-generate has no shared mutable state in its loop — safe to parallelize independently
Lines 203-219: the asset-intel-generate loop accumulates counters (ok, skipped_dedup, etc.) but these are incremented after each await in a single-threaded Deno isolate. No shared object is mutated across concurrent calls. The upsert uses onConflict: "asset_id,period_label,trigger_reason" so concurrent writes for the same asset are idempotent. asset-intel-generate can safely be parallelized with a moderate cap (e.g. 4).

Option B — Batch Param + Two Crons

BLOCKING B1: run-signals asset fetch has no ORDER BY — batch slicing is non-deterministic
Lines 105-108: .from("assets").select("*").eq("enabled", true) — no .order() clause. PostgreSQL row order without ORDER BY is undefined and changes with vacuums, inserts, and index scans. A batch split at offset 11 would return different assets on different runs. Batch 1 and batch 2 could overlap or leave gaps. This must be fixed before Option B is safe: add .order("symbol", { ascending: true }) (matching what asset-intel-generate already does at line 182).
WORTH A LOOK B2: Digest fires in every batch that satisfies !targetAsset && currentHourUTC >= 8
Lines 503-518: both batch 1 and batch 2 are full cron invocations with no targetAsset. Both will enter the digest check block. The DB check (gte("created_at", today)) prevents a second write IF batch 1 finishes and commits before batch 2 checks. In practice batch 2 starts 1 minute after batch 1, so the race window is narrow but not zero. The safest fix: add a batchIndex param and only attempt digest when batchIndex === 0 (or last batch). Alternatively, move digest generation to its own cron slot entirely.
WORTH A LOOK B3: Cron schedule has no free minute slot adjacent to existing run-signals-4h at :10
Current schedule: run-signals-4h at 10 0,4,8,12,16,20 * * *. The 5-minute spacing rule (documented in cron-schedule.md) means a second entry cannot use :05, :10, or :15. Available adjacent slots: :11, :12 (but scanner-30m fires at :12 via 12,42 * * * * — conflict), :13, :14. A slot at :13 past each 4H mark would give 3 minutes of overlap margin. cron-schedule.md says "35 active crons, ~52 invocations/hour, ceiling 80/hour" — adding 6 more per day (one per 4H window) stays well within budget. Update cron-schedule.md in the same migration commit per the documented procedure.
SAFE B4: asset-intel-generate batch approach is straightforward — DAILY_BATCH_LIMIT already exists
Line 74: const DAILY_BATCH_LIMIT = IS_STAGING ? 2 : 25;. The function already has limit logic. Adding a batchOffset param and passing it as .range(offset, offset + batchSize - 1) after the existing .order("symbol") is a 5-line change. The upsert conflict key ensures idempotency. The 5:42 UTC daily slot has room for a second invocation at 5:43 or 5:44 — but given this function runs sequentially per asset with LLM calls (~2-3s each × 22 assets = ~60s), the real fix is Option A (parallel) for this function, not batching.

Option C — Skip Stale / Reduce Sleep

WORTH A LOOK C1: SMART_REFRESH_DEDUP_MIN is 60 minutes (line 78) — does not apply to daily cron path at all
Line 256: if (triggerReason !== "scheduled_daily" && triggerReason !== "manual") — the dedup gate is explicitly bypassed for the daily cron path. This is a valid optimization IF you want to reduce compute on days where briefs are already fresh — but it does NOT solve the timeout problem unless a large fraction of assets have fresh briefs at 05:42, which is unlikely since this is the only daily run.
NIT C2: Sleep reduction in run-signals saves 5.5s on an unknown total budget — insufficient alone
Line 487: removing the 500ms sleep for HL-only assets saves 11 × 500ms = 5.5s. The comment at line 485-486 was written when there were 16 assets and claimed "total sleep under 15s." At 22 assets the author calculates 27.5s. However, the actual bottleneck is per-asset work time (candle fetch, indicator compute, LLM brief generation, notification, proposal generation), not sleep. If each asset takes 5-10s of wall time, 22 assets × 7s = 154s — already over budget before any sleep. Sleep reduction without parallelism or batching does not close the gap for run-signals. This option is correct for asset-intel-generate as a secondary optimization but should not be presented as a standalone fix for run-signals.

Final Recommendation

Implement Option A for asset-intel-generate only; implement Option B for run-signals.

For asset-intel-generate: It has no shared mutable state in the loop (verified). Apply Promise.all with a concurrency cap of 4. Each iteration is an independent LLM call + DB upsert with an idempotent conflict key. At 22 assets × ~2.5s per asset with cap-4 parallelism, expected runtime drops from ~55s to ~14s. No cron change needed — the single 05:42 daily run covers it.

For run-signals: Do NOT parallelize yet. The shared usedHeadlineTitles state and the CoinGecko rate limit make safe parallelism a larger refactor than a one-session fix. Instead, implement Option B: add an ORDER BY symbol ASC to the asset fetch (line 107), add a batchOffset + batchSize query param, schedule a second cron at 11 0,4,8,12,16,20 * * * with body {"batchOffset":11,"batchSize":11}, and suppress digest generation when batchOffset > 0.

Also apply Option C sleep reduction as a free win alongside whichever primary option is chosen. It does not solve the problem alone but reduces wasted time.

Before implementing anything: query SELECT schedule FROM cron.job WHERE jobname = 'run-signals-4h' to verify the live schedule matches cron-schedule.md. The migration history shows a mismatch (:00 vs :10) that must be reconciled before writing new cron entries.
