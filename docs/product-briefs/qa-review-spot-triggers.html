Blocking

B1. Spot TP ladder passes sizeInAsset without asserting it is > 0 before placement

trade-executor.ts ~line 1696–1706

The TP loop computes sizeInAsset = (orderResult.fill_size ?? 0) * PROFIT_LADDER_FRACTIONS[i] and passes it into placeTriggerOrder({ ..., sizeInAsset }) with the spot spread. The outer guard at line 1686 checks (orderResult.fill_size ?? 0) > 0, but it does not prevent sizeInAsset from resolving to a negligibly small float (e.g. if fill_size is unexpectedly tiny). More importantly, the adapter's spot-trigger guard at line 1239 reads if (!request.sizeInAsset || request.sizeInAsset <= 0) — this guard fires if sizeInAsset rounds to zero after roundToWire. When it fires the adapter returns success: false, the TP is silently skipped (only a console.warn), and the DB position is written with a partial or empty exchange_tp_order_ids array without any Sentry alert. For a spot position whose fill_size is at the minimum tick boundary and the smallest ladder fraction (PROFIT_LADDER_FRACTIONS[0], presumably ~0.33) the product could round to zero on low-decimal assets (e.g. HYPE szDecimals=2 at very low price).

The more acute risk is the interaction with roundToWire: the outer guard passes a positive float, the adapter guard passes, but roundToWire returns "0" for a sub-tick size. HL will then reject the order with a size error, but the code path only console.warns the failure and moves on. No Sentry capture, no audit row. The native TP is silently absent and position-monitor's retrace path has no native trigger to defer to. For a full-size SL-only position this is acceptable, but the TP ladder is advertised as active protection.

Fix: Add a minimum-size guard before TP placement: if (sizeInAsset < MIN_SPOT_ORDER_SIZE) { console.warn(...); continue; } where MIN_SPOT_ORDER_SIZE is derived from the asset's szDecimals (1 tick). Alternatively, assert roundToWire(sizeInAsset, szDecimals) !== "0" before calling placeTriggerOrder. Escalate a Sentry captureException (not just console.warn) when a TP placement fails on a spot position, matching the treatment of the SL placement failure at line 1665.

Worth-a-look

W1. cancelTriggerOrder silently falls back to perp index when isSpot=true but asset is absent from spot meta

hyperliquid-adapter.ts lines 1432–1435

The logic is const assetIndex = useSpotIndex && spotToken ? spotToken.index : (await this.resolveAsset(asset)).index. If a caller passes isSpot=true explicitly but getSpotMeta() does not return the asset (e.g. HL spot meta fetch returns a stale or empty result due to a transient API error, or the asset was recently delisted from spot), spotToken is undefined, useSpotIndex && spotToken is falsy, and the code silently falls back to the perp index. The cancel is sent to the wrong market. The order stays live on the spot market, position-monitor never sees it cancelled, and the native SL fires on the now-closed position (which would be a no-op on HL, but could cause phantom reconciliation confusion).

The doc comment says "When true, always use spot index even if the asset isn't in spot meta (shouldn't happen in practice)" — but the implementation does the opposite of "always use".

Fix: When isSpot === true and spotToken is undefined, return { success: false, error: "isSpot=true but asset not found in spot meta — cannot determine spot index" } rather than silently using perp index. This surfaces the getSpotMeta failure rather than hiding it.

W2. HL_STRING_STATUS suppression does not distinguish known-benign strings from genuine error strings

hyperliquid-adapter.ts lines 1360–1365; position-monitor/index.ts lines 2011–2013

The bare-string guard fires on typeof status === "string" and prefixes whatever string HL returned: HL_STRING_STATUS:${status}. Position-monitor then checks triggerResult.error?.startsWith("HL_STRING_STATUS:") and suppresses Sentry for the entire class. If HL returns a string like "error: position would exceed leverage limits" or "rejected" (both plausible string error responses from HL's L1), the Sentry alert is also suppressed. The trigger was NOT placed in that case but position-monitor logs it as "HL accepted, no oid echoed" and continues with software fallback — without alerting.

The original Sentry noise was caused specifically by "waitingForTrigger" being returned as a string. A tighter guard would be status === "waitingForTrigger" (or a small set of known-benign strings) rather than all strings.

Fix: Change the string guard to check for the specific known-benign value: if (status === "waitingForTrigger"). Return HL_STRING_STATUS:waitingForTrigger only for that case. For any other string value, return a plain error: String(status) so Sentry fires normally. If additional benign strings are discovered, add them explicitly to an allowlist rather than suppressing all strings.

W3. earliestOpenMs snapshot is stale if a new position opened between the cache seed and the spot-fill check

position-monitor/index.ts lines 420–422, 589–596

earliestOpenMs is computed from positions.map(p => new Date(p.opened_at).getTime()) before the position loop. If a position opened after this snapshot but before the fills cache is populated (i.e. its opened_at is newer than earliestOpenMs), getUserFills will be called with the older timestamp, which still covers the new position's fills. This is safe.

The inverse risk: if a position was opened during a prior monitor run with a very recent opened_at, and the cache was populated in a previous invocation (module-level cache reuse), the shared cache might not include fills from that position's open. However, userFillsCache is declared inside the per-user function call (not at module level), so it is freshly populated each monitor run. No stale-cache issue.

The real residual risk: userFillsCache is populated on the first spot position that needs it and then reused for all subsequent spot positions in the same user-loop. The cache is keyed only by earliestOpenMs. If two spot positions exist for the same user and the second position was opened after the first position's SL fill, the cache (seeded from earliestOpenMs = the older position) includes fills from both SL events. oid-based matching prevents a false match, but the comment at line 588 should state explicitly that oid uniqueness is the safety property being relied on, not just coin-level deduplication.

Low urgency. No code change required. Add a comment at cache-population site making explicit that oid-based matching is the cross-position safety guarantee, not coin. This prevents a future refactor from removing oid matching and breaking the assumption.

W4. T16 asserts passed = closed || priceUpdated — a price-only update passes the test even if close_reason is wrong or position remains open

e2e-testnet-test/index.ts lines 1444–1453

The pass condition is closed || priceUpdated where priceUpdated is pos?.current_price != null && Math.abs(Number(pos.current_price) - spotPrice) > 0.001. If position-monitor runs, updates current_price but does not close the position (e.g. the stop_loss_price check failed, or it closed with close_reason="phantom_reconciled" instead of "stop_loss"), the test still returns passed: true with the amber-toned message "Spot price updated... status=..." rather than failing. This masks a real position-monitor regression where the stop-loss path did not fire.

Fix: Remove the priceUpdated fallback from the pass condition. Require closed (status === "closed" && close_reason === "stop_loss"). Price-update-only should be a failing result with a descriptive error. The priceUpdated fallback was appropriate during initial test development but is too loose for a regression gate.

Nit

N1. T14's wrong-index cancel result is logged but not asserted — the "index correctness proof" comment overstates what is actually proven

e2e-testnet-test/index.ts lines 1113–1118

The comment says Step A proves "no-op on HL spot market" but the code explicitly says "Result (success or failure) is logged but NOT asserted — both outcomes are acceptable." The proof that the trigger survived Step A rests entirely on Step B succeeding. If HL returns success: true for a wrong-index cancel-of-nonexistent AND Step B also succeeds (because a new oid was assigned), the wrong-index cancel could have worked without detection. The proof relies on HL never assigning the same oid twice in a test run, which is reasonable but not formally verified. Consider a comment clarifying that the oid uniqueness assumption underlies the proof.

N2. TRIGGER-ADV: No raw floatToWire only scans the first 1500 bytes of placeTriggerOrder, missing the perp branch entirely

hyperliquid-adapter.test.ts lines 413–418

The spot branch and the perp branch together span ~1300 chars in the source. The 1500-char window catches the spot branch but the perp branch starts around char 1100 and its price fields are at chars ~1280–1380. At the edge of the window, priceToWire calls in the perp branch may be scanned or may not, depending on surrounding whitespace. Other TRIGGER-ADV tests use 4000-char windows for this method. Increase this test's window to 4000 to match, so a regression in the perp path's price field handling is also caught.

N3. T16: 1500 ms settle delay after perp-to-spot transfer is hardcoded and may be insufficient on loaded testnet

e2e-testnet-test/index.ts line 1337

The comment notes this is to "let transfer settle." On testnet, HL's spot pool update can occasionally lag more than 1.5s under load. If the buy executes before the USDC is visible in the spot pool, the order fails with "insufficient balance" and T16 throws. The outer catch block cleans up correctly, so this is not a correctness issue — just a flaky-test risk. A warning log before the buy confirming the spot balance covers the expected $20 would make intermittent failures easier to diagnose.

Risks to monitor post-deploy

1. Spot TP fill detection not yet implemented in position-monitor M1. The M1 fill-check block at line 572 handles exchange_sl_order_id (SL + trailing stop). Native TP fills on spot go through the exchange_tp_order_ids path (separate code section not in this diff's review scope). Confirm that path exists and handles spot positions, otherwise spot TP fills will be detected only via balance reconciliation (phantom path), potentially with a 10-minute delay.
2. AMZN and other equity perps with bare-string status: exchange_sl_order_id remains null. When HL returns "waitingForTrigger" as a bare string (no oid), slResult.success is false and exchange_sl_order_id is never written. Software fallback is the only protection for these positions. Monitor Sentry for "Native SL placement failed: HL_STRING_STATUS:waitingForTrigger" — if it fires frequently, the right fix is a retry loop (wait 500ms, re-fetch the resting orders to extract the oid), not continued suppression.
3. Spot trailing trigger cancel+replace window. The gap between cancel and re-place on spot is acknowledged in the code (comment at line 1939). If the price drops past the trail level during this window, the software retrace check at line 2038 is the backstop. Monitor trailing-stop events on spot positions for cases where close_reason="trailing_stop" without a preceding oid in the audit log (would indicate software path fired, not native).

Safe

  • Wire field r: false for spot and r: true for perp are correctly set and the TRIGGER-ADV source-grep test covers both unconditionally.
  • grouping: "na" for spot is correctly applied and does not regress the perp "positionTpsl" path.
  • Spot asset index (spotToken.index, not SPOT_INDEX_OFFSET + X) is correctly taken from HL's own spot meta rather than a hardcoded offset calculation.
  • b: request.side === "long" is correct for closing triggers: a long SL is a sell (b=false), a short SL is a buy (b=true). The perp branch uses the same expression and has been in production.
  • M2 cancel in trade-executor (lines 1002–1050) correctly passes isSpotClose explicitly for both SL and TP cancels, preventing dual-listed asset index confusion.
  • _parseTriggerOrderResponse is shared by both paths and handles all four cases (resting, waitingForTrigger, bare string, error) correctly. The oid !== "number" type guard on both success paths prevents undefined orderId from being stored.
  • T14 two-cancel proof design is sound: Step B failing after Step A would catch a wrong-index-actually-cancelled regression.
