Finding: health-check has an unconditional early-exit guard at the top of its handler: if (environment === "staging") { return skipped }. So disabling position-monitor and deposit-monitor on staging will not cause false-positive health alerts — health-check never runs its heartbeat checks there.

3. market-context-refresh, position-monitor, and deposit-monitor were all reset to active=true by 20260507 — this pattern will recur
Finding: 20260507000001_cron_shared_secret_header.sql does cron.unschedule + cron.schedule on position-monitor, deposit-monitor, volatility-check, market-context-refresh, signal-performance-tracker, signal-review-weekly, and social-engagement — without any staging guard and without a follow-up alter_job(active := false). This is exactly the pattern that CLAUDE.md warns causes active=true resets on staging. Every cron proposed for disabling in this batch was re-enabled by that migration, which is why 20260527000001 had to re-disable some of them.

Why it's amber: The proposed migration will correctly use cron.alter_job(jobid, active := false) to disable. But the next migration that reschedules any of these crons (e.g., a future auth header change, a schedule tweak) without a staging guard will silently re-enable them again. The fix is narrow: the new migration should be written and it will work, but it does not solve the structural drift risk.

Action (for this migration): In the migration comment, call out which crons in this list were previously disabled by 20260410000002 or 20260527000001 and were re-enabled by 20260507000001. This documents the re-enable risk for the next developer who reschedules one of them.

Action (follow-up, not this PR): Consider a linter rule or a post-cron.schedule pattern that always follows with a staging-conditional alter_job(active := false) in migrations that reschedule existing crons.

Safe (8)
health-check false-alert risk: None
health-check exits immediately on staging (ENVIRONMENT=staging). Disabling position-monitor and deposit-monitor will not cause any health alerts. Confirmed by reading health-check/index.ts lines 161-167.
position-monitor idle behavior: No heartbeat row written
When position-monitor finds zero open positions (lines 280-286), it returns immediately without writing to any heartbeat table, health_check_acks, or downstream artifact. The health-check monitors MAX(updated_at) FROM positions WHERE status='open' — if there are no open positions, the query returns null and the check is skipped entirely (lines 213-228 of health-check). Zero false alerts from this path.
deposit-monitor idle behavior: health-check conditionally skips
health-check reads MAX(updated_at) FROM user_wallets and only fires if latestWallet?.updated_at is non-null (lines 238-251). On staging with no active wallets, this will return null and the stale-wallet alert will never fire. Safe to disable deposit-monitor.
volatility-check dependency: None for active staging crons
run-signals reads volatility_state but run-signals is already disabled on staging (by 20260410000002). No other active staging cron reads volatility state. Stale volatility data has no downstream effect on staging.
market-context-refresh dependency: None for active staging crons
run-signals calls readMarketContext(supabase) (line 733) but run-signals is already disabled on staging. signal-persistence imports readMarketContext but only within run-signals's call stack. No active staging cron reads market_context. Stale data is inert.
sec-edgar-poll / earnings-calendar-sync: No active downstream on staging
These write to tables read by run-signals and brief-generator. Both are already disabled on staging. Tables going stale will not affect any active staging cron or E2E test flow.
expire-pending-proposals (kept active): No dependency on disabled crons
expire-pending-proposals is a pure DB sweep — it sets proposals from pending to expired based on expires_at timestamp. It has no dependency on position-monitor, deposit-monitor, or any of the disabled crons. Test trade proposal flows will not be affected.
Migration technique: cron.alter_job is correct
SELECT cron.alter_job(jobid, active := false) is the correct approach. It modifies the existing job record in place without resetting any other fields. It is idempotent (disabling an already-disabled job is a no-op). The staging-guard pattern via vault project_url ILIKE is the established convention in this codebase (used by three prior staging migrations). Using this pattern will not re-enable jobs on production.
