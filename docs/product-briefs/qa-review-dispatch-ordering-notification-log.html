Worth a Look (2)
W1 notification_log fire-and-forget may drop on isolate exit _shared/notify.ts:2816, notify.ts:2473
Symptom
Both inserts use .then().catch() without await. If the Deno isolate is terminated immediately after Promise.allSettled(tasks) returns, the in-flight DB insert can be dropped before Supabase gets the request.
Root cause
Deno isolates do not guarantee execution of unresolved microtask chains after the top-level handler returns. In run-signals, the handler returns a Response object synchronously after await Promise.allSettled(crosscheckPromises). The two notification_log inserts initiated inside notifyBatchProposals and notifyTradeProposal are fire-and-forget; they're not awaited before the response is returned. There's a real (if low-probability) window where the isolate shuts down before the HTTP round-trip to Supabase completes.
Context
The same pattern is already in use for cacheNewsToSupabase and is accepted as fire-and-forget there. For an audit trail, the stakes are higher: a dropped log row means the GOLD scenario that motivated the original bug would still be invisible in the DB (notification sent, no audit row). That said: in practice, the notification dispatch runs well before the isolate exits (crosscheckPromises await follows it), so the risk window is narrow. This is not a data-correctness issue in the financial sense—no money is at risk. It's a monitoring gap.
Fix
Collect the log insert promises into crosscheckPromises (or a parallel logPromises array) and include them in the final Promise.allSettled before returning. Alternatively, since notifyBatchProposals is already awaited via emailTasks in run-signals, move the log insert to be awaited inside notifyBatchProposals itself (it's already after sends complete—just change the call chain). Low effort; removes the uncertainty entirely.
W2 notifyTradeProposal's notification_log insert is not IS_STAGING-guarded _shared/notify.ts:2764–2837
Symptom
In notifyTradeProposal, the IS_STAGING early return is at line 2764. The notification_log insert at lines 2815–2836 is positioned AFTER that guard—so it is correctly skipped on staging. But only if the code reaches the guard. For the case where notifyBatchProposals calls notifyTradeProposal for the n=1 path, IS_STAGING inside notifyBatchProposals returns early at line 2400 before the delegating call, so it's safe. However, if notifyTradeProposal is called directly on staging (not via the batch path), it will correctly return early before the insert. Verified: the guard covers the single-call path correctly.
Actual risk
Minor: the guard is correctly placed for all paths. The concern is notifyBatchProposals's log insert at line 2473: that function's IS_STAGING guard is at line 2400, and the insert is at line 2473 (after the guard), so it is correctly unreachable on staging in the batch (n=2–15) path. No gap in the implemented code. However: the test for this (BATCH-ADV-SRC: batch dispatch guarded by !IS_STAGING) only checks run-signals dispatch, not the notify.ts-level guard on the log insert. A test verifying the insert never fires on staging (via source pattern: that the insert block appears after the IS_STAGING return) would add confidence.
Fix
Add a source-verification test asserting that the notification_log insert in notifyBatchProposals appears lexically after the IS_STAGING guard line. This is a test gap, not a code gap—the production behavior is correct.

Add a source-verification test: read run-signals/index.ts, find the index of pendingEmailsByUser.size > 0 && !IS_STAGING and the index of generateDigestInline, assert the dispatch index is lower. One assertLess call; low maintenance burden.
Risks to Monitor Post-Deploy
Duplicate notification_log rows (low risk)
If run-signals retries after an HTTP 546 (the original trigger condition), the asset loop replays. Proposals already written to the DB would be de-duped by generateTradeProposals (existing idempotency). But if n proposals were already dispatched and logged in the first run, the retry could log them again if the first run's emails were sent but the isolate died before returning 200. This is bounded by the 15-minute cron interval. Monitor notification_log for duplicate proposal_id entries after any 546 event.
Digest skip on retry after 546 (acceptable, but watch)
If batch-1 546s after the dispatch block but before the digest, the next recovery window (12:10 UTC) will generate the digest. The shouldAttemptDigest check uses todaysDigest DB lookup, so digest won't double-fire. This was the pre-existing design. No new risk introduced by the ordering change.
