Verdict: Root Cause — PARTIALLY CORRECT, ONE MAJOR CAVEAT

OOM is a plausible explanation for the batch-1 546 but the evidence is circumstantial. The concurrency fix addresses one real pressure (simultaneous LLM + HL candle allocations), but the single most likely cause of the NEAR+SKHX notification miss is independent of OOM — it is a structural race between the asset loop and the post-loop dispatch, which the concurrency fix does not address. See Blocking finding B-1.

Verdict: Fix — INSUFFICIENT. Does not close the notification-miss race.

Reducing concurrency from 5 to 3 may reduce peak heap, but it does not fix the failure path that explains zero notification_log entries for NEAR+SKHX. The dispatch block at line 595 is guarded by pendingEmailsByUser.size > 0. There is a code path where proposals are inserted to trade_proposals but pendingEmailsByUser ends up empty, causing silent skip — and that path has nothing to do with OOM. The fix needs to be paired with investigation of this path before any deploy.

Blocking Issues

BLOCKING B-1: There is a non-OOM code path that explains zero notification_log with proposals in DB

The diagnosis assumes OOM killed the process before the dispatch block. But there is a structural path that produces the exact same symptom — proposals in trade_proposals, empty pendingEmailsByUser, no notification_log rows — without any OOM.

The path:

// run-signals/index.ts lines 406-434
if (signalResult.signalChanged && signalResult.signal.color !== "grey" && signalId && !IS_STAGING) {
  const proposalResult = await generateTradeProposals(supabase, {
    ...
    deferEmail: true,         // <── instructs createProposal to return pendingEmail
  });

  for (const pending of proposalResult.pendingEmails) {
    const userId = pending.userId;
    if (!userId) {
      console.warn("...missing userId — skipping email", pending.proposalId);
      continue;                // <── THIS skips the map insertion silently
    }
    if (!pendingEmailsByUser.has(userId)) pendingEmailsByUser.set(userId, []);
    pendingEmailsByUser.get(userId)!.push(pending);
  }
}

Now look at createProposal (proposal-generator.ts line 824):

const proposalNotification: TradeProposalNotification = {
  ...
  userId: user.privy_did,   // <── set here
};

if (options?.deferEmail) {
  return { created: true, autoApproved: false, pendingEmail: proposalNotification };
}

userId is always set on the notification object (it comes from user.privy_did from fetchEligibleUsers). So the !userId guard at line 428 should never fire in normal operation. However — if generateTradeProposals throws an exception (DB error, HL API error, Anthropic rate limit during the eligibility checks) after the trade_proposals INSERT completes but before returning pendingEmails, the proposal row exists in the DB but proposalResult.pendingEmails is never populated into the map.

Inspect generateTradeProposals: the per-user loop has a try/catch (line 423) that catches errors and pushes to result.errors[] without aborting. However, if an error occurs inside createProposal after the DB insert but before returning pendingEmail — specifically between the INSERT (line 757) and the return (line 847) — the error is caught, but pendingEmail is never pushed to result.pendingEmails. More critically: the logAudit call at line 771 makes async DB writes after the INSERT. If logAudit throws synchronously (rare, but possible if supabase client is in a bad state), the return at line 847 is never reached.

The most concrete non-OOM path: if the asset loop's per-asset runWithConcurrency worker catches an error at the outer level (line 528-536) — i.e., generateTradeProposals itself throws rather than returning an error-bearing result — then the for (const pending of proposalResult.pendingEmails) block at line 426 is never reached. The insert was already committed by Postgres (it doesn't get rolled back by a JS exception). Result: proposals in DB, nothing in map, zero notification_log.

What to check before dismissing this: Query notification_log for the NEAR/SKHX run time window. Also query supabase_audit or Sentry for errors from that batch. If there are Sentry events from the NEAR/SKHX asset task at +42s or +97s, this path is confirmed and the fix needed is not concurrency reduction but error propagation hardening in the asset worker.

Why this matters: If B-1 is the actual cause, reducing concurrency from 5 to 3 does nothing to prevent the next miss. The fix is wrong.

Worth-A-Look Items

WORTH-A-LOOK W-1: Timing analysis after fix — batch-1 can still approach 150s under heavy signal-change conditions

With concurrency 3 and 11 assets in batch-1, the pattern is:

WaveAssetsWorst-case time (all signal-change)
Wave 13 assets concurrently~50s (HL candles + LLM brief + proposal gen)
Wave 23 assets concurrently~50s
Wave 33 assets concurrently~50s
Wave 42 assets concurrently~50s

Worst-case asset loop: ~200s — which exceeds the 150s wall-clock limit. In practice, most assets don't change signal on every run, so typical time is much lower. But on a major market move (BTC flash crash, for example), all 11 batch-1 assets could change simultaneously. The news pre-fetch adds ~5-15s before the loop. Even at concurrency 3, the 150s risk is not eliminated — it's reduced in the average case but not in the worst case.

Before deploying, estimate the probability of simultaneous signal changes across all 11 assets. If it has happened historically (e.g., March 2020, FTX collapse style events), this fix provides false assurance.

WORTH-A-LOOK W-2: Brief staleness recovery fires per-asset and adds LLM calls even with no signal change

In signal-persistence.ts lines 559-592, every asset that hasn't had a brief in 4+ hours triggers a generateAndSaveBrief LLM call — the "staleness recovery" path. This fires regardless of whether the signal changed. With 11 assets in batch-1 and concurrency 3, if 6+ assets haven't had a brief recently (e.g., after a cron gap or deploy), wave 1 could run 3 simultaneous staleness-recovery LLM calls on top of market-context fetches. This is a real heap contributor that isn't gated by signalChanged.

The concurrency fix helps here too, but the existence of this path means that even "quiet" signal runs (no changes) can still hit substantial LLM-call concurrency. This is worth documenting as a second memory pressure driver beyond the obvious signal-change path.

WORTH-A-LOOK W-3: auto-execute queue runs sequentially BEFORE dispatch — adds unbounded time before notifications

Post-loop ordering (lines 545-606):

1. autoExecuteQueue (sequential, unbounded)
2. pendingEmailsByUser dispatch
3. generateDigestInline

If multiple auto-approved proposals land in the same batch (e.g., 3 users with full_auto each have a close+open pair on different assets), the auto-execute queue processes sequentially. Each executeTradeProposal involves an HL API call + DB update. Under high market-activity conditions this could add 20-40s before the email dispatch even starts. This is not new to this fix, but the concurrency reduction makes the loop run longer, so the window before the dispatch grows. Worth timing in staging.

Question-by-question analysis

Question Finding Confidence
Q1: Is OOM the correct root cause for the 546? Plausible but unconfirmed. At concurrency 5, simultaneous per-asset memory includes: HL candle arrays (~500 candles × 6 fields × 8 bytes × 5 assets = modest), LLM response strings (each Haiku/Sonnet response ~2-10KB), news fetch results (already in news_cache, not re-fetched per asset), persistSignalResult intermediate objects. The news pre-fetch runs BEFORE the loop and is already complete — it is NOT held during asset processing (only the cached DB rows are read per-asset). The most likely heap driver is simultaneous LLM brief generation: if 5 assets all trigger briefs simultaneously, you have 5 in-flight Anthropic API responses buffered. That's non-trivial but also not enormous. An alternative cause: cold-boot memory from dynamic imports in generateDigestInline (7 modules loaded simultaneously if digest fires). At hour ≥8, the digest triggers AFTER the asset loop, so this doesn't explain 44s failure. 546 at 44s into an 11-asset batch suggests the loop itself is the culprit, not digest. Moderate
Q2: Does concurrency 3 actually reduce memory? Yes, modestly. Each concurrent task holds: HL candle response (parse/decode in-flight), computeIndicators output (small), LLM API response stream (if signal change), proposal generation state. Reducing from 5 to 3 cuts simultaneous LLM responses from 5 to 3. News items are NOT per-asset memory — they're read from news_cache via a DB query that returns a filtered subset. The reduction is real but the delta per concurrent task is modest (~2-5MB saved). Whether this clears the Micro plan limit depends on how close to the limit the baseline is. Moderate
Q3: Timing after fix Batch-1 (11 assets, concurrency 3): 4 waves. Worst case (all signal-change): ~200s. Exceeds 150s wall-clock. Typical case (2-3 assets change): ~60-80s. Batch-2 (8 assets, concurrency 3): 3 waves. Worst case: ~150s. Right at the limit. Moderate
Q4: Non-OOM path to zero notification_log? YES — see B-1. If the per-asset worker task throws an exception that is caught by the outer catch (line 528-536), the pendingEmails loop at line 426 is never reached for that asset. Proposals already inserted to DB by Postgres are NOT rolled back (no transaction wrapping the INSERT + notification flow). This produces the exact failure signature: proposals in trade_proposals, nothing in pendingEmailsByUser, zero notification_log rows. High
Q5: New failure modes from slower execution? Slower loop means more elapsed time before email dispatch. For auto-execute users, the window between signal change and position entry grows (proposal sits in auto_approved longer before execution). Not a correctness issue but could mean worse fills on fast-moving assets. The auto-execute queue is sequential post-loop, which is unchanged. Low concern

No. Even if OOM is confirmed as the batch-1 cause, the NEAR+SKHX miss (batch-2) at 100s is not explained by OOM from concurrency-5 asset processing — batch-2 only has 8 assets and skips the news pre-fetch. The reported +97s for SKHX proposal suggests the batch ran near but possibly under the 150s limit before the dispatch. A non-OOM error in the worker task is the more likely explanation for that miss. The fix is incomplete. High

Memory profile: what's actually in-flight at concurrency 5

Per concurrent asset task, in-flight simultaneously:

  • HL candle fetch: 4H candles (~500 bars) + daily closes (~200 bars) as raw arrays. After computeIndicators runs, the raw arrays can be GC'd. Estimate: ~200KB per asset at parse peak.
  • LLM responses (signal-change only): Haiku event extraction response + Sonnet brief response. Streamed but buffered until complete. Estimate: ~5-20KB each. 5 simultaneous = ~200KB total.
  • News context (NOT re-fetched per asset): Read from news_cache table via DB query. Filtered subset returned per asset. The full news corpus (~100-300 items) stays in the DB, not JS heap.
  • DB result sets from fetchEligibleUsers, fetchPreviousState, etc.: Small (few rows). Not a concern.
  • Proposal notification objects: Tiny, deferred to map.

Raw estimate at concurrency 5 with all assets signal-changing: ~5MB LLM + ~1MB candles + overhead. This seems modest for Micro plan OOM, but Supabase Edge Functions (Deno) may have a much lower practical heap limit than the nominal MB figure. If the Micro plan limit is 150-200MB and the baseline worker footprint (all shared imports, supabase client, news-fetcher module) consumes 100-120MB, 5×~1MB concurrent tasks could plausibly tip it. This is speculative without an actual heap trace.

Required steps before implementing this fix

  1. Query Sentry for the NEAR/SKHX batch-2 run time window. Look for any exceptions thrown from the asset worker at +42s or +97s. If you find a JS error that was caught by the outer catch (err) at line 528-536, that is B-1 and the fix required is not concurrency reduction.
  2. Query notification_log for the exact run time. Confirm there are truly zero rows, and also check whether any Telegram send for NEAR or SKHX appears (the Telegram path in notifyBatchProposals fires before the audit row insert — absence of Telegram AND audit rows together is strong evidence the dispatch block was never reached).
  3. If B-1 is confirmed: Fix the error propagation in the asset worker so exceptions from generateTradeProposals still result in any already-constructed notifications being added to the map before the exception is caught. Consider wrapping the inner block more tightly so the map-population code is reached even on partial failures.
  4. If OOM-only is confirmed: The concurrency change is a reasonable mitigation, but also add a per-batch wall-clock guard so that if the loop takes >120s, a warning is logged and the dispatch still fires.
  5. Staging verification: Force a run with NEAR+SKHX by manually triggering a signal change for both assets in the same batch window. Confirm notification_log rows appear.

Confidence assessment: Will this fix resolve the NEAR+SKHX miss permanently?

Low. The concurrency change addresses memory pressure on a warm worker, which may reduce batch-1 546 frequency. But the NEAR+SKHX miss was in batch-2 (8 assets, no news pre-fetch, skips digest). The proposed fix does not address the code path where a mid-loop exception causes proposals to land in DB without being added to pendingEmailsByUser. Until Sentry logs from that run are checked and the exact failure point is identified, deploying this fix as a complete solution risks a false sense of closure. The miss will likely recur under the next high-volatility multi-asset signal event.

Confidence in root-cause diagnosis: Moderate. OOM is plausible for batch-1. The NEAR+SKHX batch-2 miss has a more likely non-OOM explanation that was not investigated before proposing this fix.
