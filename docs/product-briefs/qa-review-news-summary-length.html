Worth a Look

Worth a look #1
Prompt says "will be rejected" but nothing rejects it
news-classifier.ts:128: the prompt reads "Anything longer is over-budget and will be rejected." No code in classifyNewsItem() (lines 277-285) validates that the returned summary is ≤110 chars before persisting it. A model that runs to 140 chars will pass schema validation (maxLength=180), pass the typeof guard, and be written to news_cache.summary unchanged. The render-side cap then silently truncates with "…" — same symptom as before, just less frequently.

The fix is self-consistent as a "best effort" approach, and the render-side safety net is real. But the prompt's "will be rejected" language overpromises: there is no rejection path. The net result is the truncation problem is mitigated in frequency, not eliminated, for models that ignore the tighter limit.

This is acceptable to ship if the intent is "reduce, not eliminate" — but the stated intent in the brief is "so new summaries are generated to fit the display", which implies a stronger guarantee than the code provides.
Consider: add a post-call guard in classifyNewsItem() — if summary.length > 110, either null it out (to degrade gracefully) or log a warning counter. Alternatively, adjust the prompt to say "aim for" rather than "will be rejected" to match actual behavior.
Worth a look #2
news-enrich has its own summary extraction that does not share the classifier helper
news-enrich/index.ts:309-315: news-enrich has its own inline classifyOne() that duplicates the summary extraction logic from classifyNewsItem(). Both use the shared NEWS_CLASSIFY_SYSTEM_PROMPT from news-classifier.ts, so the prompt fix reaches news-enrich correctly. However, the duplication is a maintenance surface: if a future prompt change is applied in one place but not the other, the behavior diverges. This is pre-existing, not introduced by this PR, but worth flagging as a follow-up dedup task.
Flag for cleanup: consolidate news-enrich's classifyOne() to delegate to the shared classifyNewsItem() from news-classifier.ts, rather than duplicating the extraction.

Safe

Safe: Legacy rows
Legacy rows (written under the 160-char prompt) will still render truncated. This is explicitly documented in notify.ts:1799-1801 and is the expected transient state. The render-side truncation (lines 1804-1807) handles it gracefully with a word-boundary search before cutting, which is correct.
Safe: Other summary consumers
The news_cache.summary field is consumed by: (a) renderNewsModule in email (already has the 110-char cap), (b) digest-context.ts:494 which only checks ≥30 chars (length-agnostic beyond that), (c) news-detail-generate which reads the same field but renders it in a full-page card via Paragraphs component with no length cap (longer is fine there), and (d) the NewsDetail frontend page (same — uncapped, no truncation issue). The fix is scoped correctly: it only addresses the email rendering path where length matters.
Safe: news-detail-generate summary is a separate task
news-detail-generate/index.ts:389 uses taskName: "news_summary", not news_classify. Its prompt is independent and generates a multi-sentence article summary for the detail page. The 60-110 char constraint change does not affect it.
Safe: maxLength:180 rationale is sound
Keeping maxLength:180 in the schema (registry.ts:101) while tightening the prompt to 110 is the right call. Lowering maxLength to 110 would cause schema_fail when a model runs slightly over, losing the classification result entirely (null summary + no chips). The render-side cap provides the visual limit without a hard failure. The comment at registry.ts:94-100 explains this correctly.
