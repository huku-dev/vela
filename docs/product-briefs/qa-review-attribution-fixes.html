Blocking

Blocking
_shared/llm/registry.ts:262 + position-holder-brief/index.ts:482–487

Schema enforces type: "string" on clause, so {"clause": null} always triggers schema_fail — null path unreachable, wrong-asset headlines still get through via the raw-headline fallback

Symptom
When the LLM correctly identifies a misattributed headline (e.g. "Solana activity hits record high" tagged to "spx") and returns {"clause": null}, the in-process validator in schema.ts:walk() rejects it because null does not satisfy {type: "string", minLength: 10}. The adapter sets status = "schema_fail" and returns. Back in rewriteCatalyst, line 482 reads:
if (res.status !== "schema_fail" && res.parsed)
When status is schema_fail this condition is false — the null check on line 485 is never reached. The function falls through to the raw-headline fallback at line 500 and returns the cleaned headline string. pickCatalystClause gets a non-null string, returns it, and the wrong-asset catalyst line appears in the message. The original bug is not fixed.
Root cause
POSITION_CATALYST_SCHEMA (registry.ts line 262) declares clause: { type: "string", minLength: 10, maxLength: 120 }. This is correct for the happy path but does not accommodate the deliberate null sentinel the prompt instructs the model to return. The schema has no cascadeOnSchemaFail: true, so a schema_fail on the primary (Groq) does NOT cascade to the fallback; it returns with status schema_fail immediately. The combined effect: the null path defined in the prompt and in rewriteCatalyst's code is 100% unreachable in production.
Evidence chain
  • schema.ts:22-26: typeMatches(null, "string") returns false; error pushed, validation fails.
  • openai-compatible.ts:113-116: status = "schema_fail" set; parsed is set to the raw extracted JSON object.
  • rewriteCatalyst line 482: condition gates on res.status !== "schema_fail" — short-circuits to false.
  • rewriteCatalyst line 500: fallback returns ensureTrailingPeriod(shortHeadline(title)) — the wrong-asset headline, cleaned but still wrong.
Fix (two options, pick one):

Option A — Fix the schema (preferred). Change POSITION_CATALYST_SCHEMA in registry.ts to allow null:
clause: { oneOf: [{ type: "string", minLength: 10, maxLength: 120 }, { type: "null" }] }
Note: the home-grown validator in schema.ts may not support oneOf. Verify first. If not, use an enum approach: remove type from the clause property and add enum validation is skipped for non-enum fields — or expand schema.ts to handle type: ["string", "null"] union. Simplest: add type: "null" as a secondary pass or change the schema property to have no type field and only enforce presence of the key.

Option B — Change the fallback in rewriteCatalyst. Instead of guarding on res.status !== "schema_fail", also check res.parsed when schema_fail occurs: 
// After the try block:
if (res.status === "schema_fail" && res.parsed) {
  const parsed = res.parsed as { clause?: string | null };
  if (parsed.clause === null) return null;
}
// Then fall through to raw headline
This is the minimal diff: it peeks at the schema_fail parsed payload before giving up on the null path. Option A is cleaner long-term; Option B ships faster.

Also add cascadeOnSchemaFail: true to the position_catalyst task in registry.ts so that if Groq schema-fails on a legitimate rewrite attempt (not a null), the fallback chain fires instead of silently swallowing the failure.

_shared/digest-context.ts:481 — NON_RELEVANCE_PATTERNS[0]

Pattern /\bno direct impact\b/i can false-positive on legitimate macro summaries for certain assets

Concern
The phrase "no direct impact" appears in legitimate analyst summaries for assets that are genuinely insulated from a macro event. Example: "The rate hike has no direct impact on Bitcoin's supply schedule, but sentiment may shift." This is an accurate, relevant Vela summary for BTC — it tells the user the asset is unaffected by the event, which IS useful context. The pattern would drop this card entirely.
Concrete reproduction path
A news-classifier summary for BTC tagged with relevant_assets: ["btc"] that reads "The Fed rate decision has no direct impact on BTC's on-chain dynamics, but risk-off sentiment could weigh on price" — importance 4, primary asset BTC — gets silently dropped by line 488.
Counterargument
The original problem was summaries that explained WHY the asset is irrelevant ("The headline concerns Zoom, not Amazon..."). The pattern targets that. The "rate hike has no direct impact on BTC supply" case is a different meaning of the phrase. The risk is real but may be rare in practice depending on how the classifier phrases its summaries.
Consider: Tighten the pattern to match the classifier's actual output format. The Zoom/Amazon case reads like "no direct impact on [asset]" or "the tagged asset sees no direct impact". Change the first pattern from /\bno direct impact\b/i to /\btagged asset sees no direct impact\b/i or anchor to the classifier's templated phrasing. Alternatively, combine with the concerns .{1,40}, not pattern as a required co-occurrence.
Worth a Look
position-holder-brief/index.ts:424–428 — normCandidate search scope

normCandidate searches only highRows (48h window), not a fresh 24h query — could miss importance=3 stories published between 24h and 48h ago that aren't in the 20-row limit

Concern
highRows is fetched with .gte("published_at", sinceHigh) (48h) and .limit(20). The importance=4 search uses this full set. The normCandidate search then re-filters highRows to only rows within the 24h window. This is logically correct: any importance=3 story within the last 24h that exists in the DB will appear in highRows IF the 20-row limit wasn't hit. If there are more than 20 news rows in the last 48h for a popular asset (e.g. BTC during a volatile week), the 20-row limit cuts off older rows — and since results are ordered published_at DESC, older rows (including those in the 24–48h band, plus some 0–24h rows near the cutoff) may be excluded.
Practical severity
For high-activity assets (BTC, ETH) during market events, 20 rows in 48h is reachable. If the 24h window has 15 importance=3 rows near the boundary and 5 importance=4 rows dominate the top, the limit is fine. But if all 20 rows are from the past 6h, importance=3 stories from 20h ago are invisible to normCandidate. The position brief may show no catalyst line when one existed. This is a silent degradation, not an incorrect line.
Consider: Issue a separate query for the normal-importance candidate with .gte("published_at", sinceNormal) and .limit(5), independent of highRows. Or increase highRows limit to 50. Low urgency — silent miss, not wrong output.

Risks to Monitor Post-Deploy

If Fix 1 is applied (schema fix):
  • Monitor llm_call_log for position_catalyst schema_fail rate. Before the fix, every null-return was a schema_fail silently swallowed. After the fix, null-return should be status=ok. If you see a spike in raw headline fallback use (no LLM-rewrite), that's a signal the null path is firing more often than expected — check whether the tagging pipeline is over-tagging assets.
  • Add cascadeOnSchemaFail: true to position_catalyst at the same time — without it, a legitimate rewrite attempt that fails schema (e.g. LLM over-produces, violates maxLength) silently swallows rather than cascading to Groq fallback.
  • The NVIDIA qwen fallback does not have additionalProperties: false bypassed for this task. If qwen is used as fallback after fixing, its extra-key habit (adding "reasoning", "_meta") will hit additionalProperties: false in the schema and produce schema_fail again. Consider also setting additionalProperties: true on POSITION_CATALYST_SCHEMA (consistent with the NVIDIA-safe pattern used on other tasks).
Pattern monitoring for NON_RELEVANCE_PATTERNS:
  • There is no logging when a row is dropped by NON_RELEVANCE_PATTERNS. A false-positive drop is invisible. Add a console.log("[digest-context] skipped non-relevance row: ...", r.id, primary.symbol) so it's diagnosable via Supabase log viewer after deploy.
  • After first deploy, scan the logs for any unexpectedly dropped rows for BTC, ETH, SOL — assets with high news volume most likely to trigger the "no direct impact" pattern on legitimate macro summaries.
