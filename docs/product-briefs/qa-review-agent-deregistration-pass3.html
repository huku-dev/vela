Worth a Look (Amber)

A-1: No test asserts that captureException is called on DB write failure

_shared/wallet-provisioner.test.ts (no such test) + wallet-provisioner.ts:495

The three existing tests in wallet-provisioner.test.ts cover: regex matching, the agent_registered: false update, the dbErr destructure, and the export. None of them assert that captureException is called when dbErr is truthy. The new Sentry call is the entire point of W-1 and it has zero coverage. If a future refactor accidentally removes or gates the call, no test will catch it.

Add a test: WALLET-ADV: maybeHandleDeregistration calls captureException on DB write failure. Source-grep the function body for captureException inside the dbErr block, the same pattern the existing tests use for dbErr and agent_registered: false. This is a one-liner assertion consistent with the existing test style.

Nit (Gray)

Nit-1: Identical message string duplicated across all 5 captureMessage callsites

position-monitor/index.ts:805, 1755, 1912, 2206, 2369

Every callsite emits "Agent wallet deregistered on HL for <userId>: recovery triggered" with identical tags. This is not wrong, but if the message text ever needs to change, it must be updated in 5 places. It also means the Sentry issue title doesn't indicate which code path fired.

Consider extracting a helper function emitDeregistrationEvent(userId: string, context: string) that calls captureMessage with the context label appended (e.g. "trailing_stop_close", "profit_ladder_trim"). One change point, and Sentry groups by meaningful context labels instead of collapsing all 5 paths into one issue.

Nit-2: W-1 captureException passes user_id as a tag string, not in the Sentry user context

wallet-provisioner.ts:496

captureException(..., { tags: { ..., user_id: wallet.user_id } }) puts the user ID in the tag bag. Sentry has a dedicated user context (setUser) that enables person-level search and privacy controls. The existing sentry.ts exports setUser(userId) for exactly this. This is low-severity (tag is still visible in Sentry), but inconsistent with how user context is surfaced elsewhere.

Consider calling setUser(wallet.user_id) before captureException, or at minimum removing user_id from tags (Sentry user context is the canonical place).

Risks to Monitor Post-Deploy

R-1: trade-executor emits a captureMessage for the same deregistration event without a deregistration-specific Sentry event

_shared/trade-executor.ts:1204, 1209

When a trade-executor order fails and maybeHandleDeregistration returns true, trade-executor emits a generic "Order failed: ..." captureMessage with tags.deregistration = "true". It does NOT emit a separate deregistration event. Position-monitor now emits its own dedicated captureMessage with event: "agent_deregistration". These are different Sentry issues: trade-executor's event is tagged but not named the same way.

This is not a double-fire risk (they're on different code paths: trade-executor handles open proposals, position-monitor handles close/trim failures). However, a deregistration that triggers both a trade open AND a monitored close will appear in Sentry as two separate issues with different formats. Watch for this inconsistency causing confusion when triaging; consider aligning trade-executor to emit the same event: "agent_deregistration" tag and message format in a follow-up.
