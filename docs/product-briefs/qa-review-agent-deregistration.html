Blocking

B-1. position-monitor has five HL order callsites with no deregistration recovery

position-monitor/index.ts: lines 784, 1620, 1627, 1806, 1813, 2100, 2107, 2272, 2281

Symptom. When an agent wallet is deregistered on HL's side, trade-executor now correctly flips agent_registered=false on the next trade proposal. But position-monitor calls adapter.placeOrder and adapter.placeSpotOrder in at least five separate close-action paths (stop-loss, trailing stop, BB2 expiry, spot phantom-close, ladder trim at lines 2272/2281). None of these paths inspect the HL error for the deregistration string. When a deregistration happens mid-session, all open position closes will silently fail: closeResult.success === false, the error is logged to console, but agent_registered stays true in the DB. On the next position-monitor tick, the wallet query at line 308 filters .eq("agent_registered", true), so the user's wallet is still included, and every close attempt fails again until trade-executor next fires for that user and happens to hit the deregistration error. If the user has no pending proposals, recovery never triggers.

Root cause. The fix was applied only to trade-executor.ts's order failure handler. position-monitor's closeResult failure branches (e.g. lines 1752-1755, 1906-1910) log the error but do not call the deregistration detector.

Fix. Extract the deregistration-detection logic into a shared helper in _shared/trade-executor.ts or a new _shared/hl-errors.ts. Call it from every closeResult.success === false branch in position-monitor. The UPDATE pattern is identical: .from("user_wallets").update({ agent_registered: false }).eq("id", wallet.id). The wallet reference is already in scope at each callsite via walletMap.get(userId). Consider a single helper maybeHandleDeregistration(supabase, wallet, errorStr) that both callers can await.

Worth a Look

W-1. Raw HL error string sent to admin Telegram on deregistration path

trade-executor.ts: lines 1204-1267

orderError is set to rawError when isAgentDeregistered is true (the ternary at line 1205 only translates margin shortfalls). That raw HL string, e.g. "User or API Wallet 0xefe8...does not exist.", is then passed into notifyTradeResult at line 1267, which sends it to admin Telegram. It is also stored in trade_proposals.error_message at line 1234. The raw hex address leaking to Telegram is low-risk but noisy. More importantly, the admin will see a generic "order failed" notification with no signal that auto-recovery is already in progress, leading to unnecessary manual intervention.

Consider. Add a third branch for isAgentDeregistered that sets orderError to a plain-English string like "Agent wallet deregistered on Hyperliquid. Auto-recovery triggered.", or suppress the notifyTradeResult call entirely for this case and let the console.error + Sentry message be the signal.

W-2. Sentry event for deregistration is indistinguishable from generic order failure

trade-executor.ts: lines 1221-1231

captureMessage("Order failed: ...") fires for all order failures. There is no additional Sentry tag when isAgentDeregistered is true. In Sentry's issue stream, a deregistration event looks identical to a leverage glitch or RPC error. This makes it impossible to write a Sentry alert rule that pages on deregistration specifically, and post-incident queries require eyeballing the message text.

Consider. Add a deregistration: true tag to the captureMessage call inside the if (isAgentDeregistered) block, or call a second captureMessage there with a distinct name. Cheap change, high signal value.

W-3. UPDATE failure on user_wallets is silently swallowed

trade-executor.ts: lines 1212-1218

The await supabase.from("user_wallets").update(...). call at line 1212 destructures no error return. If the UPDATE fails (network blip, RLS policy mismatch, wrong wallet.id), execution continues silently: the console.error at line 1216 fires unconditionally, giving false confidence that recovery was triggered when in fact the DB was not updated and deposit-monitor will not re-register.

Consider. Destructure the result: const { error: deregErr } = await supabase.... If deregErr is set, log it and call captureException so the failure surfaces in Sentry. The console.error line should move inside the success branch.

Nit

N-1. Source-scan tests are structurally weak regression guards

trade-executor.test.ts: lines 1612-1645

Both new WALLET-ADV tests read the source file as a string and assert on the presence of identifiers or substrings. This means they pass as long as the relevant code is present anywhere in the file, regardless of whether it is actually reachable, correctly conditional, or even commented out. A refactor that moves the deregistration block behind a feature flag, or that renames isAgentDeregistered, would break the behavior but keep the tests green. The tests also do not verify that the UPDATE runs after the detection, only that both strings appear in file order. Consider adding at least one behavioral test using a mock Supabase client that asserts update({ agent_registered: false }) was called when the HL error string is returned.

N-2. Em dash in console.error log line

trade-executor.ts: line 1217

"Agent wallet deregistered on HL for ... — flagged for auto-recovery" contains an em dash. Project rules ban em dashes. Replace with a comma or colon: "Agent wallet deregistered on HL for ... flagged for auto-recovery".

Risks to Monitor Post-Deploy

M-1. Recovery depends on deposit-monitor seeing positive balance within 10 minutes

The deposit-monitor recovery path at line 652 fires only when !wallet.agent_registered && newBalance > 0. If the user's balance is exactly zero at the time deposit-monitor runs (unlikely but possible if a withdrawal just completed), re-registration is skipped that tick. The wallet stays unregistered until the next deposit. This is pre-existing behavior, not introduced by this change, but it is worth watching on the first real deregistration event post-deploy.

M-2. Up to 10-minute window where subsequent proposals for the same user continue to fail

After agent_registered=false is set, deposit-monitor re-registers on its next tick (up to 10 min later). If run-signals fires another proposal for the same user in that window, trade-executor will attempt the order again. The pre-flight adapter.getBalance() at line 344 will succeed (the wallet exists; it is just the agent registration that is gone), so the pre-flight does not block the order, and the order fails again, triggering another agent_registered=false UPDATE (idempotent but noisy). This is acceptable, but worth noting for triage.

Safe

  • Regex correctness. /user or api wallet .* does not exist/i matches the exact HL error string from the incident ("User or API Wallet 0xefe8...does not exist."). Case-insensitive, uses .* for the address portion. No false-positive risk against the margin-shortfall regex or trading-halted patterns.
  • Column target. The UPDATE targets user_wallets.agent_registered via .eq("id", wallet.id). wallet is confirmed non-null at this point (null-guard at line 329 throws before reaching this code path). No null-dereference risk.
  • Race condition on concurrent failures. Two parallel proposals failing simultaneously both issue UPDATE agent_registered=false on the same row. This is safe: both writes are idempotent (false → false). PostgreSQL last-write-wins semantics apply; no data corruption.
  • User-facing error message. orderError for the deregistration path falls through to rawError, which is the HL technical string. This is stored in error_message and passed to notifyTradeResult (admin-only). It is not shown to the user directly. No voice-rule violation for user-facing surfaces.
  • deposit-monitor recovery trigger. The recovery path at line 652 is correctly guarded and already battle-tested for the initial-registration case. Reusing it for deregistration is sound.
  • wallet-provisioner re-registration path. registerAgentIfNeeded is idempotent and handles the existing && !agent_registered case already. No changes needed there.
