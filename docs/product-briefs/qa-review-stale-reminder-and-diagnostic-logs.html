Worth a look

Diagnostic log fires inside staging-guarded block, but staging path never reaches it

The first diagnostic log ([run-signals] ${asset.symbol}: N deferred proposal email(s) queued) is placed at line 399–403, which is inside the block guarded by if (signalResult.signalChanged && signalResult.signal.color !== "grey" && signalId && !IS_STAGING) (line 386). On staging, proposal generation is skipped entirely, so proposalResult doesn't exist and the log never fires. That is functionally correct (the log would be meaningless with zero proposals on staging), but it means the first log provides zero observability on staging runs. The second log at line 557 correctly fires unconditionally and notes (staging — skipping), so post-loop visibility is present. The asymmetry is fine in practice, but worth confirming this is the intended behavior: no per-asset deferred-email logging on staging.

No action required unless per-asset staging observability is desired. Confirm intentional.
