Findings

AMBER-1: Software-fallback trailing-stop cancel passes isSpot=false for builder perp "na" OIDs position-monitor/index.ts:2093

When the trailing-stop triggers via the software-fallback path (i.e. pos.exchange_sl_order_id is set but the retrace-skip was bypassed), the cancel at line 2093 passes isSpot to cancelTriggerOrder: 

await adapter.cancelTriggerOrder(hlCoin, pos.exchange_sl_order_id, isSpot);

For a builder perp position, isSpot is false. cancelTriggerOrder resolves the asset index by checking spot meta first (getSpotMeta()), then falling back to resolveAsset() when isSpot=false. resolveAsset("xyz:GOLD") returns the builder-perp index, which is correct for cancelling builder perp "na" orders. This works today.

The gap: The comment at line 2090 says "Pass isSpot so cancelTriggerOrder uses the correct spot index" but says nothing about builder perps. A reader editing this cancel in the future may not realise builder perp OIDs are also "na" orders and must use the perp (not spot) index. If they add isSpot || isBuilderPerp to the cancel call by analogy with line 1975, they will break cancels for builder perps (builder perp assets are not in spot meta, so useSpotIndex=true and no spotToken returns an error without cancelling).

Recommended fix: Add a clarifying comment at line 2092 stating that builder perp OIDs use isSpot=false (perp cancel path) because builder perp assets are not in spot meta. No code change required, just documentation.

AMBER-2: Profit ladder TP placement skips builder perps silently; no isBuilderPerp check in skip condition trade-executor.ts:1739

The TP ladder skip condition at line 1739 is: 

const skipTpLadder = posType === "bb2" || posType === "bb2_30m" || openedAsSpot;

Builder perps (isBuilderPerp=true) are NOT in this skip list. This means when a builder perp trade executes, the code enters the TP placement loop at line 1740 and sends three partial-close trigger orders. Those orders do NOT set isSpot: true but they do correctly pass sizeInAsset (it is always passed for partial orders regardless of asset type since isPartial is true). The grouping in the adapter resolves to "na" because sizeInAsset makes isPartial=true, which is the correct grouping for partial TP orders.

Functionally this is safe. However: (a) the skip comment at line 1737 mentions "spot positions skip native TP" but does not mention whether builder perps are intended to place TPs; (b) if HL builder perp dexes do NOT support partial TP trigger orders, they will accumulate stale TP orders just like the SL orders did before this fix.

Recommended action: Verify with HL documentation or a test trade whether builder perp dexes support "na"-grouped partial TP trigger orders. If they do, add a comment confirming the intent. If they do not, add isBuilderPerp to the skipTpLadder condition before production traffic reaches builder perp assets with active TPs.

NIT-1: Tests are source-string matchers; none would catch a runtime-wrong pos.size value position-monitor.test.ts:1052-1066

All three BUILDER-PERP-ADV tests use assertMatch(src, /pattern/) against the raw TypeScript source. Test at line 1055 confirms the literal text isBuilderPerp && { sizeInAsset: pos.size } is present, but it does not verify that pos.size at the call site reflects the post-trim remaining size (not original size). The size update path at line 1571 writes pos.size = newSize after TP fills, which is correct. But a behavioral test that exercises a position with a prior trim and then checks the size passed to placeTriggerOrder would catch a regression here that the current tests cannot.

Not blocking: the existing code path is correct since local state is updated at 1571 before the trailing trigger path at 1988. Flagging as an improvement candidate.

NIT-2: Phantom-reconciliation and SL-fill TP cancel paths do not pass isSpot or isBuilderPerp position-monitor/index.ts:986, 1358

Lines 986 and 1358 cancel orphaned TP orders without passing an explicit isSpot flag: 

await adapter.cancelTriggerOrder(hlCoin, tpIds);

cancelTriggerOrder auto-detects from spot meta when the third argument is omitted (isSpot=undefined). For builder perp assets with hlCoin="xyz:GOLD", the auto-detect will not find it in spot meta, so it falls back to resolveAsset("xyz:GOLD") and gets the correct builder perp index. The cancels work correctly. This is a style inconsistency with the explicit isSpot pattern established elsewhere, not a bug.

Safety Analysis by Review Category

SAFE-1: Grouping logic is correctly scoped to non-spot builder perps hyperliquid-adapter.ts:1310-1314

isBuilderPerpAsset = !request.isSpot && request.asset.includes(":"). The !request.isSpot guard prevents accidentally routing genuine spot orders through builder-perp logic. No native perp coin names include a colon in HL's API (confirmed: HL asset names are plain symbols like "BTC", "ETH", "AMZN" for native perps). The condition is tight.

SAFE-2: Tick-1 coexistence window is safe (review question 3)

Existing positions with exchange_sl_order_id=null will, on tick 1, enter the cancel gate check (isSpot || isBuilderPerp) && pos.exchange_sl_order_id with a null OID, so the cancel does not fire. A fresh "na" order is placed and the OID is written. There is a brief coexistence window where the old stale positionTpsl order and the new "na" order are both live on HL. This is safe: both are reduce-only at the same (or very similar) price. If both fill, the second fill is a reduce-only against a zero position and HL rejects it. The stale positionTpsl orders will continue to accumulate on tick 2 and beyond only if the new "na" cancel-before-replace cycle fails, which Sentry catches via the non-HL_STRING_STATUS error path.

SAFE-3: isBuilderPerp scope in trade-executor is correct trade-executor.ts:636 vs 1665

hlCoin is declared at line 373 and isBuilderPerp at line 636, both in the outer function body. Line 1665 references isBuilderPerp without re-declaration. TypeScript's lexical scoping confirms it resolves to line 636's value. Both declarations use the same hlCoin.includes(":") check, so they are semantically identical.

SAFE-4: cancelTriggerOrder uses correct asset index for builder perp cancels hyperliquid-adapter.ts:1440-1457

With isSpot=false, the adapter skips the spot-index path and calls resolveAsset("xyz:GOLD"), which returns the builder-perp-dex asset index. This is the same index used when placing the "na" order, so the cancel targets the correct order on HL.

SAFE-5: isHLStringStatus guard change is logically equivalent and correct trade-executor.ts:1703

Old guard: !openedAsSpot && slResult.error?.startsWith("HL_STRING_STATUS:"). New guard: !needsExplicitSize && slResult.error?.startsWith("HL_STRING_STATUS:"). Since needsExplicitSize = openedAsSpot || isBuilderPerp, the new guard excludes both spot AND builder perp from the HL_STRING_STATUS suppression. Builder perps now use "na" grouping and return a real OID, so they should never produce HL_STRING_STATUS anyway. The guard correctly surfaces any unexpected string-status error from a builder perp as a Sentry alert rather than suppressing it silently.
