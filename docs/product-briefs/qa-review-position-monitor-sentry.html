Blocking

Blocking captureException is a no-op: initSentry is never called in position-monitor
position-monitor/index.ts line 35 (import), lines 1805 and 1849 (call sites)

captureException is guarded by if (!initialized || !Sentry) return; in _shared/sentry.ts:59. initialized is set to true only inside initSentry(). position-monitor/index.ts imports captureException but never imports or calls initSentry, and does not use the withSentry wrapper. The entire function uses a bare Deno.serve(async (req) => { ... }) at line 243 with no Sentry setup. Both new captureException calls will silently no-op on every invocation. The HYPE incident that motivated this change would produce zero Sentry events after deploy.

Comparable functions that correctly instrument Sentry: run-signals/index.ts calls await initSentry("run-signals") at the top of its handler (line 70) and imports flush for end-of-invocation flush. trade-webhook/index.ts calls await initSentry("trade-webhook") at line 826. Neither pattern is present in position-monitor.

Secondary issue: even if initSentry is added, there is no flush() call before the function returns at lines 372-375. Supabase edge functions terminate the isolate quickly; without a flush, events queued by the Sentry SDK may be dropped before they reach the network. run-signals calls sentryFlush before returning.

Fix
Add the missing import and init at the top of the handler. Also add a flush before the final return. 
// line 35 — change the import to:
import { initSentry, captureException, flush as sentryFlush } from "../_shared/sentry.ts";

// line 246, immediately after verifyCronAuth passes — add:
await initSentry("position-monitor");

// line 372, before the final return new Response — add:
await sentryFlush();

Worth-a-look

Worth-a-look Change A fires every 2-minute cycle during an HL API outage: no rate limiting on Sentry events
position-monitor/index.ts lines 1802-1808

placeTriggerOrder is called every 2-minute cycle per active trailing-stop position. Change A emits a captureException on every failed attempt. With N positions in trailing-stop state and a sustained HL API outage, this emits N * 30 Sentry events per hour. Sentry's default SDK-level deduplication does not apply across separate invocations of a short-lived function (each cold start has a fresh SDK state). During the HYPE incident (10 hours), this would have produced ~300 events for a single position. This is a noise risk, not a correctness risk, but it could obscure real errors in the Sentry feed and hit Sentry rate limits.

Change B (the perp-with-no-trigger alert at line 1848) fires only at close, so it emits at most once per position. No rate concern there.

Suggestion (non-blocking)
Consider replacing Change A's captureException with captureMessage(..., { level: "warning" }) and using Sentry's "group by fingerprint" or a per-position-per-hour counter to deduplicate. Alternatively, only emit on the first failure per position (check !pos.exchange_sl_order_id as a proxy for "never placed") and rely on Change B for the close-time alert. The simplest immediate fix: change captureException(new Error(triggerErr)) to captureMessage(triggerErr, { level: "warning", tags: { ... } }) with a Sentry fingerprint including pos.id so duplicates collapse.
Worth-a-look Two new source-inspection tests match on surrounding context, not the precise call shape
position-monitor.test.ts lines 1000-1012

Both tests use [\s\S]*? (lazy any-char) between the condition and the call, meaning they pass as long as both substrings exist somewhere in the file with the condition appearing before the call, not necessarily in the same block or even the same function. For example, Test 1 (triggerResult.error ... captureException(new Error(triggerErr)) would still pass if someone moved the captureException to a different branch as long as it remained after triggerResult.error in the source. Test 3 (!isSpot && !pos.exchange_sl_order_id ... captureException) would pass even if the captureException inside the block was removed and a different captureException existed later in the file (the current file has one at line 1805 which appears after the condition at 1830).

Importantly, running the tests right now confirms all four assertions pass against the real source, so they do detect the current code. The fragility is for future regressions where the call is moved rather than deleted.

Suggestion (non-blocking)
Tighten the regexes to require the condition and call to be on adjacent lines or within a small character window. For Test 3, anchor closer: /if \(!isSpot && !pos\.exchange_sl_order_id\)[\s\S]{0,200}?captureException/ with a character budget prevents false passes from unrelated later calls.

_shared/tradeable-assets.ts line 36

Per project memory (MEMORY.md): "SPCX enabled=false (pre-launch, auto-flip poller running)". The FALLBACK_TRADEABLE_ASSETS list includes spcx unconditionally. The fallback is used when the DB is unreachable, so on a DB outage the breaking-news quality gate would count SPCX chips as valid when SPCX is actually disabled. Either remove spcx from the fallback until it goes live, or add a comment noting the intended state mismatch.

Safe

Safe Change B false-positive analysis: no race with native-fill path

The concern was: could Change B fire on a position where a native trigger WAS placed and filled by HL, if the close routine cleared exchange_sl_order_id to null before reaching this code? Answer: no. When HL fills a native trigger, the position disappears from the exchange. On the next 2-minute tick, !exchangePos is true (line 761), and the function enters the "position missing from exchange" reconciliation path (lines 755-1200). That path calls return / continue before ever reaching the trailing-stop retrace check at line 1822. The exchange_sl_order_id: null write inside the native-fill path (lines 840, 969, 1197) happens in that separate branch. The software fallback path at line 1848 is only reachable when the position still exists on the exchange AND the retrace threshold is met, meaning HL has NOT filled a native trigger. The change B condition is logically sound.

Safe fallback asset list correctness for the 7 new assets

All 7 assets (zec, silver, copper, intc, skhx, spcx, xyz100) were added in prior documented sessions (MEMORY.md completed work log). Their lowercase IDs match the project convention (asset IDs are always lowercase symbol strings). The comment on line 23 correctly describes the fallback semantics and drift risk.

Safe tradeable-assets.ts Sentry usage is guarded with try/catch

The existing captureException in getTradeableAssetIds is already wrapped in a try/catch (lines 65-72) with a comment explaining that Sentry init failure should not block the fallback return. This file calls captureException directly without initSentry, which will also no-op silently since initialized is false. The no-op is harmless here because the fallback behavior (return hardcoded set) is correct regardless of whether Sentry captures the event.

Safe Change B does not double-fire at position close: ordering is correct

Change B fires before the software close executes (line 1848 precedes the placeOrder call at line 1879). The exchange_sl_order_id: null write at line 1919 is inside the if (closeResult.success) block, after the close. So the Sentry event captures the state before any DB mutation, and the null write happens only on a successful close. No risk of reading stale state.
