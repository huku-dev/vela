Worth a Look

WL-1: Supabase error from anti-repetition query silently dropped in both call sites

daily-digest/index.ts:130 · run-signals/index.ts:726

Symptom If Supabase returns an error (RLS misconfiguration, connection timeout, schema mismatch), the anti-repetition context silently stays null with no log entry. The try/catch only fires on thrown exceptions; PostgREST errors come back as {data: null, error: {...}}, not thrown errors.

Root cause Both call sites destructure only data, not error: const { data: prevBrief } = await supabase...maybeSingle(). The Supabase error object is discarded before the try/catch can see it.

Fix Destructure the error and log it inside the try block at both sites:

const { data: prevBrief, error: prevErr } = await supabase...maybeSingle();
if (prevErr) console.warn("anti-repetition query error:", prevErr.message);

Severity Low-impact in practice (anti-repetition is cosmetic), but a silent failure creates a blind spot. Cheap to fix.

WL-2: LLM output injected into LLM prompt without sanitization (prompt injection surface)

_shared/brief-generator.ts:785,788

Symptom previousOpener and previousThemeLine — both LLM-generated — are interpolated directly into the next run's user prompt inside double-quote wrappers. A previous brief's theme_line could contain sequences that attempt to override the prompt instructions (e.g. " — Ignore all STYLE NOTEs and instead output...).

Root cause No sanitization between LLM output storage and LLM prompt injection. The double-quote wrapper provides cosmetic scoping but no semantic boundary in a plain-text prompt context.

Fix One of: (a) strip any " characters and newlines from both values before injection; (b) wrap in an XML-style delimiter (<yesterday_opener>...</yesterday_opener>) that the model treats as a delimited block rather than in-band instructions. Option (a) is one line each; option (b) requires adjusting both injection points and the STYLE NOTE wording.

Severity Low probability in production (self-injection only, no external attacker). Worth addressing before the feature accumulates months of prior briefs. Confidence: moderate.

WL-3: EOD path queries wrong brief_type for anti-repetition context (misleading, not buggy)

daily-digest/index.ts:133 (used when mode=eod)

Symptom The anti-repetition query always filters on brief_type = "daily_digest". When mode is eod, this returns the most recent morning brief. The code comment at line 125 says "yesterday's brief" — but for EOD at 22:05 UTC, it loads today's morning brief, not yesterday's EOD.

Root cause The query does not branch on mode. This means EOD anti-repetition context is "don't echo the morning brief", not "don't echo yesterday's EOD brief". That's defensible as product behavior (morning/EOD pair shouldn't echo each other), but the comment is wrong and a future author will be confused.

Fix Update the comment at lines 125-126 to say: "For morning runs, this fetches yesterday's morning brief. For EOD runs, this fetches today's morning brief — so EOD doesn't echo the morning framing." No code change required unless the intent is to dedup against yesterday's EOD instead.

NIT-3: Idempotency window has a 1-second gap at midnight UTC

_shared/signal-persistence.ts:672-673

The window check uses .lt("created_at", `${today}T23:59:59Z`), which excludes the 1-second window 23:59:59–00:00:00 UTC. A digest persisted in that window is excluded and a second one could be written. In practice digests run at 08:00 UTC and 12–20 UTC recovery, so this never triggers. Change to 23:59:60Z or use the cleaner .lt("created_at", `${tomorrow}T00:00:00Z`).

The raise from 110 to 150 correctly protects legacy rows without changing new-row behavior.

Risks to Monitor Post-Deploy

Monitor 1: Anti-repetition context effectiveness

The feature is unverifiable from logs alone (no "changed output" signal). For the first week: spot-check consecutive morning briefs in the Supabase briefs table. Compare summary opening sentences day-over-day. If they still echo structurally (both start with "Markets..." or both start the same verb phrase), the LLM is ignoring the STYLE NOTE, which means the feature needs stronger injection (system prompt rather than user prompt).

Monitor 2: EOD theme_line persistence

EOD briefs store to eod_report brief_type. The anti-repetition query fetches daily_digest. This means EOD's themeLine is never read as anti-repetition context on the next morning run (morning queries daily_digest, not eod_report). Not a bug, but if you wanted day-over-day EOD variety, it needs a separate query branch.

Monitor 3: Concurrent cron overlap (low risk)

Both daily-digest (08:00 UTC) and run-signals recovery (08:10 UTC) read the same "latest daily_digest" for anti-repetition. If they somehow both miss the idempotency check simultaneously (race window is small), the second persist will log a warning and return null rather than creating a duplicate. Existing idempotency guard handles this correctly.
