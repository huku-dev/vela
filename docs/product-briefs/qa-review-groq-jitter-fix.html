BLOCKING

B1. Jitter insertion point is wrong — delays every NVIDIA call, not just Groq fallbacks.
news-fetcher.ts ~line 1564 (inside rankNewsByRelevance, before complete() call)

NVIDIA succeeds 86.4% of the time (369 of 427 non-timeout calls). Adding 0–2s random sleep before complete() degrades every successful NVIDIA call. The Groq 429 problem is a cascade-path problem: it only manifests when NVIDIA fails. Jitter must be applied inside the factory fallback branch, not before the primary attempt.

Specifically: in factory.ts, after NVIDIA failure is confirmed and before groqProvider.complete() is called, insert the jitter. This way, the 86.4% happy path is untouched, and only the ~5–13% fallback path absorbs the delay.

Action: Move jitter to factory.ts Groq fallback branch. Remove from rankNewsByRelevance.
B2. Jitter spread (0–2s) is too narrow to desynchronize calls after a 45s NVIDIA timeout window.
factory.ts cascade logic + registry.ts timeoutMs: 15_000

The registry sets timeoutMs: 15_000. The factory retries up to 3 times before cascading. That means NVIDIA can consume up to 45s before handing off to Groq. Five concurrent assets that start within 0–2s of each other (after jitter) converge on the same 45s timeout boundary: all their Groq calls fire within a ~2s window of each other at the 45s mark. The jitter applied at start is absorbed entirely by the NVIDIA wait time.

Asset A: jitter=0.3s → NVIDIA starts 0.3s → timeout at 45.3s → Groq fires at ~45.3s Asset B: jitter=1.8s → NVIDIA starts 1.8s → timeout at 46.8s → Groq fires at ~46.8s Asset C: jitter=0.9s → NVIDIA starts 0.9s → timeout at 45.9s → Groq fires at ~45.9s Window: Groq calls cluster in a ~1.5s burst at t≈45–47s. Burst NOT solved.

To actually spread Groq calls, jitter must be applied at the fallback handoff point with a range wide enough relative to the burst window (suggest 0–8s, covering the ~8s spread of a 5-slot concurrency pool).

Action: Increase jitter range to 0–8s AND apply it at factory.ts fallback branch, not at NVIDIA start time.

WORTH A LOOK

W1. Root cause verification is incomplete — !hasAssetMatch gate is unquantified.
signal-persistence.ts ~line 905

rankNewsByRelevance fires only when heuristic scoring finds zero direct asset-name matches. The developer's burst hypothesis requires 3–5 assets to simultaneously lack keyword matches. This is plausible for equity batch assets (INTC, DELL, BB, SPCX — all have generic news coverage) but not guaranteed. Before shipping any fix, query llm_call_log for the exact timestamp distribution of news_rank Groq calls to confirm clustering. If calls are spread across minutes, the burst hypothesis is wrong and jitter is solving a phantom problem.

Action: Run SELECT called_at, status FROM llm_call_log WHERE task_name='news_rank' AND provider='groq' AND status='fail' ORDER BY called_at before implementing any fix. Confirm calls cluster within <5s windows.
W2. RPM vs. TPD ambiguity — fix may not address the right 429 type.
openai-compatible.ts 429 error handling

The 41ms failure latency is highly consistent with an RPM (requests-per-minute) burst rejection: the API rejects the connection almost immediately without processing tokens. Daily quota (TPD) exhaustion typically returns after token accounting. However, the openai-compatible.ts adapter reportedly distinguishes error.type === "tokens" from other 429s. If any of the 22 Groq failures are TPD-related, jitter provides zero relief — the fix would pass but the error would recur at a different time of day. Verify by checking the raw error type from Sentry or the adapter logs.

Action: Filter Sentry errors by tags.flow=news-ranking and inspect error.type field. If any are "tokens", the fix is incomplete and a TPD guard (skip LLM ranking entirely when quota is near) is also required.
W3. Better fix exists: Groq retry-after header or exponential backoff instead of pre-emptive jitter.
factory.ts Groq fallback branch

Groq returns a Retry-After header (or x-ratelimit-reset-requests) on 429s. Rather than pre-emptive random jitter that wastes time on non-congested runs, the factory could read this header and wait the prescribed cooldown before the next Groq call. This is zero-overhead on the happy path, adapts to actual server state, and handles burst and TPD cases correctly. Worth evaluating before committing to jitter.

Action: Check Groq 429 response headers in the openai-compatible.ts adapter. If present, implement header-driven backoff as the primary strategy and use jitter only as a secondary measure.

SAFE / CONFIRMED CORRECT

S1. Cascade trigger analysis is correct.

The claim that cascadeOnSchemaFail is a no-op for news_rank (no jsonSchema defined) is accurate. The only cascade triggers are NVIDIA timeout and HTTP failure. The log data (32 NVIDIA timeouts, 26 NVIDIA fails totalling 58 NVIDIA failures vs. 22 Groq fails) is consistent with a subset of NVIDIA failures cascading to Groq — not all cascade successfully (some may hit a different error path or the asset skipped rankNewsByRelevance). The cascade logic is diagnosed correctly.

S2. Deno setTimeout is safe in this context.

Deno edge functions support setTimeout natively. The existing sleep() helper at run-signals/index.ts:999 uses the identical pattern (new Promise(resolve => setTimeout(resolve, ms))) and is already in production. No AbortController is threaded into rankNewsByRelevance, so the jitter promise has no cancellation interaction. The runtime concern is not a blocker.

Bottom line: The root cause is plausible but unconfirmed. The proposed fix (Option B) has the right instinct but the wrong insertion point and too-narrow a jitter range — it will not solve the burst and will degrade the 86% happy-path latency unnecessarily. Address B1 and B2 before implementing. Verify W1 (timestamp clustering) and W2 (RPM vs TPD) before assuming jitter is sufficient. Consider W3 (Retry-After header) as a cleaner alternative.
