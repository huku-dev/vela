Worth a Look (Amber)

1. Sentry tag reads "false" when deregistration occurred but DB write failed
trade-executor.ts:1204 & wallet-provisioner.ts:490–494
Symptom: maybeHandleDeregistration returns false in two distinct situations: (a) the error string is not a deregistration error, and (b) the error IS a deregistration error but the DB UPDATE failed. Both cases result in isAgentDeregistered = false, so the Sentry tag deregistration: "false" and the plain admin notification are emitted even though the agent is deregistered. Ops sees no "auto-recovery triggered" message, does not know re-registration is pending, and may start manual intervention unnecessarily.
Root cause: The helper conflates "no match" and "match but DB failure" into a single false return. The caller has no way to distinguish the two.
Fix: Return a three-valued result ("no_match" | "flagged" | "db_error"), or return an object { matched: boolean; flagged: boolean }. In trade-executor, use matched for the Sentry tag and admin-notification branch, and flagged for the "auto-recovery triggered" suffix. This is a small refactor, not a rewrite — the helper body changes by 2 lines and both callers get more precise information. Without it, a Supabase hiccup silently swallows a deregistration event from all monitoring.
2. Deregistration on position-close and trim paths emits no Sentry event and no admin alert
position-monitor/index.ts:804, 1753, 1909, 2202, 2364
Symptom: All 5 position-monitor callsites discard the return value of maybeHandleDeregistration. If a stop-loss or trailing-stop order fails because the agent is deregistered, the flag is written to the DB (recovery will start), but no Sentry event and no admin Telegram message are sent. The only signal ops gets is console.error, which only surfaces in Supabase function logs — not in Sentry or Telegram. Meanwhile, a position that should be closing is sitting open on exchange.
Root cause: The return value is intentionally ignored on position-monitor callsites. The trade-executor callsite was designed with Sentry tagging; the position-monitor callsites were not given equivalent observability.
Fix: Capture the return value at each position-monitor callsite. When true, call captureMessage with a deregistration: true tag and, optionally, fire the same admin Telegram fragment used in trade-executor ("agent deregistered — auto-recovery triggered"). The message is low-noise (fires once per deregistration event, not per tick) and prevents a silent open position.

Nits (Gray)

3. All tests are source-scan tests — zero behavioral coverage
wallet-provisioner.test.ts:11–65, trade-executor.test.ts:1612–1650
Consider: All 4 new tests (and the 2 executor tests) read the source file as a string and includes() for substring presence. They pass even if the function body is dead code, if the regex is syntactically correct but semantically wrong, or if the boolean returns are inverted. Add at least one behavioral test that constructs a mock Supabase client (a simple object with an from() stub that returns a canned error or success), calls maybeHandleDeregistration directly, and asserts the return value for each of the 3 branches: no match, match+DB success, match+DB failure. This is ~25 lines and would have caught the class of bug described in finding #1 above.
4. Perp ladder failure path does not log the deregistration detection result
position-monitor/index.ts:2364–2367
Consider: The profit-ladder trim failure path calls maybeHandleDeregistration then logs the raw error via console.error. If deregistration is detected, the log says "Profit ladder trim failed ... User or API wallet X does not exist" but does not say "agent flagged for recovery". The stop-loss path at 1753 is the same. Adding a simple if (wasDeregistered) console.log("...flagged for recovery") makes log triage faster without touching any other logic.

Risks to Monitor Post-Deploy

Double-UPDATE race is idempotent but not atomic
wallet-provisioner.ts:485–488
Two concurrent position-monitor ticks that both fail with a deregistration error will both attempt UPDATE user_wallets SET agent_registered=false WHERE id=X. The second write is a no-op (field already false), so correctness is preserved. However, both writes also trigger the updated_at timestamp change (check whether the UPDATE sets updated_at — if the table has an updated_at trigger, it fires twice). This is benign but worth confirming: no downstream logic should interpret a recent updated_at on user_wallets as "wallet was just modified for a different reason."

Safe (Green)

  • Three-branch boolean logic is correct. No match returns false immediately. Match + DB success returns true. Match + DB failure returns false with a console.error. The only ambiguity is in caller interpretation (covered by finding #1).
  • closeResult.error ?? "" is safe. ExecutionResult.error is typed as string | undefined. The nullish coalesce produces an empty string on the success path (where error is undefined), and the regex does not match an empty string. No false positives.
  • wallet is never null at any callsite. In trade-executor, the null check is at line 329 and throws before reaching 1204. In position-monitor, the if (!wallet) guard at line 329 skips the user entirely, so the helper is only called inside processUserPositions which requires a non-null wallet parameter.
  • Recovery loop is sound. deposit-monitor fetches wallets fresh from DB on every tick (select("*") at line 99), so agent_registered=false written by the helper is picked up on the very next deposit-monitor run (~10 min). The !wallet.agent_registered && newBalance > 0 guard at line 652 correctly gates re-registration on having a positive balance, which prevents re-registration attempts for empty wallets.
  • scanner-30m and trade-webhook are covered transitively. Both call executeTradeProposal from trade-executor, which contains the deregistration handler. Neither file calls placeOrder directly.
  • Double-UPDATE is idempotent. Writing agent_registered=false twice is a no-op for the boolean field; recovery behavior is unchanged.
  • Phantom-close timing is unaffected. On the spot phantom-close path, maybeHandleDeregistration is called before the phantom close update, but the phantom close writes only to the positions table (no HL calls), so the async DB write order is inconsequential.
  • Convention is consistent. Helper follows the same pattern as other exported helpers in wallet-provisioner.ts: async, explicit error destructuring, console.error on failure, returns a typed value.
