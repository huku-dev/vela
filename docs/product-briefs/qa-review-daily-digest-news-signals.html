Blocking

Fix white-space:nowrap on signal-relative subtext — mobile layout breaks
_shared/notify.ts, line 1890 (renderLiveNowModule, subtext div)
The subtext cell is styled white-space:nowrap. In a three-column email table with fixed widths (icon=32px, badge=52px, 16px side padding each side), the available text column width at a 375px mobile viewport is roughly 375 - 32 - 52 - 48 = 243px. That sounds fine until the signal is old or the move is large. "+100.0% since signal · 30d ago" is 34 characters; at Inter 12px ~6.6px per char that's ~224px, which fits. But larger decimals are possible: "+1024.5% since signal · 127d ago" (a >1000% move on a very old signal, e.g. BTC fired Jan 2025) renders ~36 chars = ~238px. More importantly, the badge column is fixed at width="52" but align="left" — if the badge cell overflows, it pushes into the text column, reducing the available width below 243px. The cell itself has no overflow:hidden guard, so the text extends past the table edge on narrow clients (Outlook mobile, older Samsung Mail). Action: remove white-space:nowrap from the subtext div, or add overflow:hidden;text-overflow:ellipsis;max-width:180px;. The middle dot separator is Unicode U+00B7 so it's a natural wrap point if nowrap is dropped — wrapping there is fine visually.

Worth a Look

URL match silently degrades when news_cache url column is null
_shared/digest-context.ts, lines 332–334 (fetchRecentNews sort comparator)
The sort reads sourcedNewsUrls?.has(a.url as string). If a cached row has url = null, the cast produces has(null as string). JavaScript Set.has(null) returns false (no throw), so the row just loses priority — safe but silent. However, the upsert in cacheNewsToSupabase (news-fetcher.ts:865) uses onConflict: "title,source", not url. Two RSS feeds can publish the same story with different URLs; they will be stored as separate rows with distinct urls. Meanwhile, sourcedNewsUrls is populated from the URL the ranker saw at brief-generation time. If caching then deduped a row by title+source and kept a different url variant (or the RSS feed returned a tracker-appended url vs a canonical), the Set lookup will miss even though the story is semantically the same item. This means "brief-sourced items first" degrades to "sometimes first" when urls diverge. Not a correctness bug, but the feature reliability is lower than the code suggests. Log a warning when sourcedNewsUrls is non-empty but zero rows matched after the sort — that would surface divergence in prod logs.
fetch24hPriceWindow uses the most-recent indicator_snapshot as "current price" — could be up to 4h stale
_shared/digest-context.ts, lines 198–213 (fetch24hPriceWindow)
Snapshots write every 4h via run-signals. The digest cron runs in the morning; if the last snapshot was at 04:00 UTC and the digest runs at 07:00 UTC, current reflects a price 3 hours old. The label in the rendered email is "+X.X% since signal", not "current price", so this is semantically acceptable — it's the signal-relative move as of the last known price, not live. The risk is that a sharp overnight move (a 20% BTC crash at 06:30 UTC) would show a stale +3% when the signal is actually deeply underwater. Users who understand the subtext as "how the call is doing right now" could be misled. Consider a brief inline note like "as of last snapshot" or ensure the snapshot cadence is tighter than 4h for the digest window. This is not a code bug; it is a product-accuracy concern worth acknowledging in copy or tightening the snapshot schedule.
renderClosesSubtext: "All 1 went against us" reads as ungrammatical for single-close loss edge case
_shared/notify.ts, line 1932 (renderClosesSubtext, winCount===0 branch)
When total === 1 and winCount === 0, the code falls into the winCount === 0 branch and evaluates countWord = "All 1". The output string is: "Here's what Vela closed today. All 1 went against us." The phrase "All 1" is grammatically awkward. The total === 2 case maps to "Both" and total === 3 maps to "All three" — but total === 1 has no special case. Add a total === 1 ? "It" : ... (or "This one") branch before the fallback, yielding "It went against us." Same issue exists in the reverse for a single win: if all closes are wins the function returns the generic positive string without naming the winner, but the minority-losses path names them by asset name — the single-win-plus-loss path at line 1942 calls wins.map(c => c.assetName).join(" and "), which for 1 winner produces a clean single name. So the minority path is fine; only the all-loss single-close path has the grammar bug.

Nit

VELA_GREEN_DEEP (#047857) on VELA_CREAM (#FFFBF5) is marginal for WCAG AA at 12px
_shared/notify.ts, line 1879 (subtextColor assignment)
Contrast ratio for #047857 on #FFFBF5 is approximately 5.2:1. WCAG AA requires 4.5:1 for normal text, so this technically passes. However, at exactly 12px (the subtext font-size at line 1890), this is at the lower edge of readability on poorly calibrated displays or in dark email clients that adjust gamma. The brand-system memo flags the inverse case (white on Signal Green = 1.65:1 fail). This direction is fine, but worth noting in case the text shrinks in a future iteration — dropping to 11px would push into AA-fail territory. No action required now.
sourcedNewsUrls typed as string[] in fetchDigestContext but Set internally; type comment mismatch
_shared/digest-context.ts, lines 540–543 (fetchDigestContext signature)
fetchDigestContext accepts sourcedNewsUrls?: string[] and converts to a Set internally. That is correct. But the JSDoc comment on fetchRecentNews (line 302) takes a Set<string> directly. Public API shape is string[] at the entry point, Set at the inner function — a minor inconsistency a future caller could find confusing. Consider a note in the fetchDigestContext JSDoc that the array is deduplicated into a Set internally.

Safe

SELL inversion formula is correct
_shared/digest-context.ts, lines 159–163
Verified: price 100 → 90 on a SELL: ((90-100)/100)*100*(-1) = +10. Price 100 → 110 on a SELL: ((110-100)/100)*100*(-1) = -10. Green means the call is paying off for both directions. Correct.
renderClosesSubtext does not divide by zero
_shared/notify.ts, lines 1921–1943
renderRecentClosesModule returns early on closes.length === 0 before calling renderClosesSubtext. The subtext function itself uses total only in the "All ${total}" string interpolation and the lossCount > winCount comparison — no division. Zero-close path is unreachable from the call site.

sourcedNewsUrls plumbing through both daily-digest paths is complete
daily-digest/index.ts:192, run-signals/index.ts:713
Both the primary cron path and the recovery-digest path (generateDigestInline inside run-signals) populate sourcedNewsUrls via rankedItems.map(item => item.url).filter(Boolean) and pass the array to notifyDailyDigest. The notifyDailyDigest signature accepts it at position 5 and threads it to fetchDigestContext. Plumbing is complete and consistent between both callers.
Set.has(null) returns false — no throw on null url rows
_shared/digest-context.ts, line 333
JavaScript Set.has() uses SameValueZero comparison. Set.has(null) returns false unless null was explicitly added to the Set. The Set is built from rankedItems.map(item => item.url).filter(Boolean), which strips falsy values, so null is never in the Set. The cast to string in the comparator is a TypeScript lie but has no runtime consequence — the JS call is safe.

Risks to Monitor Post-Deploy

URL match hit rate. Add a log line counting how many of the 30 fetched news_cache rows matched a URL in sourcedNewsUrls. If the hit rate is consistently 0, it means the RSS url and cached url are diverging (tracker params, canonical redirects). Without this telemetry you won't know if the cohesion feature is working.
Extreme signalRelativePct values. A signal fired 60+ days ago on a low-cap asset could show "+400% since signal". This is technically accurate but reads like a boast rather than current information. Consider a max-age cap (e.g., suppress after 30d) or a visual cap on the displayed percentage. No code change needed immediately; watch first production sends for anything that looks misleading.
indicator_snapshot recency at digest send time. If the 04:00 UTC run-signals batch is delayed or skipped, the 07:00 digest will show prices that are 7–11h stale. The signal-relative subtext will still display (no freshness guard) but could misrepresent the current trade status. Consider adding a snapshot-age check before displaying the subtext, or at minimum logging snapshot age during fetchLiveSignals.
