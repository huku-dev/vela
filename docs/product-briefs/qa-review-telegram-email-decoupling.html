Phase 1 QA: Telegram/Email Decoupling in notify.ts

Reviewed: notifyBatchProposals + notifyTradeProposal — 2026-05-29

0 Blocking 1 Worth a look 1 Nit Safe to deploy

Blocking

None. Core decoupling is correct.

Worth a look

notify.ts ~2477 — notifyBatchProposals email try/catch

Silent email skip in development, zero signal in production-like envs. When APP_BASE_URL is localhost and ENVIRONMENT === "development", the inner if falls through without throwing. The email section continues to formatBatchProposalEmail with a broken base URL (localhost links). In practice this only fires locally so no real user impact, but the intent reads as "skip email silently" and the code reads as "continue with a bad URL". A return or explicit console.warn + return inside the if (env !== "development") block would make the localhost short-circuit explicit rather than an implicit fall-through.

No test covers this path today. If someone runs the function with ENVIRONMENT=development on a real staging DB, batch emails go out with localhost hrefs.

Nits

notify.ts ~2501 — email catch log message

The catch message says "Telegram still sent" unconditionally: "Batch email failed ... — Telegram still sent". If getUserTelegramChatId also threw (caught in the outer try at ~2470) and pushed zero TG tasks, this message is misleading: Telegram was NOT sent, yet the log claims it was. Low operational impact, but confusing during incident triage. Consider: "Telegram attempted above (check logs); email failed:".

Safe

Decoupling invariant holds. tasks is declared at line 2448 (before both try/catch blocks) in notifyBatchProposals and at line 2819 in notifyTradeProposal. Both Telegram and email try/catch blocks share the same tasks array; neither re-throws. Promise.allSettled(tasks) is always reached.

getUserTelegramChatId throw is caught. The outer try/catch at lines 2452–2472 wraps the entire Telegram block including the getUserTelegramChatId await. A thrown error logs and falls through to the email section — email still attempts.

Per-proposal inner try/catch is correct. Each proposal's formatProposalTelegram call in notifyBatchProposals is wrapped in its own try/catch (lines 2456–2467). A format failure for one proposal logs and skips, leaving remaining proposals' TG tasks intact.

No Telegram failure can block email. Both Telegram paths (user chatId and fallback admin) only push to tasks; they never await before the email section begins. The email try/catch is structurally independent.

chatId = null is handled. In both functions, a null chatId from getUserTelegramChatId results in zero TG tasks pushed; execution falls through to the email section normally.

n=1 and n>MAX delegation unchanged. The proposals.length === 1 path at line 2424 delegates to notifyTradeProposal before the refactored code is reached. The n > MAX_BATCH_SIZE loop at line 2438 similarly returns before tasks is initialized. Neither path is touched.

notification_log insert still fires after allSettled. In both functions, the audit insert is positioned after await Promise.allSettled(tasks) and is itself wrapped in try/catch — correctly non-fatal. Email/Telegram failure does not skip the log row.

Existing source-grep tests still pass. All 43 tests check string literals and structural patterns that are unchanged by this refactor: IS_STAGING gate presence, trade_proposal_batch channel string, batch_count: proposals.length, notification_log write failed (non-fatal), proposals.length === 1 delegation pattern, proposals.length > MAX_BATCH_SIZE fallback, and Promise.allSettled usage — all present in the new code at the same grep-matchable positions.

Admin email fallback preserved. If profile?.email is null, both functions fall back to sendEmail (admin). No regression.
