Blocking Findings

B1. Double spot-trim race: two overlapping position-monitor cycles can both fire the same ladder level

position-monitor/index.ts ~line 2216. Severity: blocking.

Symptom. Two cron invocations overlap (cron fires every 4 min; function can run up to ~2 min). Both read pos.ladder_levels_trimmed from DB before either trim commits. Both see, say, [] at level 0. Both pass the trimmedLevels.includes(i) check at line 2219. Both call adapter.placeSpotOrder for level 0, selling twice the intended fraction of the spot position.

Root cause. The perp path has the same theoretical race but HL rejects the second reduce-only as "would increase position" because there is no margin position for spot. For spot, both sell orders are independent market sells with no reduce-only guard. HL executes both. The position DB row gets ladder_levels_trimmed: [0] written by whichever cycle commits second, but two full trim orders have already filled.

Severity escalation vs perp. For perp the race was always theoretical because reduceOnly: true provided an exchange-side idempotency gate. Spot removes that gate. This is a new correctness gap introduced by this PR.

Fix. Add an optimistic concurrency guard: include .eq("ladder_levels_trimmed", pos.ladder_levels_trimmed as unknown as Json) (or a Postgres array-length check via RPC) in the update at line 2286. If a concurrent write wins, the update returns 0 rows; skip the rest of the trim success path. Alternatively, lock the row with a SELECT FOR UPDATE before the trim order. The SL close path at line 1619 does not have this problem because closing the DB row to status: "closed" is itself idempotent (second write is a no-op on a closed row), but the ladder trim updates an array field on a still-open row, which is not idempotent.

Worth a Look

W1. fill_size === 0 is falsy — transfer guard passes but transfers $0

position-monitor/index.ts line 2265. Severity: worth a look.

Symptom. When placeSpotOrder returns { success: true, fill_price: 18.0, fill_size: 0 } (partial fill edge case: order accepted but nothing filled yet), the guard if (isSpot && trimResult.fill_price && trimResult.fill_size) at line 2265 evaluates 0 as falsy and skips the transfer correctly. So far so good. However, _parseOrderResponse at adapter line 969 sets fill_size = parseFloat(filled.totalSz); if HL returns totalSz: "0" on a fully-accepted market order (hasn't matched yet), fill_size is 0. In that case the trim DB update still runs (line 2286), recording the trim as done, but no USDC is transferred to perps. The spot sub-account now holds proceeds that the user cannot access for perp trading. Not funds-lost (they remain on-chain), but the perp account balance will be stale until a manual sweep or next phantom reconciliation.

Fix. Log a warning when trimResult.success && !(trimResult.fill_price && trimResult.fill_size) so this is observable. Do not record the trim level as done until fill is confirmed.

W2. exchangeSize used for trim fraction is stale after first trim fills

position-monitor/index.ts lines 2249–2250. Severity: worth a look.

Symptom. For spot positions exchangeSize is assigned once per cycle at line 886 from spotBalanceMap.get(hlCoin), which was populated at the top of the function at line 505 before the per-position loop. If two ladder levels qualify in the same tick (both thresholds crossed simultaneously), the break at line 2330 exits after the first trim, so only one level fires per cycle. Good. But on the next cycle, spotBalanceMap is re-fetched from HL (line 505) and exchangeSize reflects the post-first-trim balance. The fraction exchangeSize * PROFIT_LADDER_FRACTIONS[i] for level 1 will therefore be 10% of the reduced balance, not 10% of the original balance. This under-shoots: the second trim takes 10% of what is left, not 10% of what was opened. The USD-based trimSizeUsd = originalSize * PROFIT_LADDER_FRACTIONS[i] is correct but the exactSizeInAsset override will be wrong by the first-trim fraction.

Assessment. The worst outcome is a slightly smaller-than-intended trim (excess tokens remain on spot). Not a loss of funds. However, it means the spot-ladder behaviour diverges from the perp-ladder behaviour where all fractions are applied to originalSize. Flag for explicit sign-off rather than treating as a silent known-good.

Fix. Either: (a) remove exactSizeInAsset from the spot trim and let the adapter compute from sizeUsd alone (uses mark price, acceptable precision for a trim); or (b) compute exactSizeInAsset from pos.size * PROFIT_LADDER_FRACTIONS[i] (tracked token count) rather than from the live exchange balance.

W3. FEATURE-ADV regex test for skipTpLadder would pass if openedAsSpot is in an adjacent comment

trade-executor.test.ts line 2579–2583. Severity: worth a look.

Symptom. The test asserts /skipTpLadder\s*=\s*.*openedAsSpot/ against the raw source. The pattern matches any line where openedAsSpot appears anywhere after skipTpLadder =, including a comment. For example, the existing comment at line 1728 ("Spot positions skip native TP... spot profit ladder is handled by position-monitor") followed by a code change that removes openedAsSpot from the condition but leaves a nearby comment mentioning it would produce a false pass. The regex is also single-line dot (.), so a multi-line refactor that breaks the condition across two lines would defeat it.

Fix. Tighten to: /skipTpLadder\s*=\s*[^;]*?\|\|\s*openedAsSpot/ (requires the || operator before openedAsSpot on the same expression), which cannot be satisfied by a comment. Also add a complementary negative assertion that the native TP block is not entered when openedAsSpot is true.

Nits

N1. No FEATURE-ADV test covering the spot trim path in position-monitor

position-monitor/position-monitor.test.ts. Severity: nit.

The existing SPOT-ADV tests (lines 839–987) confirm the SL/trailing-stop spot paths. There is no test asserting that the poll-based profit-ladder path branches on isSpot and calls adapter.placeSpotOrder (not adapter.placeOrder) for the trim. A source-regex test similar to the SPOT-ADV pattern at line 839 would be: assertMatch(src, /isSpot[\s\S]{0,300}?adapter\.placeSpotOrder[\s\S]{0,300}?profit.*ladder|ladder[\s\S]{0,300}?isSpot[\s\S]{0,300}?placeSpotOrder/i). Low friction to add; high coverage-to-effort ratio for a financial execution path.

N2. No captureException on spot trim failure (perp failure path already captures)

position-monitor/index.ts line 2325. Severity: nit.

The else branch at line 2324 calls console.error. The perp path lands the same code, so this is not a regression. But given that this PR is the first time spot runs this code path in production, adding captureException to the else branch for the first few deploys would surface unexpected HL spot rejections without requiring log-scraping. The existing native-TP failure path at trade-executor line 1787 does call captureException for comparison. Low-priority but a useful observability nudge for a new path.

Risks to Monitor Post-Deploy

Spot trim double-execution (B1): query notification_log for profit_ladder_trim with the same entity_id and adjacent timestamps (within 5 min). A duplicate indicates the race fired.
Stale exchangeSize in fractions (W2): compare trim_fraction * original_size_usd in trim_history against expected 10% per level. Consistent under-trimming at level 1/2 indicates the stale-balance path.
Zero fill_size transfers (W1): monitor spot USDC balance on HL. If spot sub-account accumulates USDC after ladder trims without a matching perp-account increase, the transfer guard skipped on a zero-fill.
Sentry error rate on spot position opens: should drop to 0 for "trigger orders not enabled on spot" after this deploy. Any remaining occurrences indicate openedAsSpot was false at the check site.
