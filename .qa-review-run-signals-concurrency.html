Blocking

B-1: Stale comment says "cap at 3" — code sets 4; plus shared-index race in runWithConcurrency
run-signals/index.ts line ~258

The comment block above the constant reads: "Cap at 3 to balance throughput against worker memory: 5 concurrent tasks caused OOM (546)… concurrency=3 (4 waves / 5 waves)…" but the code sets ASSET_CONCURRENCY = 4. The comment also says "OOM risk at 4 is lower than 5 given the concurrent feed" — internally contradictory in a single paragraph.

This is a maintenance hazard, not just a typo. The next engineer who reads the comment ("cap at 3") to understand why 4 is set will be confused about intent, and may reduce it back to 3 "to match the comment." More critically: runWithConcurrency (line 1012) uses a shared mutable index counter across workers. In Deno, each async function worker() increments index++ without any atomic guarantee. JavaScript is single-threaded so the increment itself won't tear — but two workers that both observe index < tasks.length, both read tasks[index], and both execute index++ in the same microtask checkpoint could skip a task or double-dispatch one. This is a real race when tasks resolve synchronously (e.g. if task() short-circuits early due to cached data).

Fix: (1) Update the comment to say "Cap at 4" and remove the contradictory "cap at 3" sentence. (2) In runWithConcurrency, snapshot the current index before incrementing: const i = index++; await tasks[i](); — this is the standard worker-pool pattern that prevents the double-dispatch edge case.

Worth a look

W-1: generateTradeProposals throw after partial DB INSERT silently drops notifications with no Sentry capture
run-signals/index.ts lines ~432–438

generateTradeProposals can throw in three places that may occur after proposals have already been INSERTed for some users in the per-user loop: (a) fetchEligibleUsers throws before iteration; (b) a DB error mid-loop after earlier users' proposals were written; (c) WALLET_ENVIRONMENT not set (proposal-generator.ts line 895 throws before the user loop). For case (a) and (c), no proposals exist and the fallback is safe. For case (b), proposals exist in trade_proposals with status=pending but are absent from pendingEmailsByUser — the author names the 2h proposal-reminder as the safety net, which is correct. However:

  • The catch block logs via console.error only. There is no captureException call, unlike the auto-execute block at line 595 which does call Sentry. This means the throw is invisible in Sentry dashboards and won't page anyone.
  • The log includes signal_id but not assetSymbol in the prefix — the asset symbol IS in the outer prefix string so it's technically present, but it would help readability to be explicit.

Fix: Add captureException(proposalErr, { tags: { flow: "proposal-generation", asset: asset.symbol } }) inside the catch block, matching the pattern used at line 595.

W-2: Orphan warning fires on every batch with signal changes when all trading users are full_auto
run-signals/index.ts lines ~613–619

The warning condition is: anySignalChanged && pendingEmailsByUser.size === 0 && !IS_STAGING. pendingEmailsByUser is only populated for deferEmail:true proposals, which are emitted only for semi-auto users (full_auto users get autoApproved:true, not a deferred email — confirmed in proposal-generator.ts line 820). Therefore, any batch where all trading users are full_auto will fire this warning on every signal change, every 4 hours. This is a false-positive that will train the operator to ignore the warning, defeating its purpose.

Additionally: when generateTradeProposals is skipped because the signal is non-green/non-red (grey), signalChanged is still true but the generation block is guarded by signalResult.signal.color !== 'grey'. The warning would fire if a grey signal change occurred and semi-auto users exist — but grey proposals are never generated. This is a second false-positive path (minor, grey changes are uncommon).

Fix: The warning needs to distinguish "no semi-auto users at all" from "generateTradeProposals threw." Count semi-auto proposals created across the results (or track a separate proposalAttempted boolean) before deciding whether empty pendingEmailsByUser is unexpected.

W-3: 30s news timeout + 3 waves at concurrency=4 leaves <30s for digest — tight under worst-case conditions
run-signals/index.ts lines ~187–206, ~265

Timing budget for batch-1 (11 assets, concurrency=4, 3 waves):

  • News pre-fetch cap: 30s
  • 3 waves of 4 assets. Each wave's duration = slowest asset in that wave. A cold HL fetch + LLM brief call observed at ~20–25s worst case per asset. Three waves: ~75s in the optimistic case, up to ~90s if one wave has a slow LLM call.
  • Auto-execute queue: typically <5s, but can be longer if multiple proposals need execution.
  • Batch proposal dispatch: typically <2s.
  • Remaining headroom for digest: 150s - 30s news - 90s waves - 7s misc = ~23s. Digest is wrapped in its own try/catch and runs only when needed, so a 546 here is non-fatal.

This is not blocking because the digest 546 is explicitly handled and the dispatch already runs before it. However, if a news pre-fetch takes exactly 30s AND one wave takes 30s+ (plausible on a congested HL API), the digest may never start. This is the same risk that existed at concurrency=5 but slightly mitigated by the concurrency reduction. Worth monitoring the 08:10 UTC run on staging for wall-clock time.

Note: The comment on line 186 says "30s saves 15s vs the original cap" — this is correct. No fix required, but staging timing data would increase confidence before prod deploy.

Nits

N-1: IS_STAGING defined twice in run-signals scope
run-signals/index.ts line ~650 vs top-of-file import from notify.ts

IS_STAGING is imported from notify.ts (used throughout the file) but a second local isStaging variable is created at line 649: const isStaging = (typeof Deno !== "undefined" ? Deno.env.get("ENVIRONMENT") : undefined) === "staging". Line 651 then uses isStaging (local) for the digest gate while the rest of the file uses IS_STAGING (imported). Both read the same env var, so behavior is identical — but two variables for the same thing is a maintenance landmine. Remove the local redeclaration and use IS_STAGING on line 651.

N-2: Log message at line 258 says concurrency=3 wave counts are the comparison baseline
run-signals/index.ts line ~259

The comment says "materially faster than concurrency=3 (4 waves / 5 waves)" — this is the wave count for concurrency=3 across batch-1 (11 assets) and batch-2 (15 assets). These numbers are correct. But given the comment already references concurrency=4 as the chosen value, the "cap at 3" reference earlier in the same paragraph (see B-1) is the source of confusion. Fixing B-1 cleans this up.

Risks to monitor post-deploy

  • 546 rate on batch-1 08:10 UTC run. Concurrency=4 is intended to reduce OOM. If 546s continue on staging, further reduction to concurrency=3 should be evaluated. Check edge function invocation logs for HTTP 546 in the first 3 days.
  • Proposal-reminder safety net effectiveness. If W-1 triggers (generateTradeProposals throws after partial INSERT), proposals will be orphaned until the 2h reminder fires. Confirm the reminder cron is active in prod and surfaces those proposals correctly.
  • Orphan warning noise level. Once W-2 is evaluated, check logs to see how often the warning fires per batch. If it fires on multiple assets per run, it means full_auto users dominate and the warning is useless without the fix.
  • News timeout hit rate. A 30s timeout on fetchAllNews is tighter than the previous 45s. Monitor for the "News pre-fetch failed (non-fatal)" log line in the first week. If it fires >20% of runs, brief quality will degrade and the timeout should be raised back.

Safe

  • Change 3 fallback shape. The zero-value { pendingEmails: [], autoApproved: 0, proposalsCreated: 0, skipped: 0, errors: [] } is structurally correct. The loop over proposalResult.pendingEmails (lines 446–454) safely handles an empty array. No null-deref risk.
  • notifyBatchProposals empty-array guard. notify.ts line 2429: if (proposals.length === 0) return; — the function is safe to call with an empty array even though the caller at line 629 already guards with pendingEmailsByUser.size > 0.
  • results.some(r => r.changed) when results is empty. Array.prototype.some on an empty array returns false per spec. The warning correctly stays silent when zero assets were processed.
  • pendingEmailsByUser concurrent writes. JavaScript is single-threaded. The worker tasks run on a single event loop; Map mutations between await points are not concurrent in the memory-race sense. No lost writes possible.
  • IS_STAGING gate on proposal generation. The try/catch block at line 415 is inside the existing && !IS_STAGING guard. The fallback result is therefore only reached on prod, which is the correct behavior. Staging will never enter the try/catch path.
