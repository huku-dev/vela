Worth a look (2)

AMBER-1: Close and trim branches now silently compute isBuilderPerp without reading it — correct today, latent trap in future edits
trade-executor.ts line 634 (hoist site) vs. lines 1318–1550 (trim/close branches, confirmed zero reads of isBuilderPerp)
Situation

Before the fix, isBuilderPerp was scoped strictly inside if (!isClose && !isTrim), so it was provably dead on those paths. After the hoist it is computed unconditionally but never read in the trim branch (lines 1318–1370) or close branch (lines 1372–1550). This is harmless at this commit.

Risk

A future author adding builder-perp-specific close logic (e.g. "cancel native SL with explicit size on close") will see the variable already defined and use it without questioning its provenance. If hlCoin is ever null-coalesced to the plain symbol (the fallback at line 373 is asset.hl_symbol ?? asset.symbol), isBuilderPerp evaluates false for a builder-perp asset, silently breaking the new logic.

Recommendation

Add a one-line inline comment at line 634 to bound the intended scope: // isBuilderPerp used in the open-trade SL/TP path below (~lines 1665, 1684); hoisted here so it is available if close/trim branches ever need it too. This costs nothing and prevents misuse in the next edit session.

AMBER-2: No behavioral integration test covers builder-perp open with SL placement; all existing tests are source-scan regex assertions
trade-executor.test.ts lines 2630–2658 (only SL-related FEATURE-ADV tests); zero hits for xyz:, hl_symbol, or executeTradeProposal with builder-perp inputs
Situation

The two FEATURE-ADV tests at lines 2630–2658 scan the source text for string patterns. They do not call executeTradeProposal, do not mock the adapter, and would not have caught the ReferenceError that triggered this fix. The original bug (variable used before declaration) silently threw inside the try/catch at line 1722, causing SL to be skipped on every builder-perp open, with no test failure.

Recommendation

Add a FEATURE-ADV-prefixed behavioral test: mock adapter.placeTriggerOrder, execute a proposal where hl_symbol = "xyz:AAPL" and fill_size is populated, and assert (a) placeTriggerOrder is called and (b) the call includes sizeInAsset === fill_size. This is a follow-up task and does not block deploy. Without it, the next regression in this path will also reach production undetected.
