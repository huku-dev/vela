Worth a Look (2)

W1 — news-enrich cron path bypasses SUMMARY_MAX entirely

news-enrich/index.ts:309–311

Symptom: The PR raises SUMMARY_MAX to 110 in classifyNewsItem() inside news-classifier.ts. But news-enrich/index.ts does its own inline summary extraction (line 309–311) without calling classifyNewsItem — it imports only NEWS_CLASSIFY_SYSTEM_PROMPT, isAssetSentimentsObject, and wrapUntrustedExcerpt. The SUMMARY_MAX guard is absent entirely from this path.

// news-enrich/index.ts — no truncation guard const summary = typeof parsed.summary === "string" && parsed.summary.trim().length > 0 ? parsed.summary.trim() : null; // ↑ Summaries of 111-180 chars go straight to DB from the cron path.

Root cause: The SUMMARY_MAX guard was added to classifyNewsItem() in the shared module, but news-enrich never calls that function. It duplicates the parsing logic inline. This was already a divergence before this PR; the PR does not make it worse, but it does create a false impression that the guard is universal.

Impact: The majority of news classification goes through the cron path (news-enrich processes the backlog hourly). Only inline/live classification (via news-cache.ts, which calls classifyNewsItem) gets the 110-char cap. Summaries from the cron path can still hit the DB at 111–180 chars, where they get caught only by the 150-char render-side backstop in renderNewsModule.

Fix: Refactor news-enrich/index.ts to call classifyNewsItem() from the shared module instead of duplicating the parsing inline, OR copy the SUMMARY_MAX truncation block into the inline parse path at line 311. Either approach ensures consistent storage length across both classification paths.

Risks to Monitor Post-Deploy

R1 — all-caps briefs return unsplit regardless of length

The boundary regex /[a-z0-9][.!?] /g requires a lowercase letter or digit before the punctuation. An all-uppercase brief (unlikely but possible if an LLM generates one) returns as a single paragraph regardless of sentence count. This was the same behavior as the old implementation and is not a regression, but it is worth logging if a future LLM model changes capitalization style. Monitor: scan daily_digest rows for unsplit 400+ char briefs.

R2 — numbered-list briefs produce excess break points

If an LLM generates a brief formatted as "1. Point one. 2. Point two. 3. Point three." the digit before the period matches [a-z0-9][.!?] and produces double breaks per sentence boundary (one for the preceding digit, one for the sentence-terminal lowercase). Result: more split points than expected, potentially 4–5 "paragraphs" from a 3-point list. Low probability given current prompts, but the dash-artifact fix shows LLMs do produce unexpected structural patterns.

Confirmed Safe (7)

  • +2 offset is correct. The boundary regex matches 3 chars (e.g. "g. ") at m.index. m.index + 2 points to the space character. slice(pos, sp) ends one char before the space (at the punctuation); .trim() removes trailing punctuation's space and the leading space of the next slice. Traced on "War fears... morning. U.S.-Iran tensions... lower. Oil jumped. Gold fell.": produces exactly ["War fears are driving markets this morning.", "U.S.-Iran tensions sent the Dow lower.", "Oil jumped. Gold fell."] — "U" is preserved.
  • Final sentence is always captured. The last sentence has no trailing space, so it never registers as a break. The final cleaned.slice(pos).trim() after the split-point loop captures all remaining text including the terminal sentence. Verified.
  • breaks.length <= 2 threshold is correct. N sentences produce N-1 inter-sentence spaces (the final sentence has no trailing space). 3 sentences = 2 breaks → returns unsplit. 4 sentences = 3 breaks → splits. The comment is accurate.
  • No character loss. The new implementation uses slice() throughout with position-based boundaries. Unlike the old .match() approach, no text is ever dropped. Consecutive slices with no gaps are guaranteed because each split point is a character position in the original string.
  • Dash cleanup regex is safe for multiple matches. trimmed.replace(/(\. )-([A-Z])/g, "$1$2") uses a fresh regex literal (not the dashArtifact variable with advanced lastIndex), so all occurrences of the artifact are replaced in a single pass. No user content is logged in the console.warn — only the static string.
  • SUMMARY_MAX = 110 truncation is correct. A 110-char summary passes through unchanged (110 > 110 is false). A 111-char summary is sliced to 110, the last space is found, and if it is beyond position 90 the word-boundary trim applies. The trailing-punctuation replace uses /[.,;:]\s*$/ (where $ is end-of-string, not a literal dollar sign — confirmed by JS regex semantics). Result is clean word-boundary truncation + ellipsis. The render-side backstop in renderNewsModule (SUMMARY_CAP=150) provides a second layer for any rows that escape the 110-char guard.
  • renderLiveNowModule change is correct. All three header cells are now empty (<td></td>). The column structure (widths 32, unconstrained, 52) is preserved. The headerCellStyle still applies the bottom border and uppercase styling — the cells still render with the divider line, they just have no label text. No layout breakage.
