Worth-a-look

breaking-news/index.ts line 1150-1153 vs. line 1130-1137

The admin ping for judgeSkipped is fire-and-forget (no await). The eval-blocked admin ping at line 1130 uses await with .catch() chained — it blocks before returning the Response. The asymmetry is undocumented and could cause the ping to be dropped if the Deno isolate exits before the promise settles (though this is low-risk on the success path since the broadcast itself takes time after). More importantly, the inconsistency makes the intent unclear to future readers: is this intentional (don't slow down the broadcast) or an oversight?

Suggestion: Add a one-line comment: // fire-and-forget — don't delay the broadcast. Optionally add void prefix to make the fire-and-forget intent explicit and silence any lint rules about unhandled promises.

_shared/news-broadcast-eval.ts lines 362-373 (synthetic verdict)

The synthetic JudgeVerdict returned on AbortError has relevance_score: 0 and comprehensibility_score: 0. These are valid numbers per the interface and are never consumed by the caller on the passes: true path (verdictLine is only built inside if (!evalResult.passes)). However, if a future caller reads evalResult.verdict on the passing path (e.g. for calibration logging), they will see scores that look like an irrelevant broadcast that somehow passed. A score of -1 or a sentinel like null would make the "not a real verdict" status more obvious — but JudgeVerdict types relevance_score as number, not number | null, so 0 is the only in-type option without a schema change.

Low priority. Consider adding a comment in the synthetic verdict: // synthetic — not a real LLM verdict; scores are zero because L2 was skipped. No code change required.

The new source-grep test is correct and well-justified (testing a live AbortError requires a real network timeout, which is infeasible in unit tests). One minor point: the test name says "fail-open on judge AbortError (timeout)" but asserts three separate things (err instanceof DOMException predicate, judgeSkipped: true string, and "failing open" string). If the AbortError predicate string is ever reformatted (e.g., the DOMException check is extracted to a helper), the test would fail in a confusing way. Negligible in practice since this codebase doesn't refactor checks into helpers.

No action required. Noted for awareness only.
