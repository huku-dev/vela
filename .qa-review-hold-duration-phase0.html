Phase 0 QA: hold-duration-rule

Adversarial pre-implementation review • position-monitor/index.ts • Reviewer: independent subagent (no author context) • 2026-06-17

Blocking: 3
Worth-a-look: 5
Nit: 2
Safe/confirmed: 4

Verdict: NOT safe to implement as sketched

The architectural placement and hold-time math are sound. However three design gaps must be resolved before writing a single line of production code:

  • The close flow is underspecified. The sketch says "close position" but the full stop-loss flow has 11 steps. Missing any one (TP cancel, spot proceeds transfer, EMA cooldown, postmortem, notification) will leave the system in a broken state or silently skip user comms.
  • Position-type scope is unguarded. Without an explicit position_type === "main" guard, the rule fires on BB2 and BB2_30m positions that have their own expiry mechanism already — creating a double-close race.
  • No counter and no kill switch. The rule will fire silently with no observability and no rollback path. Both are required before deploy per the project's safety standards.

The underlying data pattern is real and the fix direction is correct. Resolve the three items above (all addressable in the design phase), then proceed to implementation.

Blocking

Blocking
B1 — Close flow is underspecified: 7 of 11 steps are missing from the sketch
index.ts:1586–1760 (stop_loss close reference)

The stop-loss close path (lines 1586–1760) does 11 things after checkStopLoss() returns true. The sketch only says "close position." The BB2 expiry path (lines 1764–1919) is the closest architectural analog for a hold-time close and performs all the same steps. Steps that must be present for the hold-duration close and are absent from the sketch:

  • Cancel native SL trigger order (exchange_sl_order_id) before placing the market close, or HL will double-fill. Applies to perps and spot. (line 1600–1608)
  • Cancel native TP trigger orders (exchange_tp_order_ids) before close, or orphaned TP orders will remain on HL. (line 1612–1617). Main positions can have native TPs; BB2 cannot. This step is explicitly needed for main positions.
  • Spot vs perp close branch: spot uses placeSpotOrder, perps use placeOrder(..., reduceOnly: true). Missing the branch causes a wrong order type for spot. (line 1621–1637)
  • Spot proceeds transfer: if isSpot && fill_price && fill_size, call adapter.transferUSDC(proceeds, true) to move proceeds from spot sub-account back to perp sub-account. Skipping this leaves funds stranded. (line 1641–1646)
  • EMA cooldown upsert: stop-loss and untracked-fill paths both upsert ema_cooldowns for position_type === "main" (line 1736–1752). A hold-duration close on a failed main position should also set the EMA cooldown, otherwise the strategy will immediately re-enter the same direction on the next 4H cycle. The existing EMA cooldown comment says "prevent re-entry in same direction after stop-loss" — a 72h hold-duration close is economically equivalent to a delayed stop-loss.
  • Postmortem: recordPostmortem (fire-and-forget) must be called for analytics. All close paths call it. (line 1679–1688)
  • Admin Telegram: all close paths send an sendAdminTelegram notification. The hold-duration rule needs one too or it fires silently. (line 1710–1715)

Steps that DO NOT apply to hold-duration (only to stop-loss): the circuit-breaker stopLosses counter increment and the bb2_cooldowns upsert (BB2 positions are excluded — see B2).

The closest complete reference to clone from is the BB2 expiry flow, lines 1764–1919, which handles all of the above for a time-based close on any position type.

Required action

Before writing code, enumerate explicitly which of the 11 steps apply and draft the full close sequence. Do not write a partial close — use the BB2 expiry block (lines 1764–1919) as a structural template. Confirm that EMA cooldown is set for main positions.

Blocking
B2 — No position_type guard: rule fires on bb2 and bb2_30m positions
index.ts:1764–1919 (BB2 expiry path); proposed insertion ~line 1941

The sketch has no position_type filter. BB2 and BB2_30m positions already have their own hold-time expiry checked at lines 1764–1919. A BB2_30m position that has held 72h will have already been closed by the existing hoursHeld >= BB2_30M_HOLD_HOURS (16h) check. But a BB2 (4H) position has a 48h limit (BB2_HOLD_DAYS=2, line 1102 in tests). If a BB2_4H close fails (network error, deregistration), the position stays open, continue is NOT hit (only hit on success, line 1909), and the loop falls through to line 1921. At 72h, an unguarded hold-duration check would then fire a second close attempt on an already-partially-processed BB2 expiry.

More critically: the production data analysis explicitly states the 0% win rate was for position_type = 'main' positions. BB2 positions were excluded from the qualifying set. The rule was designed for EMA/main positions only.

// Proposed sketch — MISSING guard:
if (holdH >= 72 && peakPnl < 0.5) { ... }

// Required:
if (pos.position_type === "main" && holdH >= 72 && peakPnl < 0.5) { ... }
Required action

Add pos.position_type === "main" as the first condition in the guard. This both matches the intent (the rule is evidence-based on main positions only) and prevents any interaction with the existing BB2 expiry logic.

Blocking
B3 — No observability counter and no kill switch
index.ts:400–412 (result object counters); line 2552–2561 (return)

The processUserPositions return object (line 2552) tracks stop_losses, trailing_stops, ladder_trims, native_tp_fills. There is no hold_duration_closes counter. If the rule fires on a user's position, the operations team has no signal: the counter is not reported, no admin alert exists in the sketch, and logAudit (required anyway per B1) uses a free-text action string that isn't aggregated anywhere automatically.

Additionally, there is no env-var kill switch. Every other non-trivial behavior in position-monitor either has a safety gate (the E2E triple-gate, the failedDexes skip, the spotBalanceFetchFailed skip) or is guarded by a position-type check that doubles as a config parameter (BB2 expiry is off for main positions). A new irreversible close path has no rollback mechanism other than a re-deploy.

Required action

Add a hold_duration_closes: number field to the return type and increment it on each successful close. Add HOLD_DURATION_RULE_ENABLED env var (defaults to false on deploy, flip to true after first staging validation). The Sentry capture / admin Telegram (already required by B1) provide the alerting.

Worth a Look

Worth a Look
W1 — peakPnl (in-memory) vs pos.trailing_stop_peak_pnl_pct (DB): which to use?
index.ts:1929 (peakPnl assignment); line 59 (field type)

At line 1929: const peakPnl = Math.max(pos.trailing_stop_peak_pnl_pct ?? 0, unrealizedPnlPct). This is the current-tick in-memory peak — it is always >= the DB value because it incorporates this tick's price. Using in-memory peakPnl for the 0.5% threshold is strictly correct and strictly safer than the DB value: it gives the position one more tick of benefit-of-the-doubt. If the position hits exactly 0.5% on this tick, peakPnl >= 0.5 and it survives — the rule only fires if the current tick is also below 0.5%.

The risk is the inverse: using DB-only pos.trailing_stop_peak_pnl_pct could fire even if the position is currently at 0.6%, because the DB may lag one tick. The proposed in-memory value is the right choice. Just confirm the author is aware of this distinction and has not accidentally written pos.trailing_stop_peak_pnl_pct < 0.5 instead of peakPnl < 0.5.

Required action

Confirm the condition uses in-memory peakPnl, not pos.trailing_stop_peak_pnl_pct. Add a code comment explaining the choice.

Worth a Look
W2 — BB2 continue path at line 1788 can leave a hold-time-expired BB2 position in the loop past the proposed insertion point
index.ts:1783–1789 (upgrade_close in-flight guard); ~line 1941 (proposed insertion)

The BB2 expiry block (line 1769) has an in-flight upgrade_close guard (line 1783–1789) that fires a continue to skip to the next position. If a BB2 position that is beyond its expiry limit has an in-flight upgrade_close, it skips the BB2 expiry close and falls through to line 1921 — the trailing stop section — and then to the proposed insertion point at ~1941.

With a missing position_type === "main" guard (B2), this would let the hold-duration rule fire on a BB2 position that has a pending upgrade_close, creating a race condition with a proposal in-flight. The B2 fix (adding the position_type guard) completely eliminates this path. This is a reinforcing argument for B2, not a separate blocking issue.

Required action

Resolved entirely by fixing B2 (adding position_type === "main" guard). No separate action needed if B2 is addressed.

Worth a Look
W3 — reasonLabels map in notifyUserPositionClosed does not include "hold_duration_rule"
index.ts:211–217 (reasonLabels); tests line 230 (fallback test)

The notifyUserPositionClosed function uses a reasonLabels map (line 211) to convert internal close reasons to user-readable strings. The current entries are stop_loss, bb2_expiry, trailing_stop, liquidation. The fallback at line 217 is reasonLabels[closeReason] ?? closeReason, so a missing entry shows the raw string "hold_duration_rule" to the user — internal terminology leaking into a user-facing message.

The existing test at line 230 checks that the fallback path works (raw string falls through), but there is no test asserting that hold_duration_rule maps to a plain-English label. The voice rule (knowledgeable-friend tone, no jargon) requires this.

Required action

Add hold_duration_rule: "Position closed — signal did not develop within expected timeframe" (or equivalent plain-English label) to reasonLabels. Add a test asserting this mapping exists. Do not expose the internal reason name to users.

Worth a Look
W4 — Close failure path: if placeOrder fails, no continue fires, loop falls to profit ladder
index.ts:1753–1759 (BB2 expiry failure path); proposed implementation

Every successful close path in position-monitor fires a continue to skip subsequent checks. The BB2 expiry else block (line 1910–1916) only logs the error — it does NOT fire a continue. This means if the BB2 expiry close fails, the position falls through to the trailing stop checks. The same will happen to the hold-duration rule if not explicitly handled.

For the hold-duration rule: if the market close fails (e.g. HL timeout), the rule should log the failure and then continue — skipping the profit ladder — because the position is in a degraded state where we already decided to close it. Falling through to the profit ladder after a failed close attempt would execute partial trims on a position we intended to fully close.

Required action

In the failure path of the hold-duration close, add continue after the error log (do not fall through to profit ladder). Match the behavior of the stop-loss failure path at line 1753–1760.

Worth a Look
W5 — opened_at is non-nullable but DB may return a different fallback for legacy rows
index.ts:62 (opened_at: string); index.ts:1765 (BB2 expiry uses pos.opened_at directly)

opened_at is typed as string (non-nullable, line 62). The BB2 expiry code at line 1765 uses new Date(pos.opened_at).getTime() with no null guard. The author's sketch uses pos.opened_at ?? pos.created_at, which implies concern about null — but the type says it's never null.

Check the DB query that populates positions: if there are legacy rows where opened_at was never backfilled (e.g. positions opened before the column existed), the TypeScript non-null type would be wrong and new Date(undefined) returns NaN, making holdH NaN and the >= 72 check silently false (never fires). This is low risk for new positions but worth confirming via a one-time query before deploy.

Required action

Run SELECT COUNT(*) FROM positions WHERE opened_at IS NULL AND status = 'open' against production before deploy. If any rows exist, either add the ?? pos.created_at fallback and assert it aligns with the BB2 expiry approach, or backfill the nulls in a migration. Either way match the null-handling to what BB2 expiry does on line 1765.

Safe / Confirmed

Placement after line 1941 is correct

The stop-loss check (line 1589) fires a continue at line 1760 when it successfully closes. The BB2 expiry block (line 1764) fires continue at line 1909. Both skip to the next position before reaching line 1941. Any position that reaches the proposed insertion point has passed SL and BB2 expiry checks, which is the correct ordering: hard stops before time-based exits. No other continue between lines 1589 and 1921 would skip a main position on a normal tick.

NULL peak / new position: holdH guard fully protects

A position with trailing_stop_peak_pnl_pct = null evaluates peakPnl = Math.max(0, unrealizedPnlPct). If unrealizedPnlPct < 0.5, then peakPnl < 0.5 is true. However, a new position cannot be 72 hours old, so holdH >= 72 is always false for it. The guard is safe. No new position will be incorrectly closed. opened_at is typed string (non-nullable), so new Date(pos.opened_at) is valid for all positions that pass TypeScript checks.

Trailing stop interaction: 0.5% is far below activation thresholds

Crypto trailing stop activation is 5.0% (line 87), non-crypto is 3.0% (line 88). Both are 6–10x above the 0.5% peak threshold. A position with peakPnl < 0.5% has never come close to trailing activation, so there is no race condition: the hold-duration rule at 0.5% and trailing stop at 5%/3% operate in entirely separate regimes. The proposed ordering (hold-duration check after peak update, before trailing stop logic) is correct.

Partial ladder positions: close is safe on remaining size

pos.size and pos.size_usd are updated in-memory by the native TP fill detection block (lines 1571–1574) before we reach line 1921. Any close placed using pos.size at the insertion point reflects the post-trim size. The close flow for all existing paths uses pos.size and pos.size_usd — they are kept current. Trimmed positions are safe to close with the standard flow.
