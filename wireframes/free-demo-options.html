Recommendation: Option 3 · Hybrid my pick

Lota's frustration on the call wasn't only that the free dashboard was useless. It was that he didn't know what the product did. A walkthrough alone (Option 1) tells but doesn't show. An interactive mock alone (Option 2) shows but risks the user wandering past the banner without registering the model. Hybrid front-loads enough context that the mock makes sense, then lets the product speak for itself.

  • Pre-demo carousel kept short — 3 cards max, swipeable, "skip" link visible. Don't bloat it into onboarding 2.0.
  • Interactive mock reuses real Vela components with a hardcoded fixture layer (think: useDemoData() hook). Same code path as the real app, different data source.
  • Coachmark fires only on first signal-tap. Stored in localStorage. Never blocks navigation.
  • Sticky subscribe CTA is the only persistent up-sell. Banner is informational, not a CTA.

Open questions for you

  1. For the interactive mock: should "Approve" lead to a fake position that always wins (best foot forward, but feels manipulative if the user notices) or to a randomized outcome from a curated set of past wins/losses (more honest, more code)?
  2. Do you want the demo data to reset on every visit (always fresh, always shows the same scripted flow) or to persist for the session (user can close and come back, position is still there)?
  3. Coachmark wording — "Tap any signal to see the trade flow" reads OK but arguably patronizing for a tech-fluent user like Lota. Should it be drier: "Try tapping a signal"?
  4. Banner color — amber feels like a warning. Should the demo banner be a calmer tone (cream-on-ink, or mint), so it reads as informational rather than cautionary?
